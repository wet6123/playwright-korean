## 소개

이 가이드는 모범 사례를 따르고 더욱 탄력적인 테스트를 작성하는 데 도움이 됩니다.

## 테스트 철학

### 사용자에게 보이는 동작 테스트

자동화된 테스트는 애플리케이션 코드가 최종 사용자에게 제대로 작동하는지 확인해야 하며, 사용자가 일반적으로 사용, 확인 또는 알지 못하는 구현 세부 사항(예: 함수 이름, 배열 여부, 일부 요소의 CSS 클래스)에 의존해서는 안 됩니다. 최종 사용자는 페이지에 렌더링된 내용을 보고 상호 작용하므로 테스트는 일반적으로 렌더링된 동일한 출력만 보고 상호 작용해야 합니다.

### 테스트를 최대한 격리

각 테스트는 다른 테스트와 완전히 격리되어야 하며 로컬 스토리지, 세션 스토리지, 데이터, 쿠키 등을 사용하여 독립적으로 실행되어야 합니다. [테스트 격리](./browser-contexts.mdx)는 재현성을 개선하고 디버깅을 용이하게 하며 연속적인 테스트 실패를 방지합니다.

테스트의 특정 부분에 대한 반복을 피하기 위해 [before 및 after hooks](/api/class-test.mdx)를 사용할 수 있습니다. 테스트 파일 내에서 각 테스트 전에 테스트의 일부를 실행하기 위한 before hook을 추가합니다(예: 특정 URL로 이동하거나 앱의 일부에 로그인). 이렇게 하면 어떤 테스트도 다른 테스트에 의존하지 않으므로 테스트가 격리된 상태로 유지됩니다. 그러나 특히 테스트가 명확하고 읽고 유지 관리하기 쉬운 경우 약간의 중복이 있는 것도 괜찮습니다.

```js
import { test } from '@playwright/test';

test.beforeEach(async ({ page }) => {
  // 각 테스트 전에 실행되고 각 페이지에 로그인합니다.
  await page.goto('https://github.com/login');
  await page.getByLabel('Username or email address').fill('username');
  await page.getByLabel('Password').fill('password');
  await page.getByRole('button', { name: 'Sign in' }).click();
});

test('first', async ({ page }) => {
  // 페이지가 로그인되었습니다.
});

test('second', async ({ page }) => {
  // 페이지가 로그인되었습니다.
});
```

[setup project](./auth.mdx#basic-shared-account-in-all-tests)를 사용하여 테스트에서 로그인된 상태를 재사용할 수도 있습니다. 이렇게 하면 한 번만 로그인한 다음 모든 테스트에 대해 로그인 단계를 건너뛸 수 있습니다.

### 타사 종속성 테스트 방지

제어하는 것만 테스트하세요. 제어하지 않는 외부 사이트 또는 타사 서버에 대한 링크를 테스트하지 마세요. 시간이 많이 걸리고 테스트 속도가 느려질 뿐만 아니라 연결하는 페이지의 콘텐츠를 제어할 수 없으며 쿠키 배너나 오버레이 페이지 또는 테스트 실패를 유발할 수 있는 다른 요소가 있는지 여부도 제어할 수 없습니다.

대신 [Playwright 네트워크 API](/network.mdx#handle-requests)를 사용하여 필요한 응답을 보장하세요.

```js
await page.route('**/api/fetch_data_third_party_dependency', (route) =>
  route.fulfill({
    status: 200,
    body: testData,
  })
);
await page.goto('https://example.com');
```

### 데이터베이스로 테스트

데이터베이스로 작업하는 경우 데이터를 제어해야 합니다. 스테이징 환경에서 테스트하고 변경되지 않았는지 확인하세요. 시각적 회귀 테스트의 경우 운영 체제 및 브라우저 버전이 동일한지 확인하세요.

## 모범 사례

### 로케이터 사용

엔드 투 엔드 테스트를 작성하려면 먼저 웹 페이지에서 요소를 찾아야 합니다. Playwright의 내장 [로케이터](./locators.mdx)를 사용하여 이 작업을 수행할 수 있습니다. 로케이터에는 자동 대기 및 재시도 기능이 제공됩니다. 자동 대기는 Playwright가 클릭을 수행하기 전에 요소가 표시되고 활성화되었는지 확인하는 등 요소에 대해 다양한 실행 가능성 검사를 수행함을 의미합니다. 테스트의 탄력성을 높이려면 사용자에게 보이는 속성과 명시적 계약을 우선시하는 것이 좋습니다.

```js
// 👍
page.getByRole('button', { name: 'submit' });
```

#### 체이닝 및 필터링 사용

로케이터를 [체이닝](./locators.mdx#matching-inside-a-locator)하여 검색 범위를 페이지의 특정 부분으로 좁힐 수 있습니다.

```js
const product = page.getByRole('listitem').filter({ hasText: 'Product 2' });
```

텍스트 또는 다른 로케이터로 [로케이터를 필터링](./locators.mdx#filtering-locators)할 수도 있습니다.

```js
await page
  .getByRole('listitem')
  .filter({ hasText: 'Product 2' })
  .getByRole('button', { name: 'Add to cart' })
  .click();
```

#### XPath 또는 CSS 선택자보다 사용자에게 보이는 속성 선호

DOM은 쉽게 변경될 수 있으므로 테스트가 DOM 구조에 의존하면 테스트가 실패할 수 있습니다. 예를 들어 CSS 클래스로 이 버튼을 선택하는 것을 생각해 보세요. 디자이너가 무언가를 변경하면 클래스가 변경되어 테스트가 중단될 수 있습니다.

```js
// 👎
page.locator('button.buttonIcon.episode-actions-later');
```

DOM의 변경에 탄력적인 로케이터를 사용하세요.

```js
// 👍
page.getByRole('button', { name: 'submit' });
```

### 로케이터 생성

Playwright에는 테스트를 생성하고 로케이터를 선택할 수 있는 [테스트 생성기](./codegen.mdx)가 있습니다. 페이지를 보고 역할, 텍스트 및 테스트 ID 로케이터를 우선시하여 최상의 로케이터를 파악합니다. 생성기가 로케이터와 일치하는 여러 요소를 찾으면 로케이터를 개선하여 탄력성을 높이고 대상 요소를 고유하게 식별하므로 로케이터로 인해 테스트가 실패할 염려가 없습니다.

### 웹 우선 어설션 사용

어설션은 예상 결과와 실제 결과가 일치하는지 여부를 확인하는 방법입니다. [웹 우선 어설션](./test-assertions.mdx)을 사용하면 Playwright는 예상 조건이 충족될 때까지 기다립니다. 예를 들어 경고 메시지를 테스트할 때 테스트는 메시지가 나타나게 하는 버튼을 클릭하고 경고 메시지가 있는지 확인합니다. 경고 메시지가 나타나는 데 0.5초가 걸리면 `toBeVisible()`과 같은 어설션은 필요에 따라 기다렸다가 다시 시도합니다.

```js
// 👍
await expect(page.getByText('welcome')).toBeVisible();

// 👎
expect(await page.getByText('welcome').isVisible()).toBe(true);
```

#### 수동 어설션 사용 금지

expect를 기다리지 않는 수동 어설션을 사용하지 마세요. 아래 코드에서 await는 앞이 아닌 expect 내부에 있습니다. `isVisible()`과 같은 어설션을 사용할 때 테스트는 1초도 기다리지 않고 로케이터가 있는지 확인하고 즉시 반환합니다.

```js
// 👎
expect(await page.getByText('welcome').isVisible()).toBe(true);
```

대신 `toBeVisible()`과 같은 웹 우선 어설션을 사용하세요.

```js
// 👍
await expect(page.getByText('welcome')).toBeVisible();
```

### Playwright 도구 사용

Playwright에는 테스트 작성에 도움이 되는 다양한 도구가 제공됩니다.

- [VS Code 확장](./getting-started-vscode.mdx)은 테스트 작성, 실행 및 디버깅 시 훌륭한 개발 경험을 제공합니다.
- [테스트 생성기](./codegen.mdx)는 테스트를 생성하고 로케이터를 선택할 수 있습니다.
- [추적 뷰어](./trace-viewer.mdx)는 쉽게 공유할 수 있는 로컬 PWA로 테스트의 전체 추적을 제공합니다. 추적 뷰어를 사용하면 타임라인을 보고, 각 작업에 대한 DOM 스냅샷을 검사하고, 네트워크 요청을 보고, 기타 작업을 수행할 수 있습니다.
- [UI 모드](./test-ui-mode)를 사용하면 타임 트래블 경험과 함께 감시 모드를 사용하여 테스트를 탐색, 실행 및 디버깅할 수 있습니다. 모든 테스트 파일은 테스트 사이드바에 로드되며, 여기서 각 파일과 describe 블록을 확장하여 각 테스트를 개별적으로 실행, 보기, 감시 및 디버깅할 수 있습니다.
- Playwright의 [TypeScript](./test-typescript)는 즉시 작동하며 더 나은 IDE 통합을 제공합니다. IDE는 수행할 수 있는 모든 작업을 표시하고 잘못된 작업을 수행하면 강조 표시합니다. TypeScript 경험이 필요하지 않으며 코드가 TypeScript에 있을 필요도 없습니다. `.ts` 확장자로 테스트를 생성하기만 하면 됩니다.

### 모든 브라우저에서 테스트

Playwright를 사용하면 어떤 플랫폼을 사용하든 모든 [브라우저](./test-projects.mdx#configure-projects-for-multiple-browsers)에서 사이트를 쉽게 테스트할 수 있습니다. 모든 브라우저에서 테스트하면 모든 사용자에게 앱이 작동하는지 확인할 수 있습니다. 구성 파일에서 이름과 사용할 브라우저 또는 기기를 추가하여 프로젝트를 설정할 수 있습니다.

```js title="playwright.config.ts"
import { defineConfig, devices } from '@playwright/test';

export default defineConfig({
  projects: [
    {
      name: 'chromium',
      use: { ...devices['Desktop Chrome'] },
    },
    {
      name: 'firefox',
      use: { ...devices['Desktop Firefox'] },
    },
    {
      name: 'webkit',
      use: { ...devices['Desktop Safari'] },
    },
  ],
});
```

### CI에서 테스트 실행

CI/CD를 설정하고 테스트를 자주 실행하세요. 테스트를 더 자주 실행할수록 좋습니다. 이상적으로는 각 커밋 및 풀 요청에서 테스트를 실행해야 합니다. Playwright에는 [GitHub 작업 워크플로](/ci-intro.mdx)가 제공되므로 설정 없이 CI에서 테스트가 실행됩니다. Playwright는 선택한 [CI 환경](/ci.mdx)에서도 설정할 수 있습니다.

CI에서 테스트를 실행할 때는 비용이 저렴한 Linux를 사용하세요. 개발자는 로컬에서 실행할 때 어떤 환경이든 사용할 수 있지만 CI에서는 linux를 사용하세요. CI 속도를 높이려면 [샤딩](./test-sharding.mdx) 설정을 고려하세요.

#### CI에서 브라우저 다운로드 최적화

실제로 필요한 브라우저만 설치하세요(특히 CI에서). 예를 들어 Chromium으로만 테스트하는 경우 Chromium만 설치하세요.

```bash title=".github/workflows/playwright.yml"
# 모든 브라우저를 설치하는 대신
npx playwright install --with-deps

# Chromium만 설치
npx playwright install chromium --with-deps
```

이렇게 하면 CI 머신의 다운로드 시간과 디스크 공간이 모두 절약됩니다.

### 테스트 린트

오류를 조기에 포착하기 위해 ESLint를 사용하여 TypeScript 및 린팅하는 것이 좋습니다. [`@typescript-eslint/no-floating-promises`](https://typescript-eslint.io/rules/no-floating-promises/) [ESLint](https://eslint.org) 규칙을 사용하여 Playwright API에 대한 비동기 호출 전에 await가 누락되지 않았는지 확인하세요. CI에서 `tsc --noEmit`를 실행하여 함수가 올바른 시그니처로 호출되는지 확인할 수 있습니다.

### 병렬 처리 및 샤딩 사용

Playwright는 기본적으로 [병렬](./test-parallel.mdx)로 테스트를 실행합니다. 단일 파일의 테스트는 동일한 작업자 프로세스에서 순서대로 실행됩니다. 단일 파일에 독립적인 테스트가 많은 경우 병렬로 실행할 수 있습니다.

```js
import { test } from '@playwright/test';

test.describe.configure({ mode: 'parallel' });

test('병렬로 실행 1', async ({ page }) => {
  /* ... */
});
test('병렬로 실행 2', async ({ page }) => {
  /* ... */
});
```

Playwright는 여러 머신에서 실행할 수 있도록 테스트 스위트를 [샤딩](./test-parallel.mdx#shard-tests-between-multiple-machines)할 수 있습니다.

### 소프트 어설션 사용

테스트가 실패하면 Playwright는 VS Code, 터미널, HTML 보고서 또는 추적 뷰어에서 볼 수 있는 테스트의 어떤 부분이 실패했는지 보여주는 오류 메시지를 제공합니다. 그러나 [소프트 어설션](/test-assertions.mdx#soft-assertions)을 사용할 수도 있습니다. 이러한 어설션은 테스트 실행을 즉시 종료하지 않고 테스트가 종료되면 실패한 어설션 목록을 컴파일하고 표시합니다.

```js
// 실패해도 테스트를 중지하지 않는 몇 가지 검사를 수행합니다...
await expect.soft(page.getByTestId('status')).toHaveText('Success');

// ... 그리고 테스트를 계속하여 더 많은 것을 확인합니다.
await page.getByRole('link', { name: 'next page' }).click();
```
