## Introduction

Playwright Test는 테스트 픽스처 개념을 기반으로 합니다. 테스트 픽스처는 각 테스트에 필요한 모든 것과 그 외의 것을 제공하여 테스트 환경을 설정하는 데 사용됩니다. 테스트 픽스처는 테스트 간에 격리됩니다. 픽스처를 사용하면 공통 설정 대신 의미에 따라 테스트를 그룹화할 수 있습니다.

### Built-in fixtures

첫 번째 테스트에서 이미 테스트 픽스처를 사용했습니다.

```js
import { test, expect } from '@playwright/test';

test('basic test', async ({ page }) => {
  await page.goto('https://playwright.dev/');

  await expect(page).toHaveTitle(/Playwright/);
});
```

`{ page }` 인수는 Playwright Test에 `page` 픽스처를 설정하고 테스트 함수에 제공하도록 지시합니다.

다음은 가장 많이 사용할 가능성이 높은 미리 정의된 픽스처 목록입니다.

|Fixture    |Type               |Description                                                             |
|:----------|:------------------|:-----------------------------------------------------------------------|
|page       |[Page]             |이 테스트 실행을 위한 격리된 페이지입니다.                                              |
|context    |[BrowserContext]   |이 테스트 실행을 위한 격리된 컨텍스트입니다. `page` 픽스처도 이 컨텍스트에 속합니다. [컨텍스트 구성](./test-configuration.mdx) 방법을 알아보세요. |
|browser    |[Browser]          |브라우저는 리소스를 최적화하기 위해 테스트 간에 공유됩니다. [브라우저 구성](./test-configuration.mdx) 방법을 알아보세요. |
|browserName|[string]           |현재 테스트를 실행 중인 브라우저의 이름입니다. `chromium`, `firefox` 또는 `webkit` 중 하나입니다.               |
|request    |[APIRequestContext]|이 테스트 실행을 위한 격리된 [APIRequestContext](./api/class-apirequestcontext.mdx) 인스턴스입니다.    |

### Without fixtures

다음은 일반적인 테스트 환경 설정이 기존 테스트 스타일과 픽스처 기반 스타일 간에 어떻게 다른지 보여줍니다.

`TodoPage`는 [Page Object Model](./pom.mdx) 패턴에 따라 웹 앱의 "할 일 목록" 페이지와 상호 작용하는 데 도움이 되는 클래스입니다. 내부적으로 Playwright의 `page`를 사용합니다.

<details>

<summary><code>TodoPage</code>에 대한 코드를 보려면 클릭하세요.</summary>

<div>

```js title="todo-page.ts"
import type { Page, Locator } from '@playwright/test';

export class TodoPage {
  private readonly inputBox: Locator;
  private readonly todoItems: Locator;

  constructor(public readonly page: Page) {
    this.inputBox = this.page.locator('input.new-todo');
    this.todoItems = this.page.getByTestId('todo-item');
  }

  async goto() {
    await this.page.goto('https://demo.playwright.dev/todomvc/');
  }

  async addToDo(text: string) {
    await this.inputBox.fill(text);
    await this.inputBox.press('Enter');
  }

  async remove(text: string) {
    const todo = this.todoItems.filter({ hasText: text });
    await todo.hover();
    await todo.getByLabel('Delete').click();
  }

  async removeAll() {
    while ((await this.todoItems.count()) > 0) {
      await this.todoItems.first().hover();
      await this.todoItems.getByLabel('Delete').first().click();
    }
  }
}
```

</div>

</details>

```js title="todo.spec.ts"
const { test } = require('@playwright/test');
const { TodoPage } = require('./todo-page');

test.describe('todo tests', () => {
  let todoPage;

  test.beforeEach(async ({ page }) => {
    todoPage = new TodoPage(page);
    await todoPage.goto();
    await todoPage.addToDo('item1');
    await todoPage.addToDo('item2');
  });

  test.afterEach(async () => {
    await todoPage.removeAll();
  });

  test('should add an item', async () => {
    await todoPage.addToDo('my item');
    // ...
  });

  test('should remove an item', async () => {
    await todoPage.remove('item1');
    // ...
  });
});
```

### With fixtures

픽스처는 before/after 훅보다 다음과 같은 여러 가지 장점이 있습니다.
- 픽스처는 설정 및 정리를 같은 위치에 **캡슐화**하여 작성하기가 더 쉽습니다. 따라서 before 훅에서 생성된 항목을 정리하는 after 훅이 있는 경우 픽스처로 전환하는 것이 좋습니다.
- 픽스처는 테스트 파일 간에 **재사용**할 수 있습니다. 한 번 정의하고 모든 테스트에서 사용할 수 있습니다. 이것이 Playwright의 기본 제공 `page` 픽스처가 작동하는 방식입니다. 따라서 여러 테스트에서 사용되는 헬퍼 함수가 있는 경우 픽스처로 전환하는 것이 좋습니다.
- 픽스처는 **온디맨드**입니다. 원하는 만큼 픽스처를 정의할 수 있으며 Playwright Test는 테스트에 필요한 항목만 설정하고 나머지는 설정하지 않습니다.
- 픽스처는 **구성 가능**합니다. 서로 의존하여 복잡한 동작을 제공할 수 있습니다.
- 픽스처는 **유연**합니다. 테스트는 픽스처의 조합을 사용하여 다른 테스트에 영향을 주지 않고 필요한 정확한 환경을 조정할 수 있습니다.
- 픽스처는 **그룹화**를 단순화합니다. 더 이상 환경을 설정하는 `describe`로 테스트를 래핑할 필요가 없으며 대신 의미별로 테스트를 자유롭게 그룹화할 수 있습니다.

## Creating a fixture

자체 픽스처를 생성하려면 [test.extend()](/api/class-test.mdx#test-extend)를 사용하여 픽스처를 포함할 새 `test` 객체를 생성합니다.

아래에서는 [Page Object Model](./pom.mdx) 패턴을 따르는 두 개의 픽스처 `todoPage`와 `settingsPage`를 만듭니다.

```js title="my-test.ts"
import { test as base } from '@playwright/test';
import { TodoPage } from './todo-page';
import { SettingsPage } from './settings-page';

// 픽스처의 유형을 선언합니다.
type MyFixtures = {
  todoPage: TodoPage;
  settingsPage: SettingsPage;
};

// "todoPage" 및 "settingsPage"를 제공하여 기본 테스트를 확장합니다.
// 이 새로운 "test"는 여러 테스트 파일에서 사용할 수 있으며 각 파일은 픽스처를 가져옵니다.
export const test = base.extend<MyFixtures>({
  todoPage: async ({ page }, use) => {
    // 픽스처를 설정합니다.
    const todoPage = new TodoPage(page);
    await todoPage.goto();
    await todoPage.addToDo('item1');
    await todoPage.addToDo('item2');

    // 테스트에서 픽스처 값을 사용합니다.
    await use(todoPage);

    // 픽스처를 정리합니다.
    await todoPage.removeAll();
  },

  settingsPage: async ({ page }, use) => {
    await use(new SettingsPage(page));
  },
});
export { expect } from '@playwright/test';
```

:::note
사용자 정의 픽스처 이름은 글자 또는 밑줄로 시작해야 하며 글자, 숫자, 밑줄만 포함할 수 있습니다.
:::

## Using a fixture

테스트 함수 인수에서 픽스처를 언급하기만 하면 테스트 러너가 처리합니다. 픽스처는 훅 및 기타 픽스처에서도 사용할 수 있습니다. TypeScript를 사용하는 경우 픽스처는 올바른 유형을 갖습니다.

아래에서는 위에서 정의한 `todoPage` 및 `settingsPage` 픽스처를 사용합니다.

```js
import { test, expect } from './my-test';

test.beforeEach(async ({ settingsPage }) => {
  await settingsPage.switchToDarkMode();
});

test('basic test', async ({ todoPage, page }) => {
  await todoPage.addToDo('something nice');
  await expect(page.getByTestId('todo-title')).toContainText(['something nice']);
});
```

## Overriding fixtures

자체 픽스처를 생성하는 것 외에도 기존 픽스처를 필요에 맞게 재정의할 수도 있습니다. `baseURL`로 자동 탐색하여 `page` 픽스처를 재정의하는 다음 예제를 고려하십시오.

```js
import { test as base } from '@playwright/test';

export const test = base.extend({
  page: async ({ baseURL, page }, use) => {
    await page.goto(baseURL);
    await use(page);
  },
});
```

이 예제에서 `page` 픽스처는 [testOptions.baseURL](/api/class-testoptions.mdx#test-options-base-url)과 같은 다른 내장 픽스처에 의존할 수 있습니다. 이제 구성 파일에서 `baseURL`을 구성하거나 [test.use()](/api/class-test.mdx#test-use)를 사용하여 테스트 파일에서 로컬로 구성할 수 있습니다.

```js title="example.spec.ts"

test.use({ baseURL: 'https://playwright.dev' });
```

기본 픽스처가 완전히 다른 것으로 대체되는 경우 픽스처를 재정의할 수도 있습니다. 예를 들어, [testOptions.storageState](/api/class-testoptions.mdx#test-options-storage-state) 픽스처를 재정의하여 자체 데이터를 제공할 수 있습니다.

```js
import { test as base } from '@playwright/test';

export const test = base.extend({
  storageState: async ({}, use) => {
    const cookie = await getAuthCookie();
    await use({ cookies: [cookie] });
  },
});
```

## Worker-scoped fixtures

Playwright Test는 [워커 프로세스](./test-parallel.mdx)를 사용하여 테스트 파일을 실행합니다. 개별 테스트 실행에 대해 테스트 픽스처가 설정되는 방식과 유사하게 각 워커 프로세스에 대해 워커 픽스처가 설정됩니다. 여기에서 서비스를 설정하고 서버를 실행하는 등의 작업을 수행할 수 있습니다. Playwright Test는 워커 픽스처가 일치하고 환경이 동일한 경우 가능한 한 많은 테스트 파일에 대해 워커 프로세스를 재사용합니다.

아래에서는 동일한 워커의 모든 테스트에서 공유할 `account` 픽스처를 만들고 각 테스트에 대해 이 계정으로 로그인하도록 `page` 픽스처를 재정의합니다. 고유한 계정을 생성하기 위해 모든 테스트 또는 픽스처에서 사용할 수 있는 [workerInfo.workerIndex](/api/class-workerinfo.mdx#worker-info-worker-index)를 사용합니다. 워커 픽스처에 대한 튜플과 같은 구문에 주목하십시오. 테스트 러너가 워커당 한 번 이 픽스처를 설정하도록 `{scope: 'worker'}`를 전달해야 합니다.

```js title="my-test.ts"
import { test as base } from '@playwright/test';

type Account = {
  username: string;
  password: string;
};

// 워커 픽스처 유형을 두 번째 템플릿 매개변수로 전달합니다.
export const test = base.extend<{}, { account: Account }>({
  account: [async ({ browser }, use, workerInfo) => {
    // 고유한 사용자 이름.
    const username = 'user' + workerInfo.workerIndex;
    const password = 'verysecure';

    // Playwright로 계정을 만듭니다.
    const page = await browser.newPage();
    await page.goto('/signup');
    await page.getByLabel('User Name').fill(username);
    await page.getByLabel('Password').fill(password);
    await page.getByText('Sign up').click();
    // 모든 것이 정상인지 확인하십시오.
    await expect(page.getByTestId('result')).toHaveText('Success');
    // 정리하는 것을 잊지 마십시오.
    await page.close();

    // 계정 값을 사용합니다.
    await use({ username, password });
  }, { scope: 'worker' }],

  page: async ({ page, account }, use) => {
    // 계정으로 로그인합니다.
    const { username, password } = account;
    await page.goto('/signin');
    await page.getByLabel('User Name').fill(username);
    await page.getByLabel('Password').fill(password);
    await page.getByText('Sign in').click();
    await expect(page.getByTestId('userinfo')).toHaveText(username);

    // 테스트에서 로그인한 페이지를 사용합니다.
    await use(page);
  },
});
export { expect } from '@playwright/test';
```

## Automatic fixtures

자동 픽스처는 테스트가 직접 나열하지 않더라도 각 테스트/워커에 대해 설정됩니다. 자동 픽스처를 만들려면 튜플 구문을 사용하고 `{ auto: true }`를 전달합니다.

다음은 테스트가 실패할 때 디버그 로그를 자동으로 첨부하는 픽스처의 예입니다. 나중에 리포터에서 로그를 검토할 수 있습니다. 실행 중인 테스트에 대한 메타데이터를 검색하기 위해 각 테스트/픽스처에서 사용할 수 있는 [TestInfo] 객체를 사용하는 방법에 주목하십시오.

```js title="my-test.ts"
import debug from 'debug';
import fs from 'fs';
import { test as base } from '@playwright/test';

export const test = base.extend<{ saveLogs: void }>({
  saveLogs: [async ({}, use, testInfo) => {
    // 테스트 중에 로그를 수집합니다.
    const logs = [];
    debug.log = (...args) => logs.push(args.map(String).join(''));
    debug.enable('myserver');

    await use();

    // 테스트 후 테스트가 통과했는지 또는 실패했는지 확인할 수 있습니다.
    if (testInfo.status !== testInfo.expectedStatus) {
      // outputPath() API는 고유한 파일 이름을 보장합니다.
      const logFile = testInfo.outputPath('logs.txt');
      await fs.promises.writeFile(logFile, logs.join('\n'), 'utf8');
      testInfo.attachments.push({ name: 'logs', contentType: 'text/plain', path: logFile });
    }
  }, { auto: true }],
});
export { expect } from '@playwright/test';
```

## Fixture timeout

기본적으로 픽스처는 테스트와 시간 초과를 공유합니다. 그러나 느린 픽스처, 특히 [워커 범위](#worker-scoped-fixtures) 픽스처의 경우 별도의 시간 초과를 사용하는 것이 편리합니다. 이렇게 하면 전체 테스트 시간 초과를 작게 유지하고 느린 픽스처에 더 많은 시간을 할당할 수 있습니다.

```js
import { test as base, expect } from '@playwright/test';

const test = base.extend<{ slowFixture: string }>({
  slowFixture: [async ({}, use) => {
    // ... 느린 작업을 수행합니다 ...
    await use('hello');
  }, { timeout: 60000 }]
});

test('example test', async ({ slowFixture }) => {
  // ...
});
```

## Fixtures-options

Playwright Test는 별도로 구성할 수 있는 여러 테스트 프로젝트 실행을 지원합니다. "옵션" 픽스처를 사용하여 구성 옵션을 선언적이고 유형 검사되도록 만들 수 있습니다. [테스트 매개변수화](./test-parameterize.mdx)에 대해 자세히 알아보세요.

아래에서는 다른 예제의 `todoPage` 픽스처 외에도 `defaultItem` 옵션을 만듭니다. 이 옵션은 구성 파일에서 설정됩니다. 튜플 구문과 `{ option: true }` 인수에 주목하십시오.

<details>

<summary><code>TodoPage</code>에 대한 코드를 보려면 클릭하세요.</summary>

<div>

```js title="todo-page.ts"
import type { Page, Locator } from '@playwright/test';

export class TodoPage {
  private readonly inputBox: Locator;
  private readonly todoItems: Locator;

  constructor(public readonly page: Page) {
    this.inputBox = this.page.locator('input.new-todo');
    this.todoItems = this.page.getByTestId('todo-item');
  }

  async goto() {
    await this.page.goto('https://demo.playwright.dev/todomvc/');
  }

  async addToDo(text: string) {
    await this.inputBox.fill(text);
    await this.inputBox.press('Enter');
  }

  async remove(text: string) {
    const todo = this.todoItems.filter({ hasText: text });
    await todo.hover();
    await todo.getByLabel('Delete').click();
  }

  async removeAll() {
    while ((await this.todoItems.count()) > 0) {
      await this.todoItems.first().hover();
      await this.todoItems.getByLabel('Delete').first().click();
    }
  }
}
```

</div>

```js title="my-test.ts"
import { test as base } from '@playwright/test';
import { TodoPage } from './todo-page';

// 구성의 유형을 검사하기 위해 옵션을 선언합니다.
export type MyOptions = {
  defaultItem: string;
};
type MyFixtures = {
  todoPage: TodoPage;
};

// 옵션 및 픽스처 유형을 모두 지정합니다.
export const test = base.extend<MyOptions & MyFixtures>({
  // 옵션을 정의하고 기본값을 제공합니다.
  // 나중에 구성에서 재정의할 수 있습니다.
  defaultItem: ['Something nice', { option: true }],

  // "todoPage" 픽스처는 옵션에 따라 다릅니다.
  todoPage: async ({ page, defaultItem }, use) => {
    const todoPage = new TodoPage(page);
    await todoPage.goto();
    await todoPage.addToDo(defaultItem);
    await use(todoPage);
    await todoPage.removeAll();
  },
});
export { expect } from '@playwright/test';
```

```js title="playwright.config.ts"

export default defineConfig<MyOptions>({

    {
```

**배열을 옵션 값으로 사용**

옵션 값이 배열인 경우(예: `[{ name: 'Alice' }, { name: 'Bob' }]`) 값을 제공할 때 추가 배열로 래핑해야 합니다. 이는 예제로 가장 잘 설명됩니다.

```js
type Person = { name: string };
const test = base.extend<{ persons: Person[] }>({
  // 옵션을 선언하고 기본값은 빈 배열입니다.
  persons: [[], { option: true }],
});

// 옵션 값은 사람 배열입니다.
const actualPersons = [{ name: 'Alice' }, { name: 'Bob' }];
test.use({
  // 올바름: 값을 배열로 래핑하고 범위를 전달합니다.
  persons: [actualPersons, { scope: 'test' }],
});

test.use({
  // 잘못됨: 배열 값을 직접 전달하면 작동하지 않습니다.
  persons: actualPersons,
});
```

## Execution order

각 픽스처에는 픽스처의 `await use()` 호출로 구분되는 설정 및 해체 단계가 있습니다. 설정은 테스트/훅에서 픽스처를 사용하기 전에 실행되고 해체는 테스트/훅에서 픽스처를 더 이상 사용하지 않을 때 실행됩니다.

픽스처는 다음 규칙에 따라 실행 순서를 결정합니다.
* 픽스처 A가 픽스처 B에 의존하는 경우: B는 항상 A보다 먼저 설정되고 A 다음에 해체됩니다.
* 자동이 아닌 픽스처는 테스트/훅에 필요한 경우에만 지연 방식으로 실행됩니다.
* 테스트 범위 픽스처는 각 테스트 후에 해체되는 반면 워커 범위 픽스처는 테스트를 실행하는 워커 프로세스가 종료될 때만 해체됩니다.

다음 예제를 고려하십시오.

```js
import { test as base } from '@playwright/test';

const test = base.extend<{
  testFixture: string,
  autoTestFixture: string,
  unusedFixture: string,
}, {
  workerFixture: string,
  autoWorkerFixture: string,
}>({
  workerFixture: [async ({ browser }) => {
    // workerFixture 설정...
    await use('workerFixture');
    // workerFixture 해체...
  }, { scope: 'worker' }],

  autoWorkerFixture: [async ({ browser }) => {
    // autoWorkerFixture 설정...
    await use('autoWorkerFixture');
    // autoWorkerFixture 해체...
  }, { scope: 'worker', auto: true }],

  testFixture: [async ({ page, workerFixture }) => {
    // testFixture 설정...
    await use('testFixture');
    // testFixture 해체...
  }, { scope: 'test' }],

  autoTestFixture: [async () => {
    // autoTestFixture 설정...
    await use('autoTestFixture');
    // autoTestFixture 해체...
  }, { scope: 'test', auto: true }],

  unusedFixture: [async ({ page }) => {
    // unusedFixture 설정...
    await use('unusedFixture');
    // unusedFixture 해체...
  }, { scope: 'test' }],
});

test.beforeAll(async () => { /* ... */ });
test.beforeEach(async ({ page }) => { /* ... */ });
test('first test', async ({ page }) => { /* ... */ });
test('second test', async ({ testFixture }) => { /* ... */ });
test.afterEach(async () => { /* ... */ });
test.afterAll(async () => { /* ... */ });
```

일반적으로 모든 테스트가 통과되고 오류가 발생하지 않으면 실행 순서는 다음과 같습니다.
* 워커 설정 및 `beforeAll` 섹션:
  * `autoWorkerFixture`에 필요하므로 `browser`를 설정합니다.
  * 자동 워커 픽스처는 항상 다른 항목보다 먼저 설정되므로 `autoWorkerFixture`를 설정합니다.
  * `beforeAll`을 실행합니다.
* `first test` 섹션:
  * 자동 테스트 픽스처는 항상 테스트 및 `beforeEach` 훅보다 먼저 설정되므로 `autoTestFixture`를 설정합니다.
  * `beforeEach` 훅에 필요하므로 `page`를 설정합니다.
  * `beforeEach`를 실행합니다.
  * `first test`를 실행합니다.
  * `afterEach`를 실행합니다.
  * 테스트 범위 픽스처이므로 테스트가 완료된 후 해체해야 하므로 `page`를 해체합니다.
  * 테스트 범위 픽스처이므로 테스트가 완료된 후 해체해야 하므로 `autoTestFixture`를 해체합니다.
* `second test` 섹션:
  * 자동 테스트 픽스처는 항상 테스트 및 `beforeEach` 훅보다 먼저 설정되므로 `autoTestFixture`를 설정합니다.
  * `beforeEach` 훅에 필요하므로 `page`를 설정합니다.
  * `beforeEach`를 실행합니다.
  * `second test`에 필요한 `testFixture`에 필요하므로 `workerFixture`를 설정합니다.
  * `second test`에 필요하므로 `testFixture`를 설정합니다.
  * `second test`를 실행합니다.
  * `afterEach`를 실행합니다.
  * 테스트 범위 픽스처이므로 테스트가 완료된 후 해체해야 하므로 `testFixture`를 해체합니다.
  * 테스트 범위 픽스처이므로 테스트가 완료된 후 해체해야 하므로 `page`를 해체합니다.
  * 테스트 범위 픽스처이므로 테스트가 완료된 후 해체해야 하므로 `autoTestFixture`를 해체합니다.
* `afterAll` 및 워커 해체 섹션:
  * `afterAll`을 실행합니다.
  * 워커 범위 픽스처이므로 끝에서 한 번 해체해야 하므로 `workerFixture`를 해체합니다.
  * 워커 범위 픽스처이므로 끝에서 한 번 해체해야 하므로 `autoWorkerFixture`를 해체합니다.
  * 워커 범위 픽스처이므로 끝에서 한 번 해체해야 하므로 `browser`를 해체합니다.

몇 가지 관찰 사항:
* `page` 및 `autoTestFixture`는 테스트 범위 픽스처로 각 테스트에 대해 설정 및 해체됩니다.
* `unusedFixture`는 테스트/훅에서 사용되지 않으므로 설정되지 않습니다.
* `testFixture`는 `workerFixture`에 의존하고 설정을 트리거합니다.
* `workerFixture`는 두 번째 테스트 전에 지연 방식으로 설정되지만 워커 종료 중에 워커 범위 픽스처로 한 번 해체됩니다.
* `autoWorkerFixture`는 `beforeAll` 훅에 대해 설정되지만 `autoTestFixture`는 설정되지 않습니다.

## Combine custom fixtures from multiple modules

여러 파일 또는 모듈의 테스트 픽스처를 병합할 수 있습니다.

```js title="fixtures.ts"
import { mergeTests } from '@playwright/test';
import { test as dbTest } from 'database-test-utils';
import { test as a11yTest } from 'a11y-test-utils';

export const test = mergeTests(dbTest, a11yTest);
```

```js title="test.spec.ts"
import { test } from './fixtures';

test('passes', async ({ database, page, a11y }) => {
  // database 및 a11y 픽스처를 사용합니다.
});
```

## Box fixtures

일반적으로 사용자 정의 픽스처는 UI 모드, Trace Viewer 및 다양한 테스트 보고서에서 별도의 단계로 보고됩니다. 또한 테스트 러너의 오류 메시지에도 나타납니다. 자주 사용되는 픽스처의 경우 많은 노이즈가 발생할 수 있습니다. "박싱"하여 UI에 픽스처 단계가 표시되지 않도록 할 수 있습니다.

```js
import { test as base } from '@playwright/test';

export const test = base.extend({
  helperFixture: [async ({}, use, testInfo) => {
    // ...
  }, { box: true }],
});
```

이는 흥미롭지 않은 헬퍼 픽스처에 유용합니다. 예를 들어, 일부 공통 데이터를 설정하는 [자동](./test-fixtures.mdx#automatic-fixtures) 픽스처는 테스트 보고서에서 안전하게 숨길 수 있습니다.

## Custom fixture title

일반적인 픽스처 이름 대신 테스트 보고서 및 오류 메시지에 표시될 사용자 정의 제목을 픽스처에 지정할 수 있습니다.

```js
import { test as base } from '@playwright/test';

export const test = base.extend({
  innerFixture: [async ({}, use, testInfo) => {
    // ...
  }, { title: 'my fixture' }],
});
```

## Adding global beforeEach/afterEach hooks

[test.beforeEach()](/api/class-test.mdx#test-before-each) 및 [test.afterEach()](/api/class-test.mdx#test-after-each) 훅은 동일한 파일 및 동일한 [test.describe()](/api/class-test.mdx#test-describe) 블록(있는 경우)에 선언된 각 테스트 전/후에 실행됩니다. 각 테스트 전/후에 전역적으로 실행되는 훅을 선언하려면 다음과 같이 자동 픽스처로 선언할 수 있습니다.

```ts title="fixtures.ts"
import { test as base } from '@playwright/test';

export const test = base.extend<{ forEachTest: void }>({
  forEachTest: [async ({ page }, use) => {
    // 이 코드는 모든 테스트 전에 실행됩니다.
    await page.goto('http://localhost:8000');
    await use();
    // 이 코드는 모든 테스트 후에 실행됩니다.
    console.log('Last URL:', page.url());
  }, { auto: true }],  // 모든 테스트에 대해 자동으로 시작됩니다.
});
```

그런 다음 모든 테스트에서 픽스처를 가져옵니다.

```ts title="mytest.spec.ts"
import { test } from './fixtures';
import { expect } from '@playwright/test';

test('basic', async ({ page }) => {
  expect(page).toHaveURL('http://localhost:8000');
  await page.goto('https://playwright.dev');
});
```

## Adding global beforeAll/afterAll hooks

[test.beforeAll()](/api/class-test.mdx#test-before-all) 및 [test.afterAll()](/api/class-test.mdx#test-after-all) 훅은 동일한 파일 및 동일한 [test.describe()](/api/class-test.mdx#test-describe) 블록(있는 경우)에 선언된 모든 테스트 전/후에 워커 프로세스당 한 번 실행됩니다. 모든 파일의 모든 테스트 전/후에 실행되는 훅을 선언하려면 다음과 같이 `scope: 'worker'`를 사용하여 자동 픽스처로 선언할 수 있습니다.

```ts title="fixtures.ts"
import { test as base } from '@playwright/test';

export const test = base.extend<{}, { forEachWorker: void }>({
  forEachWorker: [async ({}, use) => {
    // 이 코드는 워커 프로세스의 모든 테스트 전에 실행됩니다.
    console.log(`Starting test worker ${test.info().workerIndex}`);
    await use();
    // 이 코드는 워커 프로세스의 모든 테스트 후에 실행됩니다.
    console.log(`Stopping test worker ${test.info().workerIndex}`);
  }, { scope: 'worker', auto: true }],  // 모든 워커에 대해 자동으로 시작됩니다.
});
```

그런 다음 모든 테스트에서 픽스처를 가져옵니다.

```ts title="mytest.spec.ts"
import { test } from './fixtures';
import { expect } from '@playwright/test';

test('basic', async ({ }) => {
  // ...
});
```

픽스처는 여전히 [워커 프로세스](./test-parallel.mdx#worker-processes)당 한 번 실행되지만, 모든 파일에서 다시 선언할 필요는 없습니다.
