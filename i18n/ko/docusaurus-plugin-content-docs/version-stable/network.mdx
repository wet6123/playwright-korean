## Introduction

Playwright는 HTTP 및 HTTPS 브라우저 네트워크 트래픽을 **모니터링**하고 **수정**하는 API를 제공합니다. [XHR](https://developer.mozilla.org/en-US/docs/Web/API/XMLHttpRequest) 및 [fetch](https://developer.mozilla.org/en-US/docs/Web/API/Fetch_API) 요청을 포함하여 페이지가 수행하는 모든 요청을 추적, 수정 및 처리할 수 있습니다.

## Mock APIs

자세한 내용은 [API mocking guide](./mock.mdx)를 확인하세요.
- API 요청을 mocking하여 API에 접속하지 않기
- API 요청을 수행하고 응답을 수정하기
- HAR 파일을 사용하여 네트워크 요청을 mocking하기

## Network mocking

네트워크 요청을 mocking하기 위해 아무것도 구성할 필요가 없습니다. 브라우저 컨텍스트에 대한 네트워크를 mocking하는 사용자 정의 [Route]를 정의하기만 하면 됩니다.

```js title="example.spec.ts"
import { test, expect } from '@playwright/test';

test.beforeEach(async ({ context }) => {
  // 이 파일의 각 테스트에 대한 모든 css 요청을 차단합니다.
  await context.route(/.css$/, route => route.abort());
});

test('loads page without css', async ({ page }) => {
  await page.goto('https://playwright.dev');
  // ... 테스트는 여기에 있습니다.
});
```

또는 [page.route()](/api/class-page.mdx#page-route)를 사용하여 단일 페이지에서 네트워크를 mocking할 수 있습니다.

```js title="example.spec.ts"
import { test, expect } from '@playwright/test';

test('loads page without images', async ({ page }) => {
  // png 및 jpeg 이미지 차단.
  await page.route(/(png|jpeg)$/, route => route.abort());

  await page.goto('https://playwright.dev');
  // ... 테스트는 여기에 있습니다.
});
```

## HTTP Proxy

HTTP(S) 프록시 또는 SOCKSv5를 통해 페이지를 로드하도록 구성할 수 있습니다. 프록시는 전체 브라우저에 대해 전역적으로 설정하거나 각 브라우저 컨텍스트에 대해 개별적으로 설정할 수 있습니다.

HTTP(S) 프록시에 대한 사용자 이름과 암호를 선택적으로 지정할 수 있으며, [proxy](/api/class-browser.mdx#browser-new-context-option-proxy)를 우회할 호스트를 지정할 수도 있습니다.

## Network events

모든 [Request] 및 [Response]를 모니터링할 수 있습니다.

```js
// 'request' 및 'response' 이벤트 구독.
page.on('request', request => console.log('>>', request.method(), request.url()));
page.on('response', response => console.log('<<', response.status(), response.url()));

await page.goto('https://example.com');
```

또는 [page.waitForResponse()](/api/class-page.mdx#page-wait-for-response)를 사용하여 버튼 클릭 후 네트워크 응답을 기다립니다.

```js
// glob URL 패턴을 사용합니다. await가 없습니다.
const responsePromise = page.waitForResponse('**/api/fetch_data');
await page.getByText('Update').click();
const response = await responsePromise;
```

#### Variations

[page.waitForResponse()](/api/class-page.mdx#page-wait-for-response)로 [Response]를 기다립니다.

```js
// RegExp를 사용합니다. await가 없습니다.
const responsePromise = page.waitForResponse(/\.jpeg$/);
await page.getByText('Update').click();
const response = await responsePromise;

// Response 객체를 사용하는 predicate를 사용합니다. await가 없습니다.
const responsePromise = page.waitForResponse(response => response.url().includes(token));
await page.getByText('Update').click();
const response = await responsePromise;
```

## Handle requests

```js
await page.route('**/api/fetch_data', route => route.fulfill({
  status: 200,
  body: testData,
}));
await page.goto('https://example.com');
```

Playwright 스크립트에서 네트워크 요청을 처리하여 API 엔드포인트를 mocking할 수 있습니다.

#### Variations

[browserContext.route()](/api/class-browsercontext.mdx#browser-context-route)를 사용하여 전체 브라우저 컨텍스트에서 또는 [page.route()](/api/class-page.mdx#page-route)를 사용하여 페이지에서 경로를 설정합니다. 팝업 창과 열린 링크에 적용됩니다.

```js
await browserContext.route('**/api/login', route => route.fulfill({
  status: 200,
  body: 'accept',
}));
await page.goto('https://example.com');
```

## Modify requests

```js
// 헤더 삭제
await page.route('**/*', async route => {
  const headers = route.request().headers();
  delete headers['X-Secret'];
  await route.continue({ headers });
});

// 요청을 POST로 계속합니다.
await page.route('**/*', route => route.continue({ method: 'POST' }));
```

수정하여 요청을 계속할 수 있습니다. 위의 예제는 나가는 요청에서 HTTP 헤더를 제거합니다.

## Abort requests

[page.route()](/api/class-page.mdx#page-route) 및 [route.abort()](/api/class-route.mdx#route-abort)를 사용하여 요청을 중단할 수 있습니다.

```js
await page.route('**/*.{png,jpg,jpeg}', route => route.abort());

// 요청 유형에 따라 중단
await page.route('**/*', route => {
  return route.request().resourceType() === 'image' ? route.abort() : route.continue();
});
```

## Modify responses

응답을 수정하려면 [APIRequestContext]를 사용하여 원래 응답을 가져온 다음 응답을 [route.fulfill()](/api/class-route.mdx#route-fulfill)에 전달합니다. 옵션을 통해 응답의 개별 필드를 재정의할 수 있습니다.

```js
await page.route('**/title.html', async route => {
  // 원래 응답을 가져옵니다.
  const response = await route.fetch();
  // 제목에 접두사 추가.
  let body = await response.text();
  body = body.replace('<title>', '<title>My prefix:');
  await route.fulfill({
    // 응답의 모든 필드를 전달합니다.
    response,
    // 응답 body 재정의.
    body,
    // 콘텐츠 유형을 html로 강제합니다.
    headers: {
      ...response.headers(),
      'content-type': 'text/html'
    }
  });
});
```

## Glob URL patterns

Playwright는 [page.route()](/api/class-page.mdx#page-route) 또는 [page.waitForResponse()](/api/class-page.mdx#page-wait-for-response)와 같은 네트워크 인터셉션 메서드에서 URL 매칭을 위해 단순화된 glob 패턴을 사용합니다. 이러한 패턴은 기본 와일드카드를 지원합니다.
1. 별표:
   - 단일 `*`는 `/`를 제외한 모든 문자와 일치합니다.
   - 이중 `**`는 `/`를 포함한 모든 문자와 일치합니다.
1. 물음표 `?`는 `/`를 제외한 모든 단일 문자와 일치합니다.
1. 중괄호 `{}`를 사용하여 쉼표 `,`로 구분된 옵션 목록을 일치시킬 수 있습니다.
1. 대괄호 `[]`를 사용하여 문자 집합을 일치시킬 수 있습니다.
1. 백슬래시 `\`를 사용하여 특수 문자를 이스케이프할 수 있습니다(백슬래시 자체를 `\\`로 이스케이프해야 합니다).

예시:
- `https://example.com/*.js`는 `https://example.com/file.js`와 일치하지만 `https://example.com/path/file.js`와는 일치하지 않습니다.
- `https://example.com/\\?page=1`은 `https://example.com/?page=1`과 일치하지만 `https://example.com`과는 일치하지 않습니다.
- `**/v[0-9]*`는 `https://example.com/v1/`과 일치하지만 `https://example.com/vote/`와는 일치하지 않습니다.
- `**/*.js`는 `https://example.com/file.js`와 `https://example.com/path/file.js` 모두와 일치합니다.
- `**/*.{png,jpg,jpeg}`는 모든 이미지 요청과 일치합니다.

중요 사항:
- glob 패턴은 URL의 일부가 아닌 전체 URL과 일치해야 합니다.
- URL 매칭에 glob을 사용하는 경우 프로토콜 및 경로 구분 기호를 포함한 전체 URL 구조를 고려하십시오.
- 더 복잡한 매칭 요구 사항의 경우 glob 패턴 대신 [RegExp]를 사용하는 것이 좋습니다.

## WebSockets

Playwright는 [WebSockets](https://developer.mozilla.org/en-US/docs/Web/API/WebSockets_API) 검사, mocking 및 수정 기능을 기본적으로 지원합니다. WebSockets를 mocking하는 방법은 [API mocking guide](./mock.mdx#mock-websockets)를 참조하십시오.

WebSocket이 생성될 때마다 [page.on('websocket')](/api/class-page.mdx#page-event-web-socket) 이벤트가 발생합니다. 이 이벤트에는 웹 소켓 프레임 검사를 위한 [WebSocket] 인스턴스가 포함되어 있습니다.

```js
page.on('websocket', ws => {
  console.log(`WebSocket opened: ${ws.url()}>`);
  ws.on('framesent', event => console.log(event.payload));
  ws.on('framereceived', event => console.log(event.payload));
  ws.on('close', () => console.log('WebSocket closed'));
});
```

## Missing Network Events and Service Workers

Playwright의 내장 [browserContext.route()](/api/class-browsercontext.mdx#browser-context-route) 및 [page.route()](/api/class-page.mdx#page-route)를 사용하면 테스트에서 요청을 기본적으로 라우팅하고 mocking 및 인터셉션을 수행할 수 있습니다.
1. Playwright의 기본 [browserContext.route()](/api/class-browsercontext.mdx#browser-context-route) 및 [page.route()](/api/class-page.mdx#page-route)를 사용하는 경우 네트워크 이벤트가 누락된 것으로 보이면 [serviceWorkers](/api/class-browser.mdx#browser-new-context-option-service-workers)를 `'block'`으로 설정하여 서비스 워커를 비활성화합니다.
1. MSW(Mock Service Worker)와 같은 mock 도구를 사용하고 있을 수 있습니다. 이 도구는 응답을 mocking하는 데 즉시 사용할 수 있지만 네트워크 요청을 인계하는 자체 서비스 워커를 추가하므로 [browserContext.route()](/api/class-browsercontext.mdx#browser-context-route) 및 [page.route()](/api/class-page.mdx#page-route)에 보이지 않게 됩니다. 네트워크 테스트와 mocking 모두에 관심이 있다면 기본 제공 [browserContext.route()](/api/class-browsercontext.mdx#browser-context-route) 및 [page.route()](/api/class-page.mdx#page-route)를 사용하여 [응답 mocking](#handle-requests)을 수행하는 것이 좋습니다.
1. 테스트 및 네트워크 mocking을 위해 서비스 워커를 단독으로 사용하지 않고 서비스 워커 자체에서 수행한 요청을 라우팅하고 수신하는 데 관심이 있다면 [이 실험적 기능](https://github.com/microsoft/playwright/issues/15684)을 참조하십시오.
