## Introduction

Playwright Test는 테스트를 병렬로 실행합니다. 이를 위해 여러 개의 워커 프로세스를 동시에 실행합니다. 기본적으로 **테스트 파일**은 병렬로 실행됩니다. 단일 파일의 테스트는 동일한 워커 프로세스에서 순서대로 실행됩니다.
- [`test.describe.configure`](#parallelize-tests-in-a-single-file)를 사용하여 **단일 파일의 테스트**를 병렬로 실행하도록 구성할 수 있습니다.
- [testProject.fullyParallel](/api/class-testproject.mdx#test-project-fully-parallel) 또는 [testConfig.fullyParallel](/api/class-testconfig.mdx#test-config-fully-parallel)을 사용하여 **전체 프로젝트**를 구성하여 모든 파일의 모든 테스트를 병렬로 실행하도록 할 수 있습니다.
- 병렬 처리를 **비활성화**하려면 [워커 수를 하나로 제한](#disable-parallelism)하세요.

효율성을 위해 전체 테스트 스위트에서 [병렬 워커 프로세스 수](#limit-workers)를 제어하고 [실패 횟수를 제한](#limit-failures-and-fail-fast)할 수 있습니다.

## Worker processes

모든 테스트는 워커 프로세스에서 실행됩니다. 이러한 프로세스는 OS 프로세스이며 독립적으로 실행되며 테스트 러너에 의해 조정됩니다. 모든 워커는 동일한 환경을 가지며 각자 자신의 브라우저를 시작합니다.

워커 간에 통신할 수 없습니다. Playwright Test는 테스트 속도를 높이기 위해 가능한 한 단일 워커를 재사용하므로 일반적으로 여러 테스트 파일이 단일 워커에서 순차적으로 실행됩니다.

워커는 후속 테스트를 위한 깨끗한 환경을 보장하기 위해 [테스트 실패](./test-retries.mdx#failures) 후 항상 종료됩니다.

## Limit workers

[명령줄](./test-cli.mdx) 또는 [구성 파일](./test-configuration.mdx)에서 병렬 워커 프로세스의 최대 수를 제어할 수 있습니다.

명령줄에서:

```bash
npx playwright test --workers 4
```

구성 파일에서:

```js title="playwright.config.ts"
import { defineConfig } from '@playwright/test';

export default defineConfig({
  // CI에서 워커 수를 제한하고, 로컬에서는 기본값을 사용합니다.
  workers: process.env.CI ? 2 : undefined,
});
```

## Disable parallelism

언제든지 단일 워커만 허용하여 병렬 처리를 비활성화할 수 있습니다. 구성 파일에서 `workers: 1` 옵션을 설정하거나 명령줄에 `--workers=1`을 전달합니다.

```bash
npx playwright test --workers=1
```

## Parallelize tests in a single file

기본적으로 단일 파일의 테스트는 순서대로 실행됩니다. 단일 파일에 많은 독립적인 테스트가 있는 경우 [test.describe.configure()](/api/class-test.mdx#test-describe-configure)를 사용하여 병렬로 실행할 수 있습니다.

병렬 테스트는 별도의 워커 프로세스에서 실행되며 상태 또는 전역 변수를 공유할 수 없습니다. 각 테스트는 `beforeAll` 및 `afterAll`을 포함하여 자체적으로 관련된 모든 후크를 실행합니다.

```js
import { test } from '@playwright/test';

test.describe.configure({ mode: 'parallel' });

test('runs in parallel 1', async ({ page }) => { /* ... */ });
test('runs in parallel 2', async ({ page }) => { /* ... */ });
```

또는 구성 파일에서 모든 테스트를 완전 병렬 모드로 선택할 수 있습니다.

```js title="playwright.config.ts"
import { defineConfig } from '@playwright/test';

export default defineConfig({
  fullyParallel: true,
});
```

몇 개의 프로젝트에 대해서만 완전 병렬 모드를 선택할 수도 있습니다.

```js title="playwright.config.ts"
import { defineConfig } from '@playwright/test';

export default defineConfig({
  // 특정 프로젝트의 모든 파일에서 모든 테스트를 병렬로 실행합니다.
  projects: [
    {
      name: 'chromium',
      use: { ...devices['Desktop Chrome'] },
      fullyParallel: true,
    },
  ]
});
```

## Serial mode

상호 의존적인 테스트를 직렬로 주석 처리할 수 있습니다. 직렬 테스트 중 하나가 실패하면 후속 테스트는 모두 건너뜁니다. 그룹의 모든 테스트는 함께 다시 시도됩니다.

:::note

직렬 모드를 사용하는 것은 권장되지 않습니다. 일반적으로 테스트를 격리하여 독립적으로 실행할 수 있도록 하는 것이 좋습니다.
:::

```js
import { test, type Page } from '@playwright/test';

// 전체 파일을 직렬로 주석 처리합니다.
test.describe.configure({ mode: 'serial' });

let page: Page;

test.beforeAll(async ({ browser }) => {
  page = await browser.newPage();
});

test.afterAll(async () => {
  await page.close();
});

test('runs first', async () => {
  await page.goto('https://playwright.dev/');
});

test('runs second', async () => {
  await page.getByText('Get Started').click();
});
```

## Shard tests between multiple machines

Playwright Test는 테스트 스위트를 분할하여 여러 장비에서 실행할 수 있습니다. 자세한 내용은 [분할 가이드](./test-sharding.mdx)를 참조하세요.

```bash
npx playwright test --shard=2/3
```

## Limit failures and fail fast

`maxFailures` 구성 옵션을 설정하거나 `--max-failures` 명령줄 플래그를 전달하여 전체 테스트 스위트에서 실패한 테스트 수를 제한할 수 있습니다.

"최대 실패"를 설정하여 실행할 때 Playwright Test는 이 횟수의 실패한 테스트에 도달하면 중지하고 아직 실행되지 않은 테스트를 건너뜁니다. 이는 손상된 테스트 스위트에 리소스를 낭비하지 않도록 하는 데 유용합니다.

명령줄 옵션 전달:

```bash
npx playwright test --max-failures=10
```

구성 파일에서 설정:

```js title="playwright.config.ts"
import { defineConfig } from '@playwright/test';

export default defineConfig({
  // 리소스를 절약하기 위해 CI에서 실패 횟수를 제한합니다.
  maxFailures: process.env.CI ? 10 : undefined,
});
```

## Worker index and parallel index

각 워커 프로세스에는 두 개의 ID가 할당됩니다. 1부터 시작하는 고유한 워커 인덱스와 `0`과 `workers - 1` 사이의 병렬 인덱스입니다. 예를 들어 실패 후 워커가 다시 시작되면 새 워커 프로세스는 동일한 `parallelIndex`와 새 `workerIndex`를 가집니다.

환경 변수 `process.env.TEST_WORKER_INDEX` 및 `process.env.TEST_PARALLEL_INDEX`에서 인덱스를 읽거나 [testInfo.workerIndex](/api/class-testinfo.mdx#test-info-worker-index) 및 [testInfo.parallelIndex](/api/class-testinfo.mdx#test-info-parallel-index)를 통해 액세스할 수 있습니다.

### Isolate test data between parallel workers

위에서 언급한 `process.env.TEST_WORKER_INDEX` 또는 [testInfo.workerIndex](/api/class-testinfo.mdx#test-info-worker-index)를 활용하여 다른 워커에서 실행되는 테스트 간에 데이터베이스의 사용자 데이터를 격리할 수 있습니다. 워커에서 실행되는 모든 테스트는 동일한 사용자를 재사용합니다.

[`dbUserName` fixture를 생성](./test-fixtures#creating-a-fixture)하고 테스트 데이터베이스에서 새 사용자를 초기화하는 `playwright/fixtures.ts` 파일을 만듭니다. [testInfo.workerIndex](/api/class-testinfo.mdx#test-info-worker-index)를 사용하여 워커를 구분합니다.

```js title="playwright/fixtures.ts"
import { test as baseTest, expect } from '@playwright/test';
// 테스트 데이터베이스에서 사용자를 관리하기 위한 프로젝트 유틸리티를 가져옵니다.
import { createUserInTestDatabase, deleteUserFromTestDatabase } from './my-db-utils';

export * from '@playwright/test';
export const test = baseTest.extend<{}, { dbUserName: string }>({
  // 워커에 고유한 db 사용자 이름을 반환합니다.
  dbUserName: [async ({ }, use) => {
    // workerIndex를 각 워커에 대한 고유 식별자로 사용합니다.
    const userName = `user-${test.info().workerIndex}`;
    // 데이터베이스에서 사용자를 초기화합니다.
    await createUserInTestDatabase(userName);
    await use(userName);
    // 테스트가 완료되면 정리합니다.
    await deleteUserFromTestDatabase(userName);
  }, { scope: 'worker' }],
});
```

이제 각 테스트 파일은 `@playwright/test` 대신 픽스처 파일에서 `test`를 가져와야 합니다.

```js title="tests/example.spec.ts"
// 중요: 픽스처를 가져옵니다.
import { test, expect } from '../playwright/fixtures';

test('test', async ({ dbUserName }) => {
  // 테스트에서 사용자 이름을 사용합니다.
});
```

## Control test order

Playwright Test는 [단일 파일에서 테스트를 병렬화](#parallelize-tests-in-a-single-file)하지 않는 한 선언 순서대로 단일 파일의 테스트를 실행합니다.

Playwright Test는 기본적으로 테스트 파일을 병렬로 실행하기 때문에 파일 간 테스트 실행 순서에 대한 보장은 없습니다. 그러나 [병렬 처리를 비활성화](#disable-parallelism)하면 파일을 알파벳순으로 이름을 지정하거나 "테스트 목록" 파일을 사용하여 테스트 순서를 제어할 수 있습니다.

### Sort test files alphabetically

**병렬 테스트 실행을 비활성화**하면 Playwright Test는 테스트 파일을 알파벳순으로 실행합니다. 예를 들어 `001-user-signin-flow.spec.ts`, `002-create-new-document.spec.ts` 등과 같은 명명 규칙을 사용하여 테스트 순서를 제어할 수 있습니다.

### Use a "test list" file

:::warning

테스트 목록은 권장되지 않으며 최선을 다해 지원됩니다. VS Code Extension 및 추적과 같은 일부 기능은 테스트 목록에서 제대로 작동하지 않을 수 있습니다.
:::

여러 파일에서 도우미 함수에 테스트를 넣을 수 있습니다. 테스트가 파일에 직접 정의되지 않고 래퍼 함수에 정의되는 다음 예를 고려하십시오.

```js title="feature-a.spec.ts"
import { test, expect } from '@playwright/test';

export default function createTests() {
  test('feature-a example test', async ({ page }) => {
    // ... 테스트가 여기에 들어갑니다.
  });
}

```

```js title="feature-b.spec.ts"
import { test, expect } from '@playwright/test';

export default function createTests() {
  test.use({ viewport: { width: 500, height: 500 } });

  test('feature-b example test', async ({ page }) => {
    // ... 테스트가 여기에 들어갑니다.
  });
}
```

먼저 `feature-b` 테스트를 실행한 다음 `feature-a` 테스트를 실행하는 테스트 순서를 제어하는 테스트 목록 파일을 만들 수 있습니다. 각 테스트 파일이 테스트가 정의된 함수를 호출하는 `test.describe()` 블록으로 래핑되는 방식에 주목하세요. 이렇게 하면 `test.use()` 호출이 단일 파일의 테스트에만 영향을 미칩니다.

```js title="test.list.ts"
import { test } from '@playwright/test';
import featureBTests from './feature-b.spec.ts';
import featureATests from './feature-a.spec.ts';

test.describe(featureBTests);
test.describe(featureATests);
```

이제 워커를 하나로 설정하여 **병렬 실행을 비활성화**하고 테스트 목록 파일을 지정합니다.

```js title="playwright.config.ts"
import { defineConfig } from '@playwright/test';

export default defineConfig({
  workers: 1,
  testMatch: 'test.list.ts',
});
```

:::note
도우미 파일에 테스트를 직접 정의하지 마세요. 이렇게 하면 테스트가 이제 `import`/`require` 문의 순서에 의존하기 때문에 예기치 않은 결과가 발생할 수 있습니다. 대신 위 예와 같이 테스트 목록 파일에서 명시적으로 호출될 함수로 테스트를 래핑합니다.
:::
