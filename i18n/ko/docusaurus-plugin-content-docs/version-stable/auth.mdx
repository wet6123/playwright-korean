## 소개

Playwright는 [브라우저 컨텍스트](./browser-contexts.mdx)라고 하는 격리된 환경에서 테스트를 실행합니다. 이러한 격리 모델은 재현성을 향상시키고 테스트 실패의 연쇄 작용을 방지합니다. 테스트는 기존 인증 상태를 로드할 수 있습니다. 이렇게 하면 모든 테스트에서 인증할 필요가 없어 테스트 실행 속도가 빨라집니다.

## 핵심 개념

선택한 인증 전략에 관계없이 인증된 브라우저 상태를 파일 시스템에 저장할 가능성이 높습니다.

`playwright/.auth` 디렉터리를 만들고 `.gitignore`에 추가하는 것이 좋습니다. 인증 루틴은 인증된 브라우저 상태를 생성하고 이 `playwright/.auth` 디렉터리의 파일에 저장합니다. 나중에 테스트는 이 상태를 재사용하고 이미 인증된 상태로 시작합니다.

## 기본: 모든 테스트에서 공유 계정 사용

이는 **서버 측 상태가 없는** 테스트에 **권장되는** 방법입니다. **설정 프로젝트**에서 한 번 인증하고 인증 상태를 저장한 다음 이를 재사용하여 각 테스트를 이미 인증된 상태로 부트스트랩합니다.

**사용 시기**

- 모든 테스트가 동일한 계정으로 동시에 실행되어도 서로 영향을 미치지 않는다고 생각할 수 있는 경우.

**사용하지 않는 시기**

- 테스트가 서버 측 상태를 수정하는 경우. 예를 들어, 한 테스트는 설정 페이지의 렌더링을 확인하고 다른 테스트는 설정을 변경하며 테스트를 병렬로 실행하는 경우. 이 경우 테스트는 다른 계정을 사용해야 합니다.
- 인증이 브라우저별인 경우.

**세부 정보**

다른 모든 테스트를 위해 인증된 브라우저 상태를 준비하는 `tests/auth.setup.ts`를 만듭니다.

```js title="tests/auth.setup.ts"
import { test as setup, expect } from '@playwright/test';
import path from 'path';

const authFile = path.join(__dirname, '../playwright/.auth/user.json');

setup('authenticate', async ({ page }) => {
  // 인증 단계를 수행합니다. 이러한 작업을 사용자 고유의 작업으로 바꿉니다.
  await page.goto('https://github.com/login');
  await page.getByLabel('Username or email address').fill('username');
  await page.getByLabel('Password').fill('password');
  await page.getByRole('button', { name: 'Sign in' }).click();
  // 페이지가 쿠키를 수신할 때까지 기다립니다.
  //
  // 로그인 흐름이 여러 리디렉션 과정에서 쿠키를 설정하는 경우가 있습니다.
  // 쿠키가 실제로 설정되었는지 확인하려면 최종 URL을 기다립니다.
  await page.waitForURL('https://github.com/');
  // 또는 모든 쿠키가 설정된 상태에 도달할 때까지 기다릴 수 있습니다.
  await expect(
    page.getByRole('button', { name: 'View profile and more' })
  ).toBeVisible();

  // 인증 단계의 끝.

  await page.context().storageState({ path: authFile });
});
```

구성에서 새 `setup` 프로젝트를 만들고 모든 테스트 프로젝트에 대한 [종속성](./test-projects.mdx#dependencies)으로 선언합니다. 이 프로젝트는 항상 모든 테스트 전에 실행되고 인증됩니다. 모든 테스트 프로젝트는 인증된 상태를 `storageState`로 사용해야 합니다.

```js title="playwright.config.ts"
import { defineConfig, devices } from '@playwright/test';

export default defineConfig({
  projects: [
    // 설정 프로젝트
    { name: 'setup', testMatch: /.*\.setup\.ts/ },

    {
      name: 'chromium',
      use: {
        ...devices['Desktop Chrome'],
        // 준비된 인증 상태를 사용합니다.
        storageState: 'playwright/.auth/user.json',
      },
      dependencies: ['setup'],
    },

    {
      name: 'firefox',
      use: {
        ...devices['Desktop Firefox'],
        // 준비된 인증 상태를 사용합니다.
        storageState: 'playwright/.auth/user.json',
      },
      dependencies: ['setup'],
    },
  ],
});
```

구성에서 `storageState`를 지정했기 때문에 테스트는 이미 인증된 상태로 시작됩니다.

```js title="tests/example.spec.ts"
import { test } from '@playwright/test';

test('test', async ({ page }) => {
  // 페이지가 인증됨
});
```

저장된 상태가 만료되면 삭제해야 합니다. 테스트 실행 간에 상태를 유지할 필요가 없는 경우, 모든 테스트 실행 전에 자동으로 정리되는 [testProject.outputDir](/api/class-testproject.mdx#test-project-output-dir) 아래에 브라우저 상태를 작성합니다.

### UI 모드에서 인증

UI 모드는 테스트 속도를 향상시키기 위해 기본적으로 `setup` 프로젝트를 실행하지 않습니다. 기존 인증이 만료될 때마다 `auth.setup.ts`를 수동으로 실행하여 인증하는 것이 좋습니다.

먼저 [필터에서 `setup` 프로젝트를 활성화](./test-ui-mode#filtering-tests)한 다음 `auth.setup.ts` 파일 옆에 있는 삼각형 버튼을 클릭하고 필터에서 `setup` 프로젝트를 다시 비활성화합니다.

## 중간: 병렬 worker당 하나의 계정

이는 **서버 측 상태를 수정하는** 테스트에 **권장되는** 방법입니다. Playwright에서 worker 프로세스는 병렬로 실행됩니다. 이 방법에서는 각 병렬 worker가 한 번 인증됩니다. worker가 실행하는 모든 테스트는 동일한 인증 상태를 재사용합니다. 각 병렬 worker당 하나씩 여러 테스트 계정이 필요합니다.

**사용 시기**

- 테스트가 공유 서버 측 상태를 수정하는 경우. 예를 들어, 한 테스트는 설정 페이지의 렌더링을 확인하고 다른 테스트는 설정을 변경하는 경우.

**사용하지 않는 시기**

- 테스트가 공유 서버 측 상태를 수정하지 않는 경우. 이 경우 모든 테스트는 단일 공유 계정을 사용할 수 있습니다.

**세부 정보**

고유한 계정으로 각 [worker 프로세스](./test-parallel.mdx#worker-processes)당 한 번 인증합니다.

worker당 한 번 인증하기 위해 `storageState` 고정 장치를 [재정의](./test-fixtures.mdx#overriding-fixtures)하는 `playwright/fixtures.ts` 파일을 만듭니다. worker를 구별하려면 [testInfo.parallelIndex](/api/class-testinfo.mdx#test-info-parallel-index)를 사용합니다.

```js title="playwright/fixtures.ts"
import { test as baseTest, expect } from '@playwright/test';
import fs from 'fs';
import path from 'path';

export * from '@playwright/test';
export const test = baseTest.extend<{}, { workerStorageState: string }>({
  // 이 worker의 모든 테스트에 동일한 저장소 상태를 사용합니다.
  storageState: ({ workerStorageState }, use) => use(workerStorageState),

  // worker 범위 고정 장치를 사용하여 worker당 한 번 인증합니다.
  workerStorageState: [async ({ browser }, use) => {
    // parallelIndex를 각 worker의 고유 식별자로 사용합니다.
    const id = test.info().parallelIndex;
    const fileName = path.resolve(test.info().project.outputDir, `.auth/${id}.json`);

    if (fs.existsSync(fileName)) {
      // 기존 인증 상태가 있는 경우 재사용합니다.
      await use(fileName);
      return;
    }

    // 중요: 저장소 상태를 설정 해제하여 깨끗한 환경에서 인증하는지 확인합니다.
    const page = await browser.newPage({ storageState: undefined });


    // 고유 계정을 가져옵니다(예: 새 계정 생성).
    // 또는 테스트를 위해 미리 생성된 계정 목록을 가질 수 있습니다.
    // 여러 팀 구성원이 동시에 테스트를 실행할 수 있도록 계정이 고유한지 확인합니다.
    const account = await acquireAccount(id);

    // 인증 단계를 수행합니다. 이러한 작업을 사용자 고유의 작업으로 바꿉니다.
    await page.goto('https://github.com/login');
    await page.getByLabel('Username or email address').fill(account.username);
    await page.getByLabel('Password').fill(account.password);
    await page.getByRole('button', { name: 'Sign in' }).click();
    // 페이지가 쿠키를 수신할 때까지 기다립니다.
    //
    // 로그인 흐름이 여러 리디렉션 과정에서 쿠키를 설정하는 경우가 있습니다.
    // 쿠키가 실제로 설정되었는지 확인하려면 최종 URL을 기다립니다.
    await page.waitForURL('https://github.com/');
    // 또는 모든 쿠키가 설정된 상태에 도달할 때까지 기다릴 수 있습니다.
    await expect(page.getByRole('button', { name: 'View profile and more' })).toBeVisible();

    // 인증 단계의 끝.

    await page.context().storageState({ path: fileName });
    await page.close();
    await use(fileName);
  }, { scope: 'worker' }],
});
```

이제 각 테스트 파일은 `@playwright/test` 대신 고정 장치 파일에서 `test`를 가져와야 합니다. 구성에서는 변경할 필요가 없습니다.

```js title="tests/example.spec.ts"
// 중요: 고정 장치를 가져옵니다.
import { test, expect } from '../playwright/fixtures';

test('test', async ({ page }) => {
  // 페이지가 인증됨
});
```

## 고급 시나리오

### API 요청으로 인증

**사용 시기**

- 웹 애플리케이션이 앱 UI와 상호 작용하는 것보다 쉽고 빠른 API를 통한 인증을 지원하는 경우.

**세부 정보**

[APIRequestContext]로 API 요청을 보낸 다음 평소처럼 인증된 상태를 저장합니다.

[설정 프로젝트](#basic-shared-account-in-all-tests)에서:

```js title="tests/auth.setup.ts"
import { test as setup } from '@playwright/test';

const authFile = 'playwright/.auth/user.json';

setup('authenticate', async ({ request }) => {
  // 인증 요청을 보냅니다. 사용자 고유의 것으로 바꿉니다.
  await request.post('https://github.com/login', {
    form: {
      user: 'user',
      password: 'password',
    },
  });
  await request.storageState({ path: authFile });
});
```

또는 [worker 고정 장치](#moderate-one-account-per-parallel-worker)에서:

```js title="playwright/fixtures.ts"
import { test as baseTest, request } from '@playwright/test';
import fs from 'fs';
import path from 'path';

export * from '@playwright/test';
export const test = baseTest.extend<{}, { workerStorageState: string }>({
  // 이 worker의 모든 테스트에 동일한 저장소 상태를 사용합니다.
  storageState: ({ workerStorageState }, use) => use(workerStorageState),

  // worker 범위 고정 장치를 사용하여 worker당 한 번 인증합니다.
  workerStorageState: [async ({}, use) => {
    // parallelIndex를 각 worker의 고유 식별자로 사용합니다.
    const id = test.info().parallelIndex;
    const fileName = path.resolve(test.info().project.outputDir, `.auth/${id}.json`);

    if (fs.existsSync(fileName)) {
      // 기존 인증 상태가 있는 경우 재사용합니다.
      await use(fileName);
      return;
    }

    // 중요: 저장소 상태를 설정 해제하여 깨끗한 환경에서 인증하는지 확인합니다.
    const context = await request.newContext({ storageState: undefined });

    // 고유 계정을 가져옵니다(예: 새 계정 생성).
    // 또는 테스트를 위해 미리 생성된 계정 목록을 가질 수 있습니다.
    // 여러 팀 구성원이 동시에 테스트를 실행할 수 있도록 계정이 고유한지 확인합니다.
    const account = await acquireAccount(id);

    // 인증 요청을 보냅니다. 사용자 고유의 것으로 바꿉니다.
    await context.post('https://github.com/login', {
      form: {
        'user': 'user',
        'password': 'password'
      }
    });

    await context.storageState({ path: fileName });
    await context.dispose();
    await use(fileName);
  }, { scope: 'worker' }],
});
```

### 여러 로그인된 역할

**사용 시기**

- 엔드 투 엔드 테스트에 두 개 이상의 역할이 있지만 모든 테스트에서 계정을 재사용할 수 있는 경우.

**세부 정보**

설정 프로젝트에서 여러 번 인증합니다.

```js title="tests/auth.setup.ts"
const adminFile = 'playwright/.auth/admin.json';

setup('관리자로 인증', async ({ page }) => {

  // 로그인 흐름이 여러 리디렉션 과정에서 쿠키를 설정하는 경우가 있습니다.

const userFile = 'playwright/.auth/user.json';

    page.getByRole('button', { name: 'View profile and more' })

```

그런 다음 구성에서 설정하는 **대신** 각 테스트 파일 또는 테스트 그룹에 대해 `storageState`를 지정합니다.

```js title="tests/example.spec.ts"
import { test } from '@playwright/test';

test.use({ storageState: 'playwright/.auth/admin.json' });

test('관리자 테스트', async ({ page }) => {
  // 페이지가 관리자로 인증됨
});

test.describe(() => {
  test.use({ storageState: 'playwright/.auth/user.json' });

  test('사용자 테스트', async ({ page }) => {
    // 페이지가 사용자로 인증됨
  });
});
```

[UI 모드에서 인증](#authenticating-in-ui-mode)에 대해서도 참조하십시오.

### 여러 역할을 함께 테스트

**사용 시기**

- 단일 테스트에서 여러 인증된 역할이 서로 어떻게 상호 작용하는지 테스트해야 하는 경우.

**세부 정보**

동일한 테스트에서 서로 다른 저장소 상태를 가진 여러 [BrowserContext] 및 [Page]를 사용합니다.

```js title="tests/example.spec.ts"
import { test } from '@playwright/test';

test('관리자 및 사용자', async ({ browser }) => {
  // adminContext 및 adminPage를 포함한 내부의 모든 페이지는 "admin"으로 로그인됩니다.
  const adminContext = await browser.newContext({
    storageState: 'playwright/.auth/admin.json',
  });
  const adminPage = await adminContext.newPage();

  // userContext 및 userPage를 포함한 내부의 모든 페이지는 "user"로 로그인됩니다.
  const userContext = await browser.newContext({
    storageState: 'playwright/.auth/user.json',
  });
  const userPage = await userContext.newPage();

  // ... adminPage와 userPage 모두와 상호 작용 ...

  await adminContext.close();
  await userContext.close();
});
```

### POM 고정 장치를 사용하여 여러 역할 테스트

**사용 시기**

- 단일 테스트에서 여러 인증된 역할이 서로 어떻게 상호 작용하는지 테스트해야 하는 경우.

**세부 정보**

각 역할로 인증된 페이지를 제공하는 고정 장치를 도입할 수 있습니다.

아래는 두 개의 [페이지 객체 모델](./pom.mdx) - 관리자 POM 및 사용자 POM에 대한 [고정 장치를 만드는](./test-fixtures.mdx#creating-a-fixture) 예입니다. `adminStorageState.json` 및 `userStorageState.json` 파일이 전역 설정에서 생성되었다고 가정합니다.

```js title="playwright/fixtures.ts"
import { test as base, type Page, type Locator } from '@playwright/test';

// "admin" 페이지에 대한 페이지 객체 모델입니다.
// 여기에 관리자 페이지에 특정된 로케이터와 도우미 메서드를 추가할 수 있습니다.
class AdminPage {
  // "admin"으로 로그인된 페이지입니다.
  page: Page;

  // "Welcome, Admin" 인사말을 가리키는 예시 로케이터입니다.
  greeting: Locator;

  constructor(page: Page) {
    this.page = page;
    this.greeting = page.locator('#greeting');
  }
}

// "user" 페이지에 대한 페이지 객체 모델입니다.
// 여기에 사용자 페이지에 특정된 로케이터와 도우미 메서드를 추가할 수 있습니다.
class UserPage {
  // "user"로 로그인된 페이지입니다.
  page: Page;

  // "Welcome, User" 인사말을 가리키는 예시 로케이터입니다.
  greeting: Locator;

  constructor(page: Page) {
    this.page = page;
    this.greeting = page.locator('#greeting');
  }
}

// 고정 장치의 유형을 선언합니다.
type MyFixtures = {
  adminPage: AdminPage,
  userPage: UserPage,
};

export * from '@playwright/test';
export const test =
  base.extend <
  MyFixtures >
  {
    adminPage: async ({ browser }, use) => {
      const context = await browser.newContext({
        storageState: 'playwright/.auth/admin.json',
      });
      const adminPage = new AdminPage(await context.newPage());
      await use(adminPage);
      await context.close();
    },
    userPage: async ({ browser }, use) => {
      const context = await browser.newContext({
        storageState: 'playwright/.auth/user.json',
      });
      const userPage = new UserPage(await context.newPage());
      await use(userPage);
      await context.close();
    },
  };
```

```js title="tests/example.spec.ts"
  // ... adminPage와 userPage 모두와 상호 작용 ...
  await expect(adminPage.greeting).toHaveText('Welcome, Admin');
```

### 세션 스토리지

인증된 상태를 재사용하면 [쿠키](https://developer.mozilla.org/ko/docs/Web/HTTP/Cookies), [로컬 스토리지](https://developer.mozilla.org/ko/docs/Web/API/Storage) 및 [IndexedDB](https://developer.mozilla.org/ko/docs/Web/API/IndexedDB_API) 기반 인증이 포함됩니다. 드물게 [세션 스토리지](https://developer.mozilla.org/ko/docs/Web/API/Window/sessionStorage)가 로그인 상태와 관련된 정보를 저장하는 데 사용됩니다. 세션 스토리지는 특정 도메인에만 해당되며 페이지 로드 간에 유지되지 않습니다. Playwright는 세션 스토리지를 유지하기 위한 API를 제공하지 않지만 다음 코드 조각을 사용하여 세션 스토리지를 저장/로드할 수 있습니다.

```js
// 세션 스토리지를 가져와서 환경 변수로 저장합니다.
const sessionStorage = await page.evaluate(() =>
  JSON.stringify(sessionStorage)
);
fs.writeFileSync('playwright/.auth/session.json', sessionStorage, 'utf-8');

// 새 컨텍스트에서 세션 스토리지를 설정합니다.
const sessionStorage = JSON.parse(
  fs.readFileSync('playwright/.auth/session.json', 'utf-8')
);
await context.addInitScript((storage) => {
  if (window.location.hostname === 'example.com') {
    for (const [key, value] of Object.entries(storage))
      window.sessionStorage.setItem(key, value);
  }
}, sessionStorage);
```
