페이지가 네트워크 리소스에 대한 요청을 보낼 때마다 [Page]에서 다음 이벤트 시퀀스가 발생합니다.
* [page.on('request')](/api/class-page.mdx#page-event-request) 페이지에서 요청이 발생할 때 발생합니다.
* [page.on('response')](/api/class-page.mdx#page-event-response) 요청에 대한 응답 상태 및 헤더가 수신될 때 발생합니다.
* [page.on('requestfinished')](/api/class-page.mdx#page-event-request-finished) 응답 본문이 다운로드되고 요청이 완료되면 발생합니다.

요청이 특정 시점에서 실패하면 `'requestfinished'` 이벤트 (및 'response' 이벤트 대신) 대신 [page.on('requestfailed')](/api/class-page.mdx#page-event-request-failed) 이벤트가 발생합니다.

:::note

404 또는 503과 같은 HTTP 오류 응답은 HTTP 관점에서 여전히 성공적인 응답이므로 요청은 `'requestfinished'` 이벤트로 완료됩니다.
:::

요청이 'redirect' 응답을 받으면 요청은 `requestfinished` 이벤트로 성공적으로 완료되고 리디렉션된 URL로 새 요청이 발행됩니다.

### failure {#request-failure}

<font size="2" style={{position: "relative", top: "-20px"}}>Added before v1.9</font><x-search>request.failure</x-search>

이 메서드는 `requestfailed` 이벤트에서 보고된 대로 이 요청이 실패하지 않는 한 `null`을 반환합니다.

**Usage**

실패한 모든 요청 로깅의 예:

```js
page.on('requestfailed', request => {
  console.log(request.url() + ' ' + request.failure().errorText);
});
```

**Returns**
- [null] | [Object]<a aria-hidden="true" tabIndex="-1" class="list-anchor-link" id="request-failure-return"/><a href="#request-failure-return" class="list-anchor">#</a>
  - `errorText` [string]
    
    사람이 읽을 수 있는 오류 메시지 (예: `'net::ERR_FAILED'`).

### frame {#request-frame}

<font size="2" style={{position: "relative", top: "-20px"}}>Added before v1.9</font><x-search>request.frame</x-search>

이 요청을 시작한 [Frame]을 반환합니다.

**Usage**

```js
const frameUrl = request.frame().url();
```

**Returns**
- [Frame]<a aria-hidden="true" tabIndex="-1" class="list-anchor-link" id="request-frame-return"/><a href="#request-frame-return" class="list-anchor">#</a>

**Details**

경우에 따라 프레임을 사용할 수 없으며 이 메서드는 예외를 발생시킵니다.
* 요청이 서비스 워커에서 시작될 때. `request.serviceWorker()`를 사용하여 확인할 수 있습니다.
* 해당 프레임이 생성되기 전에 탐색 요청이 발행된 경우. [request.isNavigationRequest()](/api/class-request.mdx#request-is-navigation-request)를 사용하여 확인할 수 있습니다.

다음은 모든 경우를 처리하는 예입니다.

```js
if (request.serviceWorker())
  console.log(`request ${request.url()} from a service worker`);
else if (request.isNavigationRequest())
  console.log(`request ${request.url()} is a navigation request`);
else
  console.log(`request ${request.url()} from a frame ${request.frame().url()}`);
```

### redirectedFrom {#request-redirected-from}

<font size="2" style={{position: "relative", top: "-20px"}}>Added before v1.9</font><x-search>request.redirectedFrom</x-search>

서버에서 이 요청으로 리디렉션된 요청 (있는 경우).

서버가 리디렉션으로 응답하면 Playwright는 새 [Request] 객체를 만듭니다. 두 요청은 `redirectedFrom()` 및 `redirectedTo()` 메서드로 연결됩니다. 서버 리디렉션이 여러 번 발생한 경우 `redirectedFrom()`을 반복적으로 호출하여 전체 리디렉션 체인을 구성할 수 있습니다.

**Usage**

예를 들어 웹 사이트 `http://example.com`이 `https://example.com`으로 리디렉션되는 경우:

```js
const response = await page.goto('http://example.com');
console.log(response.request().redirectedFrom().url()); // 'http://example.com'
```

웹 사이트 `https://google.com`에 리디렉션이 없는 경우:

```js
const response = await page.goto('https://google.com');
console.log(response.request().redirectedFrom()); // null
```

**Returns**
- [null] | [Request]<a aria-hidden="true" tabIndex="-1" class="list-anchor-link" id="request-redirected-from-return"/><a href="#request-redirected-from-return" class="list-anchor">#</a>

### timing {#request-timing}

<font size="2" style={{position: "relative", top: "-20px"}}>Added before v1.9</font><x-search>request.timing</x-search>

지정된 요청에 대한 리소스 타이밍 정보를 반환합니다. 대부분의 타이밍 값은 응답 시 사용할 수 있으며 `responseEnd`는 요청이 완료되면 사용할 수 있습니다. [Resource Timing API](https://developer.mozilla.org/en-US/docs/Web/API/PerformanceResourceTiming)에서 자세한 정보를 찾으십시오.

**Usage**

```js
const requestFinishedPromise = page.waitForEvent('requestfinished');
await page.goto('http://example.com');
const request = await requestFinishedPromise;
console.log(request.timing());
```

**Returns**
- [Object]<a aria-hidden="true" tabIndex="-1" class="list-anchor-link" id="request-timing-return"/><a href="#request-timing-return" class="list-anchor">#</a>
  - `startTime` [number]
    
    1970년 1월 1일 00:00:00 UTC 이후 경과된 밀리초 단위의 요청 시작 시간
  - `domainLookupStart` [number]
    
    브라우저가 리소스에 대한 도메인 이름 조회를 시작하기 직전 시간입니다. 값은 `startTime`을 기준으로 밀리초 단위로 제공되며 사용할 수 없는 경우 -1입니다.
  - `domainLookupEnd` [number]
    
    브라우저가 리소스에 대한 도메인 이름 조회를 시작한 직후 시간입니다. 값은 `startTime`을 기준으로 밀리초 단위로 제공되며 사용할 수 없는 경우 -1입니다.
  - `connectStart` [number]
    
    사용자 에이전트가 리소스를 검색하기 위해 서버에 대한 연결 설정을 시작하기 직전 시간입니다. 값은 `startTime`을 기준으로 밀리초 단위로 제공되며 사용할 수 없는 경우 -1입니다.
  - `secureConnectionStart` [number]
    
    브라우저가 현재 연결을 보호하기 위해 핸드셰이크 프로세스를 시작하기 직전 시간입니다. 값은 `startTime`을 기준으로 밀리초 단위로 제공되며 사용할 수 없는 경우 -1입니다.
  - `connectEnd` [number]
    
    사용자 에이전트가 리소스를 검색하기 위해 서버에 대한 연결 설정을 시작하기 직전 시간입니다. 값은 `startTime`을 기준으로 밀리초 단위로 제공되며 사용할 수 없는 경우 -1입니다.
  - `requestStart` [number]
    
    브라우저가 서버, 캐시 또는 로컬 리소스에서 리소스를 요청하기 직전 시간입니다. 값은 `startTime`을 기준으로 밀리초 단위로 제공되며 사용할 수 없는 경우 -1입니다.
  - `responseStart` [number]
    
    브라우저가 서버, 캐시 또는 로컬 리소스에서 응답의 첫 번째 바이트를 받은 직후 시간입니다. 값은 `startTime`을 기준으로 밀리초 단위로 제공되며 사용할 수 없는 경우 -1입니다.
  - `responseEnd` [number]
    
    브라우저가 리소스의 마지막 바이트를 받은 직후 또는 전송 연결이 닫히기 직전 중 더 빠른 시간입니다. 값은 `startTime`을 기준으로 밀리초 단위로 제공되며 사용할 수 없는 경우 -1입니다.
