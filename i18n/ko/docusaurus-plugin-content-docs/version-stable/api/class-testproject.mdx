Playwright Test는 여러 테스트 프로젝트를 동시에 실행하는 것을 지원합니다. 이는 여러 구성에서 테스트를 실행하는 데 유용합니다. 예를 들어, 여러 브라우저에 대해 테스트를 실행하는 것을 고려해 보세요. 이 타입은 구성 파일에서 프로젝트의 형식을 설명하며, 런타임 시에 해결된 구성 매개변수에 접근하려면 [FullProject]를 사용하세요.

`TestProject`는 단일 프로젝트에 특정한 구성을 캡슐화합니다. 프로젝트는 [구성 파일](../test-configuration.mdx)에 지정된 [testConfig.projects](/api/class-testconfig.mdx#test-config-projects)에서 구성됩니다. [TestProject]의 모든 속성은 최상위 [TestConfig]에서 사용할 수 있으며, 이 경우 모든 프로젝트 간에 공유됩니다.

다음은 데스크톱 및 모바일 버전 모두에서 Chromium, Firefox 및 WebKit의 모든 테스트를 실행하는 구성의 예입니다.

```js title="playwright.config.ts"
import { defineConfig, devices } from '@playwright/test';

export default defineConfig({
  // 모든 프로젝트에 공유되는 옵션입니다.
  timeout: 30000,
  use: {
    ignoreHTTPSErrors: true,
  },

  // 각 프로젝트에 특정한 옵션입니다.
  projects: [
    {
      name: 'chromium',
      use: devices['Desktop Chrome'],
    },
    {
      name: 'firefox',
      use: devices['Desktop Firefox'],
    },
    {
      name: 'webkit',
      use: devices['Desktop Safari'],
    },
    {
      name: 'Mobile Chrome',
      use: devices['Pixel 5'],
    },
    {
      name: 'Mobile Safari',
      use: devices['iPhone 12'],
    },
  ],
});
```

## Properties

### dependencies {#test-project-dependencies}

<font size="2" style={{position: "relative", top: "-20px"}}>Added in: v1.31</font><x-search>testProject.dependencies</x-search>

이 프로젝트의 모든 테스트가 실행되기 전에 실행해야 하는 프로젝트 목록입니다. 종속성은 모든 작업이 테스트 형태로 이루어지도록 전역 설정 작업을 구성하는 데 유용할 수 있습니다. `--no-deps` 인수를 전달하면 종속성이 무시되고 지정되지 않은 것처럼 동작합니다.

종속성을 사용하면 전역 설정이 추적 및 기타 아티팩트를 생성하고, 테스트 보고서에서 설정 단계를 확인하는 등의 작업을 수행할 수 있습니다.

**Usage**

```js title="playwright.config.ts"
import { defineConfig } from '@playwright/test';

export default defineConfig({
  projects: [
    {
      name: 'setup',
      testMatch: /global.setup\.ts/,
    },
    {
      name: 'chromium',
      use: devices['Desktop Chrome'],
      dependencies: ['setup'],
    },
    {
      name: 'firefox',
      use: devices['Desktop Firefox'],
      dependencies: ['setup'],
    },
    {
      name: 'webkit',
      use: devices['Desktop Safari'],
      dependencies: ['setup'],
    },
  ],
});
```

**Type**
- [Array]&lt;[string]&gt;

### expect {#test-project-expect}

<font size="2" style={{position: "relative", top: "-20px"}}>Added in: v1.10</font><x-search>testProject.expect</x-search>

`expect` 어설션 라이브러리에 대한 구성입니다.

모든 프로젝트에 대해 이 옵션을 변경하려면 [testConfig.expect](/api/class-testconfig.mdx#test-config-expect)를 사용하세요.

**Usage**

```js
testProject.expect
```

**Type**
- [Object]
  - `timeout` [number] *(optional)*
    
    비동기 expect 매처에 대한 기본 제한 시간(밀리초)이며, 기본값은 5000ms입니다.
  - `toHaveScreenshot` [Object] *(optional)*
    - `threshold` [number] *(optional)*
      
      비교된 이미지에서 동일한 픽셀 간에 허용 가능한 인지된 색상 차이로, `0`(엄격)과 `1`(관대) 사이의 값을 가집니다. `"pixelmatch"` 비교기는 [YIQ 색 공간](https://en.wikipedia.org/wiki/YIQ)에서 색상 차이를 계산하고 기본적으로 `threshold` 값을 `0.2`로 설정합니다.
    - `maxDiffPixels` [number] *(optional)*
      
      서로 다를 수 있는 허용 가능한 픽셀 양이며, 기본적으로 설정되지 않습니다.
    - `maxDiffPixelRatio` [number] *(optional)*
      
      전체 픽셀 양에 대한 서로 다른 픽셀의 허용 가능한 비율로, `0`과 `1` 사이의 값을 가집니다. 기본적으로 설정되지 않습니다.
    - `animations` "allow" | "disabled" *(optional)*
      
      [page.screenshot()](/api/class-page.mdx#page-screenshot)에서 [animations](/api/class-page.mdx#page-screenshot-option-animations)를 참조하세요. 기본값은 `"disabled"`입니다.
    - `caret` "hide" | "initial" *(optional)*
      
      [page.screenshot()](/api/class-page.mdx#page-screenshot)에서 [caret](/api/class-page.mdx#page-screenshot-option-caret)을 참조하세요. 기본값은 `"hide"`입니다.
    - `scale` "css" | "device" *(optional)*
      
      [page.screenshot()](/api/class-page.mdx#page-screenshot)에서 [scale](/api/class-page.mdx#page-screenshot-option-scale)을 참조하세요. 기본값은 `"css"`입니다.
    - `stylePath` [string] | [Array]&lt;[string]&gt; *(optional)*
      
      [page.screenshot()](/api/class-page.mdx#page-screenshot)에서 [style](/api/class-page.mdx#page-screenshot-option-style)을 참조하세요.
    - `pathTemplate` [string] *(optional)*
      
      스크린샷의 위치를 제어하는 템플릿입니다. 자세한 내용은 [testProject.snapshotPathTemplate](/api/class-testproject.mdx#test-project-snapshot-path-template)를 참조하세요.
    
    [expect(page).toHaveScreenshot()](/api/class-pageassertions.mdx#page-assertions-to-have-screenshot-1) 메서드에 대한 구성입니다.
  - `toMatchAriaSnapshot` [Object] *(optional)*
    - `pathTemplate` [string] *(optional)*
      
      Aria 스냅샷의 위치를 제어하는 템플릿입니다. 자세한 내용은 [testProject.snapshotPathTemplate](/api/class-testproject.mdx#test-project-snapshot-path-template)를 참조하세요.
    
    [expect(locator).toMatchAriaSnapshot()](/api/class-locatorassertions.mdx#locator-assertions-to-match-aria-snapshot-2) 메서드에 대한 구성입니다.
  - `toMatchSnapshot` [Object] *(optional)*
    - `threshold` [number] *(optional)*
      
      비교된 이미지에서 동일한 픽셀 간에 허용 가능한 인지된 색상 차이로, `0`(엄격)과 `1`(관대) 사이의 값을 가집니다. `"pixelmatch"` 비교기는 [YIQ 색 공간](https://en.wikipedia.org/wiki/YIQ)에서 색상 차이를 계산하고 기본적으로 `threshold` 값을 `0.2`로 설정합니다.
    - `maxDiffPixels` [number] *(optional)*
      
      서로 다를 수 있는 허용 가능한 픽셀 양이며, 기본적으로 설정되지 않습니다.
    - `maxDiffPixelRatio` [number] *(optional)*
      
      전체 픽셀 양에 대한 서로 다른 픽셀의 허용 가능한 비율로, `0`과 `1` 사이의 값을 가집니다. 기본적으로 설정되지 않습니다.
    
    [expect(value).toMatchSnapshot()](/api/class-snapshotassertions.mdx#snapshot-assertions-to-match-snapshot-1) 메서드에 대한 구성입니다.
  - `toPass` [Object] *(optional)*
    - `timeout` [number] *(optional)*
      
      toPass 메서드의 제한 시간(밀리초)입니다.
    - `intervals` [Array]&lt;[number]&gt; *(optional)*
      
      toPass 메서드의 프로브 간격(밀리초)입니다.
    
    [expect(value).toPass()](../test-assertions.mdx) 메서드에 대한 구성입니다.

### fullyParallel {#test-project-fully-parallel}

<font size="2" style={{position: "relative", top: "-20px"}}>Added in: v1.10</font><x-search>testProject.fullyParallel</x-search>

Playwright Test는 테스트를 병렬로 실행합니다. 이를 위해 동시에 실행되는 여러 워커 프로세스를 실행합니다. 기본적으로 **테스트 파일**은 병렬로 실행됩니다. 단일 파일의 테스트는 동일한 워커 프로세스에서 순서대로 실행됩니다.

이 옵션을 사용하여 전체 테스트 프로젝트가 모든 파일의 모든 테스트를 동시에 실행하도록 구성할 수 있습니다.

**Usage**

```js
testProject.fullyParallel
```

**Type**
- [boolean]

### grep {#test-project-grep}

<font size="2" style={{position: "relative", top: "-20px"}}>Added in: v1.10</font><x-search>testProject.grep</x-search>

제목이 패턴 중 하나와 일치하는 테스트만 실행하도록 필터링합니다. 예를 들어, `grep: /cart/`를 전달하면 제목에 "cart"가 있는 테스트만 실행해야 합니다. 전역적으로 그리고 `-g` 옵션으로 [명령줄](../test-cli.mdx)에서도 사용할 수 있습니다. 정규식은 프로젝트 이름, 테스트 파일 이름, `test.describe` 이름(있는 경우), 테스트 이름 및 테스트 태그가 공백으로 구분된 문자열에 대해 테스트됩니다. 예: `chromium my-test.spec.ts my-suite my-test`.

`grep` 옵션은 [테스트 태깅](../test-annotations.mdx#tag-tests)에도 유용합니다.

**Usage**

```js
testProject.grep
```

**Type**
- [RegExp] | [Array]&lt;[RegExp]&gt;

### outputDir {#test-project-output-dir}

<font size="2" style={{position: "relative", top: "-20px"}}>Added in: v1.10</font><x-search>testProject.outputDir</x-search>

테스트 실행 중에 생성된 파일에 대한 출력 디렉터리입니다. 기본값은 `<package.json-directory>/test-results`입니다.

이 디렉터리는 시작 시 정리됩니다. 테스트를 실행할 때 [testProject.outputDir](/api/class-testproject.mdx#test-project-output-dir) 내부에 고유한 하위 디렉터리가 생성되어 병렬로 실행되는 테스트가 충돌하지 않도록 보장합니다. 이 디렉터리는 [testInfo.outputDir](/api/class-testinfo.mdx#test-info-output-dir) 및 [testInfo.outputPath()](/api/class-testinfo.mdx#test-info-output-path)를 통해 접근할 수 있습니다.

다음은 [testInfo.outputPath()](/api/class-testinfo.mdx#test-info-output-path)를 사용하여 임시 파일을 만드는 예입니다.

```js
import { test, expect } from '@playwright/test';
import fs from 'fs';

test('example test', async ({}, testInfo) => {
  const file = testInfo.outputPath('temporary-file.txt');
  await fs.promises.writeFile(file, 'Put some data to the file', 'utf8');
});
```

모든 프로젝트에 대해 이 옵션을 변경하려면 [testConfig.outputDir](/api/class-testconfig.mdx#test-config-output-dir)를 사용하세요.

**Usage**

```js
testProject.outputDir
```

**Type**
- [string]

### snapshotPathTemplate {#test-project-snapshot-path-template}

<font size="2" style={{position: "relative", top: "-20px"}}>Added in: v1.28</font><x-search>testProject.snapshotPathTemplate</x-search>

이 옵션은 [expect(page).toHaveScreenshot()](/api/class-pageassertions.mdx#page-assertions-to-have-screenshot-1), [expect(locator).toMatchAriaSnapshot()](/api/class-locatorassertions.mdx#locator-assertions-to-match-aria-snapshot-2) 및 [expect(value).toMatchSnapshot()](/api/class-snapshotassertions.mdx#snapshot-assertions-to-match-snapshot-1)에서 생성된 스냅샷의 위치를 제어하는 템플릿을 구성합니다.

[testConfig.expect](/api/class-testconfig.mdx#test-config-expect)에서 각 어설션에 대한 템플릿을 개별적으로 구성할 수 있습니다.

**Usage**

```js title="playwright.config.ts"
import { defineConfig } from '@playwright/test';

export default defineConfig({
  testDir: './tests',

  // 모든 어설션에 대한 단일 템플릿
  snapshotPathTemplate: '{testDir}/__screenshots__/{testFilePath}/{arg}{ext}',

  // 어설션별 템플릿
  expect: {
    toHaveScreenshot: {
      pathTemplate: '{testDir}/__screenshots__{/projectName}/{testFilePath}/{arg}{ext}',
    },
    toMatchAriaSnapshot: {
      pathTemplate: '{testDir}/__snapshots__/{testFilePath}/{arg}{ext}',
    },
  },
});
```

**Type**
- [string]

**Details**

값에는 테스트 실행 중에 실제 값으로 대체될 일부 "토큰"이 포함될 수 있습니다.

다음과 같은 파일 구조를 고려해 보세요.

```txt
playwright.config.ts
tests/
└── page/
    └── page-click.spec.ts
```

그리고 `toHaveScreenshot()` 호출을 사용하는 다음 `page-click.spec.ts`를 고려해 보세요.

```js title="page-click.spec.ts"
import { test, expect } from '@playwright/test';

test.describe('suite', () => {
  test('test should work', async ({ page }) => {
    await expect(page).toHaveScreenshot(['foo', 'bar', 'baz.png']);
  });
});
```

지원되는 토큰 목록:
* `{arg}` - **확장자 없는** 상대 스냅샷 경로입니다. 이는 `toHaveScreenshot()`, `toMatchAriaSnapshot()` 또는 `toMatchSnapshot()`에 전달된 인수에서 가져옵니다. 인수가 없이 호출되면 자동 생성된 스냅샷 이름이 됩니다.
  * 값: `foo/bar/baz`
* `{ext}` - 스냅샷 확장자(선행 점 포함).
  * 값: `.png`
* `{platform}` - `process.platform`의 값입니다.
* `{projectName}` - 프로젝트의 파일 시스템에서 삭제된 이름(있는 경우)입니다.
  * 값: `''` (빈 문자열).
* `{snapshotDir}` - 프로젝트의 [testProject.snapshotDir](/api/class-testproject.mdx#test-project-snapshot-dir)입니다.
  * 값: `/home/playwright/tests` (`snapshotDir`가 구성에 제공되지 않으므로 기본값은 `testDir`입니다.)
* `{testDir}` - 프로젝트의 [testProject.testDir](/api/class-testproject.mdx#test-project-test-dir)입니다.
  * 값: `/home/playwright/tests` (`testDir`가 구성이 있는 디렉터리를 기준으로 확인되므로 절대 경로입니다.)
* `{testFileDir}` - `testDir`에서 **테스트 파일**까지의 상대 경로에 있는 디렉터리입니다.
  * 값: `page`
* `{testFileName}` - 확장자가 있는 테스트 파일 이름입니다.
  * 값: `page-click.spec.ts`
* `{testFilePath}` - `testDir`에서 **테스트 파일**까지의 상대 경로입니다.
  * 값: `page/page-click.spec.ts`
* `{testName}` - 파일 시스템에서 삭제된 테스트 제목이며, 상위 describe는 포함하지만 파일 이름은 제외합니다.
  * 값: `suite-test-should-work`

각 토큰은 이 토큰에 비어 있지 않은 값이 있는 **경우에만** 사용되는 단일 문자로 시작할 수 있습니다.

다음 구성을 고려해 보세요.

```js title="playwright.config.ts"
import { defineConfig } from '@playwright/test';

export default defineConfig({
  snapshotPathTemplate: '__screenshots__{/projectName}/{testFilePath}/{arg}{ext}',
  testMatch: 'example.spec.ts',
  projects: [
    { use: { browserName: 'firefox' } },
    { name: 'chromium', use: { browserName: 'chromium' } },
  ],
});
```

이 구성에서:
1. 첫 번째 프로젝트에는 이름이 **없으므로** 스냅샷이 `<configDir>/__screenshots__/example.spec.ts/...`에 저장됩니다.
2. 두 번째 프로젝트에는 이름이 **있으므로** 스냅샷이 `<configDir>/__screenshots__/chromium/example.spec.ts/..`에 저장됩니다.
3. `snapshotPathTemplate`가 상대 경로로 확인되므로 `configDir`를 기준으로 확인됩니다.
4. 정방향 슬래시 `"/"`는 모든 플랫폼에서 경로 구분 기호로 사용할 수 있습니다.

### teardown {#test-project-teardown}

<font size="2" style={{position: "relative", top: "-20px"}}>Added in: v1.34</font><x-search>testProject.teardown</x-search>

이 프로젝트와 모든 종속 프로젝트가 완료된 후 실행해야 하는 프로젝트의 이름입니다. Teardown은 이 프로젝트에서 획득한 리소스를 정리하는 데 유용합니다.

`--no-deps` 인수를 전달하면 [testProject.teardown](/api/class-testproject.mdx#test-project-teardown)이 무시되고 지정되지 않은 것처럼 동작합니다.

**Usage**

일반적인 패턴은 해당 "teardown"이 있는 "setup" 종속성입니다.

```js title="playwright.config.ts"
import { defineConfig } from '@playwright/test';

export default defineConfig({
  projects: [
    {
      name: 'setup',
      testMatch: /global.setup\.ts/,
      teardown: 'teardown',
    },
    {
      name: 'teardown',
      testMatch: /global.teardown\.ts/,
    },
    {
      name: 'chromium',
      use: devices['Desktop Chrome'],
      dependencies: ['setup'],
    },
    {
      name: 'firefox',
      use: devices['Desktop Firefox'],
      dependencies: ['setup'],
    },
    {
      name: 'webkit',
      use: devices['Desktop Safari'],
      dependencies: ['setup'],
    },
  ],
});
```

**Type**
- [string]

### testDir {#test-project-test-dir}

<font size="2" style={{position: "relative", top: "-20px"}}>Added in: v1.10</font><x-search>testProject.testDir</x-search>

테스트 파일을 위해 재귀적으로 스캔될 디렉터리입니다. 기본값은 구성 파일의 디렉터리입니다.

각 프로젝트는 다른 디렉터리를 사용할 수 있습니다. 다음은 세 개의 브라우저에서 스모크 테스트를 실행하고 다른 모든 테스트를 안정적인 Chrome 브라우저에서 실행하는 예입니다.

```js title="playwright.config.ts"
import { defineConfig } from '@playwright/test';

export default defineConfig({
  projects: [
    {
      name: 'Smoke Chromium',
      testDir: './smoke-tests',
      use: {
        browserName: 'chromium',
      }
    },
    {
      name: 'Smoke WebKit',
      testDir: './smoke-tests',
      use: {
        browserName: 'webkit',
      }
    },
    {
      name: 'Smoke Firefox',
      testDir: './smoke-tests',
      use: {
        browserName: 'firefox',
      }
    },
    {
      name: 'Chrome Stable',
      testDir: './',
      use: {
        browserName: 'chromium',
        channel: 'chrome',
      }
    },
  ],
});
```

모든 프로젝트에 대해 이 옵션을 변경하려면 [testConfig.testDir](/api/class-testconfig.mdx#test-config-test-dir)를 사용하세요.

**Usage**

testProject.testDir

### testMatch {#test-project-test-match}

<font size="2" style={{position: "relative", top: "-20px"}}>Added in: v1.10</font><x-search>testProject.testMatch</x-search>

이러한 패턴 중 하나와 일치하는 파일만 테스트 파일로 실행됩니다. 일치는 절대 파일 경로에 대해 수행됩니다. 문자열은 glob 패턴으로 처리됩니다.

기본적으로 Playwright는 다음 glob 패턴과 일치하는 파일을 찾습니다. `**/*.@(spec|test).?(c|m)[jt]s?(x)`. 이는 JavaScript 또는 TypeScript 파일이 `".test"` 또는 `".spec"` 접미사를 갖는다는 것을 의미합니다. 예를 들어 `login-screen.wrong-credentials.spec.ts`입니다.

모든 프로젝트에 대해 이 옵션을 변경하려면 [testConfig.testMatch](/api/class-testconfig.mdx#test-config-test-match)를 사용하세요.

**Usage**

```js
testProject.testMatch
```

**Type**
- [string] | [RegExp] | [Array]&lt;[string] | [RegExp]&gt;

testProject.timeout

- [number]
