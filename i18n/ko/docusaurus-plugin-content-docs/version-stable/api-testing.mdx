## 소개

Playwright는 애플리케이션의 [REST](https://en.wikipedia.org/wiki/Representational_state_transfer) API에 접근하는 데 사용할 수 있습니다.

페이지를 로드하고 JS 코드를 실행하지 않고 Node.js에서 직접 서버에 요청을 보내고 싶을 때가 있습니다. 몇 가지 예는 다음과 같습니다.

- 서버 API 테스트.
- 테스트에서 웹 애플리케이션을 방문하기 전에 서버 측 상태 준비.
- 브라우저에서 몇 가지 작업을 실행한 후 서버 측 사후 조건 검증.

이 모든 것은 [APIRequestContext] 메서드를 통해 수행할 수 있습니다.

## API 테스트 작성

[APIRequestContext]는 네트워크를 통해 모든 종류의 HTTP(S) 요청을 보낼 수 있습니다.

다음 예제는 Playwright를 사용하여 [GitHub API](https://docs.github.com/en/rest)를 통해 이슈 생성을 테스트하는 방법을 보여줍니다. 테스트 스위트는 다음을 수행합니다.

- 테스트를 실행하기 전에 새 저장소를 생성합니다.
- 몇 가지 이슈를 생성하고 서버 상태를 검증합니다.
- 테스트를 실행한 후 저장소를 삭제합니다.

### 구성

GitHub API는 인증이 필요하므로 모든 테스트에 대해 한 번 토큰을 구성합니다. 이와 함께 테스트를 단순화하기 위해 `baseURL`도 설정합니다. 구성 파일이나 `test.use()`가 있는 테스트 파일에 넣을 수 있습니다.

```js title="playwright.config.ts"
import { defineConfig } from '@playwright/test';
export default defineConfig({
  use: {
    // 우리가 보내는 모든 요청은 이 API 엔드포인트로 이동합니다.
    baseURL: 'https://api.github.com',
    extraHTTPHeaders: {
      // GitHub 가이드라인에 따라 이 헤더를 설정합니다.
      Accept: 'application/vnd.github.v3+json',
      // 모든 요청에 인증 토큰을 추가합니다.
      // 환경에서 사용 가능한 개인 액세스 토큰을 가정합니다.
      Authorization: `token ${process.env.API_TOKEN}`,
    },
  },
});
```

**프록시 구성**

테스트를 프록시 뒤에서 실행해야 하는 경우 구성에서 이를 지정할 수 있으며 `request` fixture가 자동으로 이를 선택합니다.

```js title="playwright.config.ts"
import { defineConfig } from '@playwright/test';
export default defineConfig({
  use: {
    proxy: {
      server: 'http://my-proxy:8080',
      username: 'user',
      password: 'secret',
    },
  },
});
```

### 테스트 작성

Playwright Test에는 지정한 `baseURL` 또는 `extraHTTPHeaders`와 같은 구성 옵션을 준수하고 요청을 보낼 준비가 된 내장 `request` fixture가 제공됩니다.

이제 저장소에 새 이슈를 생성하는 몇 가지 테스트를 추가할 수 있습니다.

```js
const REPO = 'test-repo-1';
const USER = 'github-username';

test('버그 리포트를 생성해야 합니다.', async ({ request }) => {
  const newIssue = await request.post(`/repos/${USER}/${REPO}/issues`, {
    data: {
      title: '[Bug] report 1',
      body: 'Bug description',
    },
  });
  expect(newIssue.ok()).toBeTruthy();

  const issues = await request.get(`/repos/${USER}/${REPO}/issues`);
  expect(issues.ok()).toBeTruthy();
  expect(await issues.json()).toContainEqual(
    expect.objectContaining({
      title: '[Bug] report 1',
      body: 'Bug description',
    })
  );
});

test('기능 요청을 생성해야 합니다.', async ({ request }) => {
  const newIssue = await request.post(`/repos/${USER}/${REPO}/issues`, {
    data: {
      title: '[Feature] request 1',
      body: 'Feature description',
    },
  });
  expect(newIssue.ok()).toBeTruthy();

  const issues = await request.get(`/repos/${USER}/${REPO}/issues`);
  expect(issues.ok()).toBeTruthy();
  expect(await issues.json()).toContainEqual(
    expect.objectContaining({
      title: '[Feature] request 1',
      body: 'Feature description',
    })
  );
});
```

### 설정 및 해체

이러한 테스트는 저장소가 있다고 가정합니다. 테스트를 실행하기 전에 새 저장소를 생성하고 나중에 삭제하는 것이 좋습니다. 이를 위해 `beforeAll` 및 `afterAll` hook을 사용합니다.

```js
test.beforeAll(async ({ request }) => {
  // 새 저장소를 생성합니다.
  const response = await request.post('/user/repos', {
    data: {
      name: REPO,
    },
  });
  expect(response.ok()).toBeTruthy();
});

test.afterAll(async ({ request }) => {
  // 저장소를 삭제합니다.
  const response = await request.delete(`/repos/${USER}/${REPO}`);
  expect(response.ok()).toBeTruthy();
});
```

## 요청 컨텍스트 사용

내부적으로 [`request` fixture](./api/class-fixtures#fixtures-request)는 실제로 [apiRequest.newContext()](/api/class-apirequest.mdx#api-request-new-context)를 호출합니다. 더 많은 제어를 원하는 경우 언제든지 수동으로 수행할 수 있습니다. 아래는 위의 `beforeAll` 및 `afterAll`과 동일한 작업을 수행하는 독립 실행형 스크립트입니다.

```js
import { request } from '@playwright/test';
const REPO = 'test-repo-1';
const USER = 'github-username';

(async () => {
  // HTTP 요청을 발행할 컨텍스트를 생성합니다.
  const context = await request.newContext({
    baseURL: 'https://api.github.com',
  });

  // 저장소를 생성합니다.
  await context.post('/user/repos', {
    headers: {
      Accept: 'application/vnd.github.v3+json',
      // GitHub 개인 액세스 토큰을 추가합니다.
      Authorization: `token ${process.env.API_TOKEN}`,
    },
    data: {
      name: REPO,
    },
  });

  // 저장소를 삭제합니다.
  await context.delete(`/repos/${USER}/${REPO}`, {
    headers: {
      Accept: 'application/vnd.github.v3+json',
      // GitHub 개인 액세스 토큰을 추가합니다.
      Authorization: `token ${process.env.API_TOKEN}`,
    },
  });
})();
```

## UI 테스트에서 API 요청 보내기

브라우저 내에서 테스트를 실행하는 동안 애플리케이션의 HTTP API를 호출하고 싶을 수 있습니다. 테스트를 실행하기 전에 서버 상태를 준비하거나 브라우저에서 몇 가지 작업을 수행한 후 서버에서 일부 사후 조건을 확인해야 하는 경우 유용할 수 있습니다. 이 모든 것은 [APIRequestContext] 메서드를 통해 수행할 수 있습니다.

### 사전 조건 설정

다음 테스트는 API를 통해 새 이슈를 생성한 다음 프로젝트의 모든 이슈 목록으로 이동하여 목록 상단에 표시되는지 확인합니다.

```js
import { test, expect } from '@playwright/test';

const REPO = 'test-repo-1';
const USER = 'github-username';

// 요청 컨텍스트는 파일의 모든 테스트에서 재사용됩니다.
let apiContext;

test.beforeAll(async ({ playwright }) => {
  apiContext = await playwright.request.newContext({
    // 우리가 보내는 모든 요청은 이 API 엔드포인트로 이동합니다.
    baseURL: 'https://api.github.com',
    extraHTTPHeaders: {
      // GitHub 가이드라인에 따라 이 헤더를 설정합니다.
      Accept: 'application/vnd.github.v3+json',
      // 모든 요청에 인증 토큰을 추가합니다.
      // 환경에서 사용 가능한 개인 액세스 토큰을 가정합니다.
      Authorization: `token ${process.env.API_TOKEN}`,
    },
  });
});

test.afterAll(async ({}) => {
  // 모든 응답을 삭제합니다.
  await apiContext.dispose();
});

test('마지막으로 생성된 이슈가 목록의 첫 번째 항목이어야 합니다.', async ({
  page,
}) => {
  const newIssue = await apiContext.post(`/repos/${USER}/${REPO}/issues`, {
    data: {
      title: '[Feature] request 1',
    },
  });
  expect(newIssue.ok()).toBeTruthy();

  await page.goto(`https://github.com/${USER}/${REPO}/issues`);
  const firstIssue = page.locator(`a[data-hovercard-type='issue']`).first();
  await expect(firstIssue).toHaveText('[Feature] request 1');
});
```

### 사후 조건 검증

다음 테스트는 브라우저의 사용자 인터페이스를 통해 새 이슈를 생성한 다음 API를 통해 생성되었는지 확인합니다.

```js
import { test, expect } from '@playwright/test';

const REPO = 'test-repo-1';
const USER = 'github-username';

// 요청 컨텍스트는 파일의 모든 테스트에서 재사용됩니다.
let apiContext;

test.beforeAll(async ({ playwright }) => {
  apiContext = await playwright.request.newContext({
    // 우리가 보내는 모든 요청은 이 API 엔드포인트로 이동합니다.
    baseURL: 'https://api.github.com',
    extraHTTPHeaders: {
      // GitHub 가이드라인에 따라 이 헤더를 설정합니다.
      Accept: 'application/vnd.github.v3+json',
      // 모든 요청에 인증 토큰을 추가합니다.
      // 환경에서 사용 가능한 개인 액세스 토큰을 가정합니다.
      Authorization: `token ${process.env.API_TOKEN}`,
    },
  });
});

test.afterAll(async ({}) => {
  // 모든 응답을 삭제합니다.
  await apiContext.dispose();
});

test('마지막으로 생성된 이슈가 서버에 있어야 합니다.', async ({ page }) => {
  await page.goto(`https://github.com/${USER}/${REPO}/issues`);
  await page.getByText('New Issue').click();
  await page.getByRole('textbox', { name: 'Title' }).fill('Bug report 1');
  await page
    .getByRole('textbox', { name: 'Comment body' })
    .fill('Bug description');
  await page.getByText('Submit new issue').click();
  const issueId = page.url().substr(page.url().lastIndexOf('/'));

  const newIssue = await apiContext.get(
    `https://api.github.com/repos/${USER}/${REPO}/issues/${issueId}`
  );
  expect(newIssue.ok()).toBeTruthy();
  expect(newIssue.json()).toEqual(
    expect.objectContaining({
      title: 'Bug report 1',
    })
  );
});
```

## 인증 상태 재사용

웹 앱은 쿠키 기반 또는 토큰 기반 인증을 사용하며, 여기서 인증된 상태는 [쿠키](https://developer.mozilla.org/en-US/docs/Web/HTTP/Cookies)로 저장됩니다. Playwright는 인증된 컨텍스트에서 저장소 상태를 검색한 다음 해당 상태로 새 컨텍스트를 생성하는 데 사용할 수 있는 [apiRequestContext.storageState()](/api/class-apirequestcontext.mdx#api-request-context-storage-state) 메서드를 제공합니다.

저장소 상태는 [BrowserContext]와 [APIRequestContext] 간에 교환할 수 있습니다. API 호출을 통해 로그인한 다음 쿠키가 이미 있는 새 컨텍스트를 생성하는 데 사용할 수 있습니다. 다음 코드 스니펫은 인증된 [APIRequestContext]에서 상태를 검색하고 해당 상태로 새 [BrowserContext]를 생성합니다.

```js
const requestContext = await request.newContext({
  httpCredentials: {
    username: 'user',
    password: 'passwd',
  },
});
await requestContext.get(`https://api.example.com/login`);
// 저장소 상태를 파일에 저장합니다.
await requestContext.storageState({ path: 'state.json' });

// 저장된 저장소 상태로 새 컨텍스트를 생성합니다.
const context = await browser.newContext({ storageState: 'state.json' });
```

## 컨텍스트 요청 vs 전역 요청

두 가지 유형의 [APIRequestContext]가 있습니다.

- [BrowserContext]와 연결됨
- [apiRequest.newContext()](/api/class-apirequest.mdx#api-request-new-context)를 통해 생성된 격리된 인스턴스

주요 차이점은 [browserContext.request](/api/class-browsercontext.mdx#browser-context-request) 및 [page.request](/api/class-page.mdx#page-request)를 통해 액세스할 수 있는 [APIRequestContext]는 브라우저 컨텍스트에서 요청의 `Cookie` 헤더를 채우고 [APIResponse]에 `Set-Cookie` 헤더가 있는 경우 브라우저 쿠키를 자동으로 업데이트한다는 것입니다.

```js
test('컨텍스트 요청은 쿠키 저장소를 브라우저 컨텍스트와 공유합니다.', async ({
  page,
  context,
}) => {
  await context.route('https://www.github.com/', async (route) => {
    // 브라우저 컨텍스트와 쿠키 저장소를 공유하는 API 요청을 보냅니다.
    const response = await context.request.fetch(route.request());
    const responseHeaders = response.headers();

    // 응답에는 'Set-Cookie' 헤더가 있습니다.
    const responseCookies = new Map(
      responseHeaders['set-cookie']
        .split('\n')
        .map((c) => c.split(';', 2)[0].split('='))
    );
    // 응답에는 'Set-Cookie' 헤더에 3개의 쿠키가 있습니다.
    expect(responseCookies.size).toBe(3);
    const contextCookies = await context.cookies();
    // 브라우저 컨텍스트에는 이미 API 응답의 모든 쿠키가 포함되어 있습니다.
    expect(
      new Map(contextCookies.map(({ name, value }) => [name, value]))
    ).toEqual(responseCookies);

    await route.fulfill({
      response,
      headers: { ...responseHeaders, foo: 'bar' },
    });
  });
  await page.goto('https://www.github.com/');
});
```

[APIRequestContext]가 브라우저 컨텍스트의 쿠키를 사용하고 업데이트하지 않도록 하려면 격리된 쿠키가 있는 [APIRequestContext]의 새 인스턴스를 수동으로 생성할 수 있습니다.

```js
test('전역 컨텍스트 요청은 격리된 쿠키 저장소를 갖습니다.', async ({
  page,
  context,
  browser,
  playwright,
}) => {
  // 격리된 쿠키 저장소를 사용하여 APIRequestContext의 새 인스턴스를 생성합니다.
  const request = await playwright.request.newContext();
  await context.route('https://www.github.com/', async (route) => {
    const response = await request.fetch(route.request());
    const responseHeaders = response.headers();

    const responseCookies = new Map(
      responseHeaders['set-cookie']
        .split('\n')
        .map((c) => c.split(';', 2)[0].split('='))
    );
    // 응답에는 'Set-Cookie' 헤더에 3개의 쿠키가 있습니다.
    expect(responseCookies.size).toBe(3);
    const contextCookies = await context.cookies();
    // 브라우저 컨텍스트에는 격리된 API 요청의 쿠키가 없습니다.
    expect(contextCookies.length).toBe(0);

    // 쿠키 저장소를 수동으로 내보냅니다.
    const storageState = await request.storageState();
    // 새 컨텍스트를 생성하고 전역 요청의 쿠키로 초기화합니다.
    const browserContext2 = await browser.newContext({ storageState });
    const contextCookies2 = await browserContext2.cookies();
    // 새 브라우저 컨텍스트에는 이미 API 응답의 모든 쿠키가 포함되어 있습니다.
    expect(
      new Map(contextCookies2.map(({ name, value }) => [name, value]))
    ).toEqual(responseCookies);

    await route.fulfill({
      response,
      headers: { ...responseHeaders, foo: 'bar' },
    });
  });
  await page.goto('https://www.github.com/');
  await request.dispose();
});
```
