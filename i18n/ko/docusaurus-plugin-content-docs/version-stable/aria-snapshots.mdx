## 개요

Playwright의 스냅샷 테스트를 사용하면 페이지의 접근성 트리를 미리 정의된 스냅샷 템플릿과 비교하여 어설션할 수 있습니다.

```js
await page.goto('https://playwright.dev/');
await expect(page.getByRole('banner')).toMatchAriaSnapshot(`
  - banner:
    - heading /Playwright enables reliable end-to-end/ [level=1]
    - link "Get started"
    - link "Star microsoft/playwright on GitHub"
    - link /[\\d]+k\\+ stargazers on GitHub/
`);
```

## 어설션 테스트 vs 스냅샷 테스트

스냅샷 테스트와 어설션 테스트는 테스트 자동화에서 서로 다른 목적을 제공합니다.

### 어설션 테스트

어설션 테스트는 요소 또는 구성 요소에 대한 특정 값이나 조건을 어설션하는 대상 지정 방식입니다. 예를 들어 Playwright에서 [expect(locator).toHaveText()](/api/class-locatorassertions.mdx#locator-assertions-to-have-text)는 요소에 예상 텍스트가 포함되어 있는지 확인하고 [expect(locator).toHaveValue()](/api/class-locatorassertions.mdx#locator-assertions-to-have-value)는 입력 필드에 예상 값이 있는지 확인합니다. 어설션 테스트는 구체적이며 일반적으로 요소 또는 속성의 현재 상태를 예상되고 미리 정의된 상태와 비교하여 확인합니다. 예측 가능한 단일 값 검사에는 적합하지만 광범위한 구조 또는 변형을 테스트할 때는 범위가 제한됩니다.

**장점**

- **명확성**: 테스트의 의도가 명시적이고 이해하기 쉽습니다.
- **구체성**: 테스트는 기능의 특정 측면에 중점을 두어 관련 없는 변경 사항에 대해 더욱 강력합니다.
- **디버깅**: 실패는 문제가 있는 측면을 직접적으로 가리키는 대상 지정 피드백을 제공합니다.

**단점**

- **복잡한 출력에 대한 장황함**: 복잡한 데이터 구조 또는 대규모 출력에 대한 어설션을 작성하는 것은 번거롭고 오류가 발생하기 쉽습니다.
- **유지 관리 오버헤드**: 코드가 발전함에 따라 어설션을 수동으로 업데이트하는 것은 시간이 많이 걸릴 수 있습니다.

### 스냅샷 테스트

스냅샷 테스트는 특정 시점에 요소, 구성 요소 또는 데이터의 전체 상태에 대한 "스냅샷" 또는 표현을 캡처하여 나중에 비교하기 위해 저장합니다. 테스트를 다시 실행할 때 현재 상태가 스냅샷과 비교되고 차이가 있으면 테스트가 실패합니다. 이 접근 방식은 각 세부 사항을 수동으로 어설션하는 것이 너무 시간이 많이 걸리는 복잡하거나 동적인 구조에 특히 유용합니다. 스냅샷 테스트는 어설션 테스트보다 광범위하고 전체적이므로 시간이 지남에 따라 더 복잡한 변경 사항을 추적할 수 있습니다.

**장점**

- **복잡한 출력 단순화**: 예를 들어 기존 어설션을 사용하여 UI 구성 요소의 렌더링된 출력을 테스트하는 것은 지루할 수 있습니다. 스냅샷은 전체 출력을 캡처하여 쉽게 비교할 수 있습니다.
- **빠른 피드백 루프**: 개발자는 출력에서 의도하지 않은 변경 사항을 쉽게 발견할 수 있습니다.
- **일관성 장려**: 코드가 발전함에 따라 일관된 출력을 유지하는 데 도움이 됩니다.

**단점**

- **과도한 의존**: 스냅샷의 변경 사항을 완전히 이해하지 않고 수락하여 버그를 숨길 수 있습니다.
- **세분성**: 특히 사소한 변경 사항이 출력의 많은 부분에 영향을 미치는 경우 차이가 발생할 때 큰 스냅샷을 해석하기 어려울 수 있습니다.
- **적합성**: 출력이 자주 또는 예측할 수 없게 변경되는 매우 동적인 콘텐츠에는 적합하지 않습니다.

### 사용 시기

- **스냅샷 테스트**는 다음과 같은 경우에 이상적입니다.
  - 전체 페이지 및 구성 요소의 UI 테스트.
  - 복잡한 UI 구성 요소에 대한 광범위한 구조 검사.
  - 구조가 거의 변경되지 않는 출력에 대한 회귀 테스트.
- **어설션 테스트**는 다음과 같은 경우에 이상적입니다.
  - 핵심 로직 검증.
  - 계산된 값 테스트.
  - 정밀한 조건이 필요한 세분화된 테스트.

광범위한 구조적 검사를 위한 스냅샷 테스트와 특정 기능에 대한 어설션 테스트를 결합하여 잘 균형 잡힌 테스트 전략을 달성할 수 있습니다.

## Aria 스냅샷

Playwright에서 aria 스냅샷은 페이지의 접근성 트리에 대한 YAML 표현을 제공합니다. 이러한 스냅샷은 나중에 저장하고 비교하여 페이지 구조가 일관성을 유지하는지 또는 정의된 기대치를 충족하는지 확인할 수 있습니다.

YAML 형식은 페이지에서 접근 가능한 요소의 계층적 구조를 설명하고 **역할**, **속성**, **값** 및 **텍스트 콘텐츠**를 자세히 설명합니다. 구조는 트리와 같은 구문을 따르며 각 노드는 접근 가능한 요소를 나타내고 들여쓰기는 중첩된 요소를 나타냅니다.

트리의 각 접근 가능한 요소는 YAML 노드로 표시됩니다.

```yaml
- role "name" [attribute=value]
```

- **role**: 요소의 ARIA 또는 HTML 역할을 지정합니다(예: `heading`, `list`, `listitem`, `button`).
- **"name"**: 요소의 접근 가능한 이름입니다. 따옴표로 묶인 문자열은 정확한 값을 나타내고 `/patterns/`는 정규 표현식에 사용됩니다.
- **[attribute=value]**: 대괄호 안의 속성과 값은 `checked`, `disabled`, `expanded`, `level`, `pressed` 또는 `selected`와 같은 특정 ARIA 속성을 나타냅니다.

이러한 값은 ARIA 속성에서 파생되거나 HTML 의미 체계를 기반으로 계산됩니다. 페이지의 접근성 트리 구조를 검사하려면 [Chrome DevTools 접근성 창](https://developer.chrome.com/docs/devtools/accessibility/reference#pane)을 사용하십시오.

## 스냅샷 매칭

Playwright의 [expect(locator).toMatchAriaSnapshot()](/api/class-locatorassertions.mdx#locator-assertions-to-match-aria-snapshot) 어설션 메서드는 로케이터 범위의 접근 가능한 구조를 미리 정의된 aria 스냅샷 템플릿과 비교하여 페이지 상태가 테스트 요구 사항에 대해 유효성을 검사하는 데 도움이 됩니다.

다음 DOM의 경우:

```html
<h1>title</h1>
```

다음 스냅샷 템플릿을 사용하여 일치시킬 수 있습니다.

```js
await expect(page.locator('body')).toMatchAriaSnapshot(`
  - heading "title"
`);
```

매칭할 때 스냅샷 템플릿은 페이지의 현재 접근성 트리와 비교됩니다.

- 트리 구조가 템플릿과 일치하면 테스트가 통과됩니다. 그렇지 않으면 테스트가 실패하고 예상 접근성 상태와 실제 접근성 상태가 일치하지 않음을 나타냅니다.
- 비교는 대소문자를 구분하고 공백을 축소하므로 들여쓰기와 줄 바꿈은 무시됩니다.
- 비교는 순서에 민감합니다. 즉, 스냅샷 템플릿의 요소 순서는 페이지의 접근성 트리의 순서와 일치해야 합니다.

### 부분 매칭

속성이나 접근 가능한 이름을 생략하여 노드에서 부분 일치를 수행하여 정확한 일치 없이 접근성 트리의 특정 부분을 확인할 수 있습니다. 이러한 유연성은 동적 또는 관련 없는 속성에 유용합니다.

```html
<button>Submit</button>
```

_aria 스냅샷_

```yaml
- button
```

이 예에서 버튼 역할은 일치하지만 접근 가능한 이름("Submit")은 지정되지 않으므로 버튼의 레이블에 관계없이 테스트가 통과됩니다.

<hr />

`checked` 또는 `disabled`와 같은 ARIA 속성이 있는 요소의 경우 이러한 속성을 생략하면 부분 일치가 가능하여 역할과 계층 구조에만 집중할 수 있습니다.

```html
<input type="checkbox" checked />
```

_부분 일치를 위한 aria 스냅샷_

```yaml
- checkbox
```

이 부분 일치에서 `checked` 속성은 무시되므로 체크박스 상태에 관계없이 테스트가 통과됩니다.

<hr />

마찬가지로 특정 목록 항목이나 중첩된 요소를 생략하여 목록 또는 그룹의 자식을 부분적으로 일치시킬 수 있습니다.

```html
<ul>
  <li>Feature A</li>
  <li>Feature B</li>
  <li>Feature C</li>
</ul>
```

_부분 일치를 위한 aria 스냅샷_

```yaml
- list
  - listitem: Feature B
```

부분 일치를 사용하면 특정 콘텐츠 또는 속성을 적용하지 않고 필수 페이지 구조를 확인하는 유연한 스냅샷 테스트를 만들 수 있습니다.

### 정규 표현식과 매칭

정규 표현식을 사용하면 동적 또는 가변 텍스트가 있는 요소에 대해 유연하게 매칭할 수 있습니다. 접근 가능한 이름과 텍스트는 정규식 패턴을 지원할 수 있습니다.

```html
<h1>Issues 12</h1>
```

_정규 표현식이 있는 aria 스냅샷_

```yaml
- heading /Issues \d+/
```

## 스냅샷 생성

Playwright에서 aria 스냅샷을 생성하면 애플리케이션의 구조를 보장하고 유지하는 데 도움이 됩니다. 테스트 설정 및 워크플로에 따라 다양한 방식으로 스냅샷을 생성할 수 있습니다.

### Playwright 코드 생성기를 사용하여 스냅샷 생성

Playwright의 [코드 생성기](./codegen.mdx)를 사용하는 경우 대화형 인터페이스를 통해 aria 스냅샷 생성이 간소화됩니다.

- **"스냅샷 어설션" 작업**: 코드 생성기에서 "스냅샷 어설션" 작업을 사용하여 선택한 요소에 대한 스냅샷 어설션을 자동으로 생성할 수 있습니다. 이는 기록된 테스트 흐름의 일부로 aria 스냅샷을 캡처하는 빠른 방법입니다.
- **"Aria 스냅샷" 탭**: 코드 생성기 인터페이스 내의 "Aria 스냅샷" 탭은 선택한 로케이터에 대한 aria 스냅샷을 시각적으로 나타내므로 요소 역할, 속성 및 접근 가능한 이름을 탐색, 검사 및 확인하여 스냅샷 생성 및 검토를 지원할 수 있습니다.

### `@playwright/test` 및 `--update-snapshots` 플래그를 사용하여 스냅샷 업데이트

Playwright 테스트 러너(`@playwright/test`)를 사용할 때 `--update-snapshots` 플래그(줄여서 `-u`)를 사용하여 스냅샷을 자동으로 업데이트할 수 있습니다.

`--update-snapshots` 플래그를 사용하여 테스트를 실행하면 일치하지 않는 스냅샷이 업데이트됩니다. 일치하는 스냅샷은 업데이트되지 않습니다.

```bash
npx playwright test --update-snapshots
```

스냅샷 업데이트는 애플리케이션 구조 변경으로 인해 새로운 스냅샷이 기준선으로 필요한 경우 유용합니다. Playwright는 테스트 러너 구성에 지정된 최대 예상 시간 초과를 기다려 스냅샷을 찍기 전에 페이지가 안정화되었는지 확인합니다. 스냅샷을 생성하는 동안 테스트가 시간 초과에 도달하면 `--timeout`을 조정해야 할 수 있습니다.

#### 스냅샷 생성을 위한 빈 템플릿

빈 문자열을 어설션의 템플릿으로 전달하면 즉석에서 스냅샷이 생성됩니다.

```js
await expect(locator).toMatchAriaSnapshot('');
```

Playwright는 테스트 러너 구성에 지정된 최대 예상 시간 초과를 기다려 스냅샷을 찍기 전에 페이지가 안정화되었는지 확인합니다. 스냅샷을 생성하는 동안 테스트가 시간 초과에 도달하면 `--timeout`을 조정해야 할 수 있습니다.

#### 스냅샷 패치 파일

스냅샷을 업데이트할 때 Playwright는 차이점을 캡처하는 패치 파일을 만듭니다. 이러한 패치 파일을 검토, 적용 및 소스 제어에 커밋하여 팀이 시간 경과에 따른 구조적 변경 사항을 추적하고 업데이트가 애플리케이션 요구 사항과 일치하는지 확인할 수 있습니다.

소스 코드가 업데이트되는 방식은 `--update-source-method` 플래그를 사용하여 변경할 수 있습니다. 사용 가능한 옵션은 다음과 같습니다.

- **"patch"**(기본값): `git apply`를 사용하여 소스 코드에 적용할 수 있는 통합 diff 파일을 생성합니다.
- **"3way"**: 소스 코드에 병합 충돌 마커를 생성하여 변경 사항을 수락할지 여부를 선택할 수 있습니다.
- **"overwrite"**: 소스 코드를 새 스냅샷 값으로 덮어씁니다.

```bash
npx playwright test --update-snapshots --update-source-mode=3way
```

#### 별도의 파일로 스냅샷

스냅샷을 별도의 파일에 저장하려면 `name` 옵션과 함께 `.aria.yml` 파일 확장자를 지정하여 `toMatchAriaSnapshot` 메서드를 사용하십시오.

```js
await expect(page.getByRole('main')).toMatchAriaSnapshot({
  name: 'main.aria.yml',
});
```

기본적으로 테스트 파일 `example.spec.ts`의 스냅샷은 `example.spec.ts-snapshots` 디렉터리에 배치됩니다. 스냅샷은 여러 브라우저로 테스트하더라도 브라우저 간에 동일해야 하므로 하나의 스냅샷만 저장됩니다. 원하는 경우 다음 구성을 사용하여 [스냅샷 경로 템플릿](./api/class-testconfig#test-config-snapshot-path-template)을 사용자 지정할 수 있습니다.

```js
export default defineConfig({
  expect: {
    toMatchAriaSnapshot: {
      pathTemplate: '__snapshots__/{testFilePath}/{arg}{ext}',
    },
  },
});
```

### `Locator.ariaSnapshot` 메서드 사용

[locator.ariaSnapshot()](/api/class-locator.mdx#locator-aria-snapshot) 메서드를 사용하면 로케이터 범위 내에서 접근 가능한 요소의 YAML 표현을 프로그래밍 방식으로 만들 수 있습니다. 이는 특히 테스트 실행 중에 동적으로 스냅샷을 생성하는 데 유용합니다.

**예**:

```js
const snapshot = await page.locator('body').ariaSnapshot();
console.log(snapshot);
```

이 명령은 지정된 로케이터 범위 내의 aria 스냅샷을 YAML 형식으로 출력합니다. 필요에 따라 유효성을 검사하거나 저장할 수 있습니다.

## 접근성 트리 예제

### 레벨 속성이 있는 제목

제목에는 제목 레벨을 나타내는 `level` 속성이 포함될 수 있습니다.

```html
<h1>Title</h1>
<h2>Subtitle</h2>
```

_aria 스냅샷_

```yaml
- heading "Title" [level=1]
- heading "Subtitle" [level=2]
```

### 텍스트 노드

독립 실행형 또는 설명 텍스트 요소는 텍스트 노드로 나타납니다.

```html
<div>Sample accessible name</div>
```

_aria 스냅샷_

```yaml
- text: Sample accessible name
```

### 인라인 여러 줄 텍스트

단락과 같은 여러 줄 텍스트는 aria 스냅샷에서 정규화됩니다.

```html
<p>Line 1<br />Line 2</p>
```

_aria 스냅샷_

```yaml
- paragraph: Line 1 Line 2
```

### 링크

링크는 의사 요소의 텍스트 또는 구성된 콘텐츠를 표시합니다.

```html
<a href="#more-info">Read more about Accessibility</a>
```

_aria 스냅샷_

```yaml
- link "Read more about Accessibility"
```

### 텍스트 상자

`text` 유형의 입력 요소는 `value` 속성 콘텐츠를 표시합니다.

```html
<input type="text" value="Enter your name" />
```

_aria 스냅샷_

```yaml
- textbox: Enter your name
```

### 항목이 있는 목록

정렬 및 비정렬 목록에는 목록 항목이 포함됩니다.

```html
<ul aria-label="Main Features">
  <li>Feature 1</li>
  <li>Feature 2</li>
</ul>
```

_aria 스냅샷_

```yaml
- list "Main Features":
    - listitem: Feature 1
    - listitem: Feature 2
```

### 그룹화된 요소

그룹은 요약 콘텐츠가 있는 `<details>` 요소와 같이 중첩된 요소를 캡처합니다.

```html
<details>
  <summary>Summary</summary>
  <p>Detail content here</p>
</details>
```

_aria 스냅샷_

```yaml
- group: Summary
```

### 속성 및 상태

`checked`, `disabled`, `expanded`, `level`, `pressed` 및 `selected`와 같은 일반적으로 사용되는 ARIA 속성은 컨트롤 상태를 나타냅니다.

#### `checked` 속성이 있는 체크박스

```html
<input type="checkbox" checked />
```

_aria 스냅샷_

```yaml
- checkbox [checked]
```

#### `pressed` 속성이 있는 버튼

```html
<button aria-pressed="true">Toggle</button>
```

_aria 스냅샷_

```yaml
- button "Toggle" [pressed=true]
```
