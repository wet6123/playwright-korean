## Introduction

기본적으로 Playwright는 [병렬](./test-parallel.mdx)로 테스트 파일을 실행하고 시스템의 CPU 코어를 최적으로 활용하려고 노력합니다. 훨씬 더 큰 병렬화를 달성하기 위해 여러 시스템에서 동시에 테스트를 실행하여 Playwright 테스트 실행을 더욱 확장할 수 있습니다. 이러한 작업 모드를 "샤딩"이라고 부릅니다. Playwright에서 샤딩은 테스트를 "샤드"라는 더 작은 부분으로 분할하는 것을 의미합니다. 각 샤드는 독립적으로 실행할 수 있는 별도의 작업과 같습니다. 전체 목적은 테스트 런타임을 가속화하기 위해 테스트를 분할하는 것입니다.

테스트를 샤딩하면 각 샤드는 사용 가능한 CPU 코어를 활용하여 자체적으로 실행할 수 있습니다. 이는 작업을 동시에 수행하여 테스트 프로세스 속도를 높이는 데 도움이 됩니다.

CI 파이프라인에서 각 샤드는 별도의 작업으로 실행되어 CPU 코어와 같은 CI 파이프라인에서 사용 가능한 하드웨어 리소스를 활용하여 테스트를 더 빠르게 실행할 수 있습니다.

## Sharding tests between multiple machines

테스트 스위트를 샤딩하려면 명령줄에 `--shard=x/y`를 전달합니다. 예를 들어 스위트를 4개의 샤드로 분할하여 각 샤드가 테스트의 1/4을 실행하려면 다음과 같이 합니다.

```bash
npx playwright test --shard=1/4
npx playwright test --shard=2/4
npx playwright test --shard=3/4
npx playwright test --shard=4/4
```

이제 이러한 샤드를 서로 다른 작업에서 병렬로 실행하면 테스트 스위트가 4배 더 빨리 완료됩니다.

Playwright는 병렬로 실행할 수 있는 테스트만 샤딩할 수 있습니다. 기본적으로 이는 Playwright가 테스트 파일을 샤딩함을 의미합니다. [병렬 처리 가이드](./test-parallel.mdx)에서 다른 옵션에 대해 알아보세요.

## Balancing Shards

샤딩은 [testProject.fullyParallel](/api/class-testproject.mdx#test-project-fully-parallel) 옵션 사용 여부에 따라 두 가지 수준의 세분성으로 수행할 수 있습니다. 이는 테스트가 샤드 간에 균형을 이루는 방식에 영향을 줍니다.

**Sharding with fullyParallel**

`fullyParallel: true`가 활성화되면 Playwright Test는 개별 테스트를 여러 샤드에서 병렬로 실행하여 각 샤드가 테스트를 균등하게 분배받도록 합니다. 이를 통해 테스트 수준의 세분성을 확보할 수 있으며, 이는 각 샤드가 실행하는 개별 테스트 수를 균형 있게 조정하려고 시도함을 의미합니다. 이는 샤딩 시 부하를 균등하게 분배하는 데 선호되는 모드이며, Playwright는 총 테스트 수를 기준으로 샤드 실행을 최적화할 수 있습니다.

**Sharding without fullyParallel**

fullyParallel 설정이 없으면 Playwright Test는 기본적으로 파일 수준의 세분성을 사용합니다. 즉, 전체 테스트 파일이 샤드에 할당됩니다(동일한 파일이 여러 프로젝트에서 서로 다른 샤드에 할당될 수 있음). 이 경우 파일당 테스트 수는 샤드 분배에 큰 영향을 미칠 수 있습니다. 테스트 파일 크기가 균등하지 않은 경우(즉, 일부 파일에 다른 파일보다 훨씬 많은 테스트가 포함된 경우) 특정 샤드는 훨씬 더 많은 테스트를 실행하는 반면 다른 샤드는 더 적거나 심지어 전혀 실행하지 않을 수 있습니다.

**Key Takeaways:**
- `fullyParallel: true`인 **경우**: 테스트는 개별 테스트 수준에서 분할되어 샤드 실행이 더 균형 있게 이루어집니다.
- `fullyParallel`이 **없는 경우**: 테스트는 파일 수준에서 분할되므로 샤드의 균형을 맞추려면 테스트 파일을 작고 균등한 크기로 유지하는 것이 중요합니다.
- 샤딩을 가장 효과적으로 사용하려면 특히 CI 환경에서 샤드 간에 균형 잡힌 분배를 목표로 할 때 `fullyParallel: true`를 사용하는 것이 좋습니다. 그렇지 않으면 불균형을 방지하기 위해 테스트 파일을 수동으로 구성해야 할 수 있습니다.

## Merging reports from multiple shards

이전 예제에서는 각 테스트 샤드에 자체 테스트 보고서가 있습니다. 모든 샤드의 모든 테스트 결과를 보여주는 결합된 보고서를 만들려면 해당 보고서를 병합할 수 있습니다.

CI에서 실행할 때 `blob` 리포터를 구성에 추가하는 것으로 시작합니다.

```ts title="playwright.config.ts"
export default defineConfig({
  testDir: './tests',
  reporter: process.env.CI ? 'blob' : 'html',
});
```

Blob 보고서에는 실행된 모든 테스트와 해당 결과는 물론 추적 및 스크린샷 차이와 같은 모든 테스트 첨부 파일에 대한 정보가 포함되어 있습니다. Blob 보고서는 병합되어 다른 Playwright 보고서로 변환될 수 있습니다. 기본적으로 blob 보고서는 `blob-report` 디렉터리에 생성됩니다.

여러 샤드의 보고서를 병합하려면 blob 보고서 파일을 단일 디렉터리(예: `all-blob-reports`)에 넣습니다. Blob 보고서 이름에는 샤드 번호가 포함되어 있으므로 충돌하지 않습니다.

그런 다음 `npx playwright merge-reports` 명령을 실행합니다.

```bash
npx playwright merge-reports --reporter html ./all-blob-reports
```

그러면 표준 HTML 보고서가 `playwright-report` 디렉터리에 생성됩니다.

## GitHub Actions example

GitHub Actions는 [`jobs.<job_id>.strategy.matrix`](https://docs.github.com/en/actions/using-workflows/workflow-syntax-for-github-actions#jobsjob_idstrategymatrix) 옵션을 사용하여 [여러 작업 간에 테스트를 샤딩](https://docs.github.com/en/actions/using-jobs/using-a-matrix-for-your-jobs)하는 것을 지원합니다. `matrix` 옵션은 제공된 옵션의 가능한 모든 조합에 대해 별도의 작업을 실행합니다.

다음 예제에서는 4대의 시스템에서 병렬로 테스트를 실행한 다음 보고서를 단일 보고서로 병합하도록 작업을 구성하는 방법을 보여줍니다. 위의 예제와 같이 `playwright.config.ts` 파일에 `reporter: process.env.CI ? 'blob' : 'html',`을 추가하는 것을 잊지 마십시오.
1. 먼저 만들려는 총 샤드 수를 포함하는 `shardTotal: [4]` 옵션과 샤드 번호 배열이 있는 `shardIndex: [1, 2, 3, 4]` 옵션을 사용하여 작업 구성에 `matrix` 옵션을 추가합니다.
2. 그런 다음 `--shard=${{ matrix.shardIndex }}/${{ matrix.shardTotal }}` 옵션을 사용하여 Playwright 테스트를 실행합니다. 이렇게 하면 각 샤드에 대해 테스트 명령이 실행됩니다.
3. 마지막으로 blob 보고서를 GitHub Actions Artifacts에 업로드합니다. 이렇게 하면 워크플로의 다른 작업에서 blob 보고서를 사용할 수 있습니다.

```yaml title=".github/workflows/playwright.yml"
name: Playwright Tests
on:
  push:
    branches: [ main, master ]
  pull_request:
    branches: [ main, master ]
jobs:
  playwright-tests:
    timeout-minutes: 60
    runs-on: ubuntu-latest
    strategy:
      fail-fast: false
      matrix:
        shardIndex: [1, 2, 3, 4]
        shardTotal: [4]
    steps:
    - uses: actions/checkout@v4
    - uses: actions/setup-node@v4
      with:
        node-version: lts/*
    - name: Install dependencies
      run: npm ci
    - name: Install Playwright browsers
      run: npx playwright install --with-deps

    - name: Run Playwright tests
      run: npx playwright test --shard=${{ matrix.shardIndex }}/${{ matrix.shardTotal }}

    - name: Upload blob report to GitHub Actions Artifacts
      if: ${{ !cancelled() }}
      uses: actions/upload-artifact@v4
      with:
        name: blob-report-${{ matrix.shardIndex }}
        path: blob-report
        retention-days: 1
```

1. 모든 샤드가 완료되면 보고서를 병합하고 결합된 [HTML 보고서](./test-reporters.mdx#html-reporter)를 생성하는 별도의 작업을 실행할 수 있습니다. 실행 순서를 보장하기 위해 `merge-reports` 작업이 `needs: [playwright-tests]`를 추가하여 샤딩된 `playwright-tests` 작업에 [종속](https://docs.github.com/en/actions/using-jobs/using-jobs-in-a-workflow#defining-prerequisite-jobs)되도록 합니다.

```yaml title=".github/workflows/playwright.yml"
jobs:
...
  merge-reports:
    # 일부 샤드가 실패한 경우에도 playwright-tests 이후에 보고서를 병합합니다.
    if: ${{ !cancelled() }}
    needs: [playwright-tests]

    runs-on: ubuntu-latest
    steps:
    - uses: actions/checkout@v4
    - uses: actions/setup-node@v4
      with:
        node-version: lts/*
    - name: Install dependencies
      run: npm ci

    # GitHub Actions 아티팩트에서 blob 보고서를 다운로드합니다.
    - name: Download blob reports from GitHub Actions Artifacts
      uses: actions/download-artifact@v4
      with:
        path: all-blob-reports
        pattern: blob-report-*
        merge-multiple: true

    # HTML 보고서로 병합
    - name: Merge into HTML Report
      run: npx playwright merge-reports --reporter html ./all-blob-reports

    # HTML 보고서 업로드
    - name: Upload HTML report
      uses: actions/upload-artifact@v4
      with:
        name: html-report--attempt-${{ github.run_attempt }}
        path: playwright-report
        retention-days: 14
```

이제 보고서가 병합되었고 결합된 HTML 보고서를 GitHub Actions Artifacts 탭에서 사용할 수 있습니다.

## Merge-reports CLI

`npx playwright merge-reports path/to/blob-reports-dir`는 전달된 디렉터리에서 모든 blob 보고서를 읽고 단일 보고서로 병합합니다.

서로 다른 OS의 보고서를 병합할 때 테스트 루트로 사용할 디렉터리를 명확히 하기 위해 명시적 병합 구성을 제공해야 합니다.

지원되는 옵션:
- `--reporter reporter-to-use`
  
  생성할 보고서입니다. 쉼표로 구분된 여러 리포터일 수 있습니다.
  
  예:
  
  ```bash
  npx playwright merge-reports --reporter=html,github ./blob-reports
  ```
  
- `--config path/to/config/file`
  
  출력 리포터가 있는 Playwright 구성 파일을 지정합니다. 이 옵션을 사용하여 추가 구성을 출력 리포터에 전달합니다. 이 구성 파일은 blob 보고서를 생성하는 동안 사용된 파일과 다를 수 있습니다.
  
  예:
  
  ```bash
  npx playwright merge-reports --config=merge.config.ts ./blob-reports
  ```
  
  ```ts title="merge.config.ts"
  export default {
    testDir: 'e2e',
    reporter: [['html', { open: 'never' }]],
  };
  ```
