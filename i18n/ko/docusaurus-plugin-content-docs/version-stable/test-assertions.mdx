## Introduction

Playwright는 `expect` 함수 형태로 테스트 어설션을 포함합니다. 어설션을 만들려면 `expect(value)`를 호출하고 예상과 일치하는 매처를 선택하세요. 모든 조건을 어설션하는 데 사용할 수 있는 `toEqual`, `toContain`, `toBeTruthy`와 같은 많은 [일반 매처](./api/class-genericassertions.mdx)가 있습니다.

```js
expect(success).toBeTruthy();
```

Playwright는 또한 예상 조건이 충족될 때까지 기다리는 웹 관련 [비동기 매처](./api/class-locatorassertions.mdx)를 포함합니다. 다음 예를 고려해 보세요.

```js
await expect(page.getByTestId('status')).toHaveText('Submitted');
```

Playwright는 가져온 요소가 `"Submitted"` 텍스트를 가질 때까지 `status`의 테스트 ID로 요소를 다시 테스트합니다. 조건이 충족되거나 시간 초과에 도달할 때까지 요소를 반복적으로 다시 가져오고 확인합니다. 이 시간 초과를 전달하거나 테스트 구성에서 [testConfig.expect](/api/class-testconfig.mdx#test-config-expect) 값을 통해 한 번 구성할 수 있습니다.

기본적으로 어설션의 시간 초과는 5초로 설정됩니다. [다양한 시간 초과](./test-timeouts.mdx)에 대해 자세히 알아보세요.

## 재시도하지 않는 어설션

이러한 어설션을 사용하면 모든 조건을 테스트할 수 있지만 자동으로 재시도하지 않습니다. 대부분의 경우 웹 페이지는 정보를 비동기적으로 표시하므로 재시도하지 않는 어설션을 사용하면 테스트가 불안정해질 수 있습니다.

가능하면 언제든지 [자동 재시도](#auto-retrying-assertions) 어설션을 선호하세요. 재시도가 필요한 더 복잡한 어설션의 경우 [`expect.poll`](#expectpoll) 또는 [`expect.toPass`](#expecttopass)를 사용하세요.

## 매처 부정

일반적으로 매처 앞에 `.not`을 추가하여 반대가 참인지 확인할 수 있습니다.

```js
expect(value).not.toEqual(0);
await expect(locator).not.toContainText('some text');
```

## 소프트 어설션

기본적으로 어설션이 실패하면 테스트 실행이 종료됩니다. Playwright는 *소프트 어설션*도 지원합니다. 소프트 어설션이 실패하면 테스트 실행이 종료**되지 않지만** 테스트가 실패로 표시됩니다.

```js
// 실패해도 테스트를 중단하지 않는 몇 가지 검사를 수행합니다...
await expect.soft(page.getByTestId('status')).toHaveText('Success');
await expect.soft(page.getByTestId('eta')).toHaveText('1 day');

// ...테스트를 계속하여 더 많은 것을 확인합니다.
await page.getByRole('link', { name: 'next page' }).click();
await expect.soft(page.getByRole('heading', { name: 'Make another order' })).toBeVisible();
```

테스트 실행 중 언제든지 소프트 어설션 실패가 있는지 확인할 수 있습니다.

```js
// 실패해도 테스트를 중단하지 않는 몇 가지 검사를 수행합니다...
await expect.soft(page.getByTestId('status')).toHaveText('Success');
await expect.soft(page.getByTestId('eta')).toHaveText('1 day');

// 소프트 어설션 실패가 있는 경우 더 이상 실행하지 마십시오.
expect(test.info().errors).toHaveLength(0);
```

소프트 어설션은 Playwright 테스트 러너에서만 작동합니다.

## 사용자 지정 expect 메시지

`expect` 함수의 두 번째 인수로 사용자 지정 expect 메시지를 지정할 수 있습니다(예:

```js
await expect(page.getByText('Name'), 'should be logged in').toBeVisible();
```

이 메시지는 어설션에 대한 더 많은 컨텍스트를 제공하여 통과 및 실패 expect 모두에 대해 보고자에 표시됩니다.

expect가 통과되면 다음과 같은 성공적인 단계를 볼 수 있습니다.

```txt
✅ should be logged in    @example.spec.ts:18
```

expect가 실패하면 오류는 다음과 같습니다.

```bash
    Error: should be logged in

    Call log:
      - expect.toBeVisible with timeout 5000ms
      - waiting for "getByText('Name')"


      2 |
      3 | test('example test', async({ page }) => {
    > 4 |   await expect(page.getByText('Name'), 'should be logged in').toBeVisible();
        |                                                                  ^
      5 | });
      6 |
```

소프트 어설션은 사용자 지정 메시지도 지원합니다.

```js
expect.soft(value, 'my soft assertion').toBe(56);
```

## expect.configure

자체 `timeout` 및 `soft`와 같은 자체 기본값을 갖도록 미리 구성된 `expect` 인스턴스를 만들 수 있습니다.

```js
const slowExpect = expect.configure({ timeout: 10000 });
await slowExpect(locator).toHaveText('Submit');

// 항상 소프트 어설션을 수행합니다.
const softExpect = expect.configure({ soft: true });
await softExpect(locator).toHaveText('Submit');
```

## expect.poll

`expect.poll`을 사용하여 모든 동기식 `expect`를 비동기식 폴링으로 변환할 수 있습니다.

다음 메서드는 HTTP 상태 200을 반환할 때까지 지정된 함수를 폴링합니다.

```js
await expect.poll(async () => {
  const response = await page.request.get('https://api.example.com');
  return response.status();
}, {
  // 보고를 위한 사용자 지정 expect 메시지(선택 사항).
  message: 'make sure API eventually succeeds',
  // 10초 동안 폴링합니다. 기본값은 5초입니다. 시간 초과를 비활성화하려면 0을 전달합니다.
  timeout: 10000,
}).toBe(200);
```

사용자 지정 폴링 간격을 지정할 수도 있습니다.

```js
await expect.poll(async () => {
  const response = await page.request.get('https://api.example.com');
  return response.status();
}, {
  // 프로브, 1초 대기, 프로브, 2초 대기, 프로브, 10초 대기, 프로브, 10초 대기, 프로브
  // ... 기본값은 [100, 250, 500, 1000]입니다.
  intervals: [1_000, 2_000, 10_000],
  timeout: 60_000
}).toBe(200);
```

## expect.toPass

성공적으로 통과할 때까지 코드 블록을 재시도할 수 있습니다.

```js
await expect(async () => {
  const response = await page.request.get('https://api.example.com');
  expect(response.status()).toBe(200);
}).toPass();
```

사용자 지정 시간 초과 및 재시도 간격을 지정할 수도 있습니다.

```js
await expect(async () => {
  const response = await page.request.get('https://api.example.com');
  expect(response.status()).toBe(200);
}).toPass({
  // 프로브, 1초 대기, 프로브, 2초 대기, 프로브, 10초 대기, 프로브, 10초 대기, 프로브
  // ... 기본값은 [100, 250, 500, 1000]입니다.
  intervals: [1_000, 2_000, 10_000],
  timeout: 60_000
});
```

기본적으로 `toPass`에는 시간 초과 0이 있으며 사용자 지정 [expect 시간 초과](./test-timeouts.mdx#expect-timeout)를 준수하지 않습니다.

## expect.extend를 사용하여 사용자 지정 매처 추가

사용자 지정 매처를 제공하여 Playwright 어설션을 확장할 수 있습니다. 이러한 매처는 `expect` 객체에서 사용할 수 있습니다.

이 예에서는 사용자 지정 `toHaveAmount` 함수를 추가합니다. 사용자 지정 매처는 어설션이 통과되었는지 여부를 나타내는 `pass` 플래그와 어설션이 실패할 때 사용되는 `message` 콜백을 반환해야 합니다.

```js title="fixtures.ts"
import { expect as baseExpect } from '@playwright/test';
import type { Page, Locator } from '@playwright/test';

export { test } from '@playwright/test';

export const expect = baseExpect.extend({
  async toHaveAmount(locator: Locator, expected: number, options?: { timeout?: number }) {
    const assertionName = 'toHaveAmount';
    let pass: boolean;
    let matcherResult: any;
    try {
      await baseExpect(locator).toHaveAttribute('data-amount', String(expected), options);
      pass = true;
    } catch (e: any) {
      matcherResult = e.matcherResult;
      pass = false;
    }

    const message = pass
      ? () => this.utils.matcherHint(assertionName, undefined, undefined, { isNot: this.isNot }) +
          '\n\n' +
          `Locator: ${locator}\n` +
          `Expected: not ${this.utils.printExpected(expected)}\n` +
          (matcherResult ? `Received: ${this.utils.printReceived(matcherResult.actual)}` : '')
      : () =>  this.utils.matcherHint(assertionName, undefined, undefined, { isNot: this.isNot }) +
          '\n\n' +
          `Locator: ${locator}\n` +
          `Expected: ${this.utils.printExpected(expected)}\n` +
          (matcherResult ? `Received: ${this.utils.printReceived(matcherResult.actual)}` : '');

    return {
      message,
      pass,
      name: assertionName,
      expected,
      actual: matcherResult?.actual,
    };
  },
});
```

이제 테스트에서 `toHaveAmount`를 사용할 수 있습니다.

```js title="example.spec.ts"
import { test, expect } from './fixtures';

test('amount', async () => {
  await expect(page.locator('.cart')).toHaveAmount(4);
});
```

### expect 라이브러리와의 호환성

:::note

Playwright의 `expect`를 [`expect` 라이브러리](https://jestjs.io/docs/expect)와 혼동하지 마십시오. 후자는 Playwright 테스트 러너와 완전히 통합되지 않으므로 Playwright 자체 `expect`를 사용해야 합니다.
:::

### 여러 모듈의 사용자 지정 매처 결합

여러 파일 또는 모듈의 사용자 지정 매처를 결합할 수 있습니다.

```js title="fixtures.ts"
import { mergeTests, mergeExpects } from '@playwright/test';
import { test as dbTest, expect as dbExpect } from 'database-test-utils';
import { test as a11yTest, expect as a11yExpect } from 'a11y-test-utils';

export const expect = mergeExpects(dbExpect, a11yExpect);
export const test = mergeTests(dbTest, a11yTest);
```

```js title="test.spec.ts"
import { test, expect } from './fixtures';

test('passes', async ({ database }) => {
  await expect(database).toHaveDatabaseUser('admin');
});
```
