## Introduction

Playwright Test는 이제 컴포넌트를 테스트할 수 있습니다.

## Example

다음은 일반적인 컴포넌트 테스트의 모습입니다.

```js
test('event should work', async ({ mount }) => {
  let clicked = false;

  // 컴포넌트를 마운트합니다. 컴포넌트를 가리키는 로케이터를 반환합니다.
  const component = await mount(
    <Button title="Submit" onClick={() => { clicked = true }}></Button>
  );

  // 모든 Playwright 테스트와 마찬가지로 로케이터 텍스트를 단언합니다.
  await expect(component).toContainText('Submit');

  // 로케이터 클릭을 수행합니다. 이벤트가 트리거됩니다.
  await component.click();

  // 해당 이벤트가 발생했는지 단언합니다.
  expect(clicked).toBeTruthy();
});
```

## Test stories

Playwright Test가 웹 컴포넌트를 테스트하는 데 사용되면 테스트는 Node.js에서 실행되는 반면 컴포넌트는 실제 브라우저에서 실행됩니다. 이를 통해 최고의 장점을 모두 누릴 수 있습니다. 즉, 컴포넌트는 실제 브라우저 환경에서 실행되고, 실제 클릭이 트리거되고, 실제 레이아웃이 실행되고, 시각적 회귀가 가능합니다. 동시에 테스트는 Node.js의 모든 기능은 물론 모든 Playwright Test 기능을 사용할 수 있습니다. 결과적으로 컴포넌트 테스트 중에도 동일한 병렬, 매개변수화된 테스트와 동일한 사후 분석 추적 스토리를 사용할 수 있습니다.

그러나 이로 인해 다음과 같은 여러 가지 제한 사항이 발생합니다.
- 복잡한 라이브 객체를 컴포넌트에 전달할 수 없습니다. 일반 JavaScript 객체와 문자열, 숫자, 날짜 등과 같은 내장 유형만 전달할 수 있습니다.

```js
test('this will work', async ({ mount }) => {
  const component = await mount(<ProcessViewer process={{ name: 'playwright' }}/>);
});

test('this will not work', async ({ mount }) => {
  // `process`는 Node 객체이며, 브라우저에 전달하여 작동할 것으로 기대할 수 없습니다.
  const component = await mount(<ProcessViewer process={process}/>);
});
```

- 콜백에서 데이터를 동기적으로 컴포넌트에 전달할 수 없습니다.

```js
test('this will not work', async ({ mount }) => {
  // () => 'red' 콜백은 Node에 있습니다. 브라우저의 `ColorPicker` 컴포넌트가 파라미터 함수를 호출하는 경우
  // `colorGetter`는 결과를 동기적으로 얻지 못합니다. await를 통해 결과를 얻을 수 있지만, 일반적으로
  // 컴포넌트는 그렇게 구축되지 않습니다.
  const component = await mount(<ColorPicker colorGetter={() => 'red'}/>);
});
```

이러한 제한 사항 및 기타 제한 사항을 해결하는 것은 빠르고 간단합니다. 테스트할 컴포넌트의 모든 사용 사례에 대해 테스트를 위해 특별히 설계된 이 컴포넌트의 래퍼를 만듭니다. 이렇게 하면 제한 사항이 완화될 뿐만 아니라 컴포넌트 렌더링의 환경, 테마 및 기타 측면을 정의할 수 있는 강력한 테스트 추상화를 제공합니다.

다음 컴포넌트를 테스트하고 싶다고 가정해 보겠습니다.

```js title="input-media.tsx"
import React from 'react';

type InputMediaProps = {
  // Media는 테스트 중에 Node로 보낼 수 없는 복잡한 브라우저 객체입니다.
  onChange(media: Media): void;
};

export function InputMedia(props: InputMediaProps) {
  return <></> as any;
}
```

컴포넌트에 대한 스토리 파일을 만듭니다.

```js title="input-media.story.tsx"
import React from 'react';
import InputMedia from './import-media';

type InputMediaForTestProps = {
  onMediaChange(mediaName: string): void;
};

export function InputMediaForTest(props: InputMediaForTestProps) {
  // 복잡한 `media` 객체를 테스트로 보내는 대신 미디어 이름을 보냅니다.
  return <InputMedia onChange={media => props.onMediaChange(media.name)} />;
}
// 여기에 더 많은 스토리를 내보냅니다.
```

그런 다음 스토리를 테스트하여 컴포넌트를 테스트합니다.

```js title="input-media.spec.tsx"
import { test, expect } from '@playwright/experimental-ct-react';
import { InputMediaForTest } from './input-media.story.tsx';

test('changes the image', async ({ mount }) => {
  let mediaSelected: string | null = null;

  const component = await mount(
    <InputMediaForTest
      onMediaChange={mediaName => {
        mediaSelected = mediaName;
      }}
    />
  );
  await component
    .getByTestId('imageInput')
    .setInputFiles('src/assets/logo.png');

  await expect(component.getByAltText(/selected image/i)).toBeVisible();
  await expect.poll(() => mediaSelected).toBe('logo.png');
});
```

결과적으로 모든 컴포넌트에 대해 실제로 테스트되는 모든 스토리를 내보내는 스토리 파일이 있습니다. 이러한 스토리는 브라우저에 있으며 복잡한 객체를 테스트에서 액세스할 수 있는 간단한 객체로 "변환"합니다.

## Under the hood

컴포넌트 테스트 작동 방식은 다음과 같습니다.
- 테스트가 실행되면 Playwright는 테스트에 필요한 컴포넌트 목록을 만듭니다.
- 그런 다음 이러한 컴포넌트를 포함하는 번들을 컴파일하고 로컬 정적 웹 서버를 사용하여 제공합니다.
- 테스트 내에서 `mount` 호출 시 Playwright는 이 번들의 파사드 페이지 `/playwright/index.html`로 이동하여 컴포넌트를 렌더링하도록 지시합니다.
- 이벤트는 확인을 허용하기 위해 Node.js 환경으로 다시 마샬링됩니다.

Playwright는 [Vite](https://vitejs.dev/)를 사용하여 컴포넌트 번들을 만들고 제공합니다.

### hooks

`beforeMount` 및 `afterMount` 후크를 사용하여 앱을 구성할 수 있습니다. 이렇게 하면 앱 라우터, 가짜 서버 등과 같은 항목을 설정하여 필요한 유연성을 제공할 수 있습니다. 또한 `hooksConfig` 픽스처에서 액세스할 수 있는 테스트에서 `mount` 호출에서 사용자 지정 구성을 전달할 수도 있습니다. 여기에는 컴포넌트를 마운트하기 전이나 후에 실행해야 하는 모든 구성이 포함됩니다. 라우터를 구성하는 예제가 아래에 제공되어 있습니다.

<Tabs defaultValue="react" values={[ {label: 'React', value: 'react'}, {label: 'Vue3', value: 'vue3'}, ] }>

<TabItem value="react">

```js title="playwright/index.tsx"
import { beforeMount, afterMount } from '@playwright/experimental-ct-react/hooks';
import { BrowserRouter } from 'react-router-dom';

export type HooksConfig = {
  enableRouting?: boolean;
}

beforeMount<HooksConfig>(async ({ App, hooksConfig }) => {
  if (hooksConfig?.enableRouting)
    return <BrowserRouter><App /></BrowserRouter>;
});
```

```js title="src/pages/ProductsPage.spec.tsx"
import { test, expect } from '@playwright/experimental-ct-react';
import type { HooksConfig } from '../playwright';
import { ProductsPage } from './pages/ProductsPage';

test('configure routing through hooks config', async ({ page, mount }) => {
  const component = await mount<HooksConfig>(<ProductsPage />, {
    hooksConfig: { enableRouting: true },
  });
  await expect(component.getByRole('link')).toHaveAttribute('href', '/products/42');
});
```

</TabItem>

<TabItem value="vue3">

### update

### Handling network requests

Playwright는 네트워크 요청을 가로채고 처리하기 위한 **실험적** `router` 픽스처를 제공합니다. `router` 픽스처를 사용하는 방법에는 두 가지가 있습니다.
* [page.route()](/api/class-page.mdx#page-route)와 유사하게 동작하는 `router.route(url, handler)`를 호출합니다. 자세한 내용은 [네트워크 모의 가이드](./mock.mdx)를 참조하세요.
* `router.use(handlers)`를 호출하고 [MSW 라이브러리](https://mswjs.io/) 요청 처리기를 전달합니다.

다음은 테스트에서 기존 MSW 처리기를 재사용하는 예입니다.

```ts
import { handlers } from '@src/mocks/handlers';

test.beforeEach(async ({ router }) => {
  // 각 테스트 전에 공통 처리기를 설치합니다.
  await router.use(...handlers);
});

test('example test', async ({ mount }) => {
  // 평소와 같이 테스트합니다. 처리기가 활성 상태입니다.
  // ...
});
```

특정 테스트에 대해 일회성 처리기를 도입할 수도 있습니다.

```ts
import { http, HttpResponse } from 'msw';

test('example test', async ({ mount, router }) => {
  await router.use(http.get('/data', async ({ request }) => {
    return HttpResponse.json({ value: 'mocked' });
  }));

  // 평소와 같이 테스트합니다. 처리기가 활성 상태입니다.
  // ...
});
```

### I have a project that already uses Vite. Can I reuse the config?

현재 Playwright는 번들러에 구애받지 않으므로 기존 Vite 구성을 재사용하지 않습니다. 구성에 재사용할 수 없는 많은 항목이 있을 수 있습니다. 따라서 지금은 경로 매핑 및 기타 상위 수준 설정을 Playwright 구성의 `ctViteConfig` 속성에 복사합니다.

```js
import { defineConfig } from '@playwright/experimental-ct-react';

export default defineConfig({
  use: {
    ctViteConfig: {
      // ...
    },
  },
});
```

테스트 설정에 대한 Vite 구성을 통해 플러그인을 지정할 수 있습니다. 플러그인 지정을 시작하면 프레임워크 플러그인(`vue()` in this case)도 지정해야 합니다.

```js
import { defineConfig, devices } from '@playwright/experimental-ct-vue';

import { resolve } from 'path';
import vue from '@vitejs/plugin-vue';
import AutoImport from 'unplugin-auto-import/vite';
import Components from 'unplugin-vue-components/vite';

export default defineConfig({
  testDir: './tests/component',
  use: {
    trace: 'on-first-retry',
    ctViteConfig: {
      plugins: [
        vue(),
        AutoImport({
          imports: [
            'vue',
            'vue-router',
            '@vueuse/head',
            'pinia',
            {
              '@/store': ['useStore'],
            },
          ],
          dts: 'src/auto-imports.d.ts',
          eslintrc: {
            enabled: true,
          },
        }),
        Components({
          dirs: ['src/components'],
          extensions: ['vue'],
        }),
      ],
      resolve: {
        alias: {
          '@': resolve(__dirname, './src'),
        },
      },
    },
  },
});
```

### How do I use CSS imports?

CSS를 가져오는 컴포넌트가 있는 경우 Vite가 자동으로 처리합니다. Sass, Less 또는 Stylus와 같은 CSS 전처리기를 사용할 수도 있으며 Vite는 추가 구성 없이도 이를 처리합니다. 그러나 해당 CSS 전처리기를 설치해야 합니다.

Vite는 모든 CSS 모듈의 이름을 `*.module.[css extension]`로 지정해야 하는 엄격한 요구 사항이 있습니다. 프로젝트에 대한 사용자 지정 빌드 구성이 있고 일반적으로 `import styles from 'styles.css'` 형식으로 가져오는 경우 파일을 모듈로 처리되도록 적절하게 이름을 바꿔야 합니다. 또는 이를 처리하기 위해 Vite 플러그인을 작성할 수도 있습니다.

자세한 내용은 [Vite 설명서](https://vite.dev/guide/features#css)를 확인하세요.

### How can I test components that uses Pinia?

Pinia는 `playwright/index.{js,ts,jsx,tsx}`에서 초기화해야 합니다. `beforeMount` 후크 내부에서 이를 수행하면 `initialState`를 테스트별로 덮어쓸 수 있습니다.

```js title="playwright/index.ts"
import { beforeMount, afterMount } from '@playwright/experimental-ct-vue/hooks';
import { createTestingPinia } from '@pinia/testing';
import type { StoreState } from 'pinia';
import type { useStore } from '../src/store';

export type HooksConfig = {
  store?: StoreState<ReturnType<typeof useStore>>;
}

beforeMount<HooksConfig>(async ({ hooksConfig }) => {
  createTestingPinia({
    initialState: hooksConfig?.store,
    /**
     * http intercepting을 사용하여 API 호출을 mock 하십시오:
     * https://playwright.dev/docs/mock#mock-api-requests
     */
    stubActions: false,
    createSpy(args) {
      console.log('spy', args)
      return () => console.log('spy-returns')
    },
  });
});
```

```js title="src/pinia.spec.ts"
import { test, expect } from '@playwright/experimental-ct-vue';
import type { HooksConfig } from '../playwright';
import Store from './Store.vue';

test('override initialState ', async ({ mount }) => {
  const component = await mount<HooksConfig>(Store, {
    hooksConfig: {
      store: { name: 'override initialState' }
    }
  });
  await expect(component).toContainText('override initialState');
});
```

### How do I access the component's methods or its instance?

테스트 코드 내에서 컴포넌트의 내부 메서드 또는 인스턴스에 액세스하는 것은 권장되지 않으며 지원되지도 않습니다. 대신 일반적으로 클릭하거나 페이지에 무엇인가가 보이는지 확인하여 사용자 관점에서 컴포넌트를 관찰하고 상호 작용하는 데 집중하십시오. 테스트는 컴포넌트 인스턴스 또는 해당 메서드와 같은 내부 구현 세부 사항과의 상호 작용을 피할 때 덜 취약하고 더 가치가 있습니다. 사용자 관점에서 실행할 때 테스트가 실패하면 자동화된 테스트가 코드에서 실제 버그를 발견했을 가능성이 높습니다.
