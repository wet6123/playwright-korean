## 소개

Playwright는 애플리케이션의 다양한 유형의 접근성 문제를 테스트하는 데 사용할 수 있습니다.

이를 통해 다음과 같은 문제를 발견할 수 있습니다:

- 배경색과의 대비가 낮아 시각 장애가 있는 사용자가 읽기 어려운 텍스트
- 스크린 리더가 식별할 수 없는 레이블이 없는 UI 컨트롤 및 폼 요소
- 보조 기술을 혼란스럽게 할 수 있는 중복 ID를 가진 상호작용 요소

다음 예제는 Playwright 테스트의 일부로 [axe 접근성 테스트 엔진](https://www.deque.com/axe/)을 실행하기 위한 지원을 추가하는 [`@axe-core/playwright`](https://npmjs.org/@axe-core/playwright) 패키지에 의존합니다.

:::note[주의사항]

자동화된 접근성 테스트는 누락되거나 유효하지 않은 속성과 같은 일반적인 접근성 문제를 감지할 수 있습니다. 그러나 많은 접근성 문제는 수동 테스트를 통해서만 발견할 수 있습니다. 자동화된 테스트, 수동 접근성 평가 및 포용적인 사용자 테스트를 조합하여 사용하는 것을 권장합니다.

수동 평가의 경우, [Accessibility Insights for Web](https://accessibilityinsights.io/docs/web/overview/?referrer=playwright-accessibility-testing-js)을 권장합니다. 이는 웹사이트의 [WCAG 2.1 AA](https://www.w3.org/WAI/WCAG21/quickref/?currentsidebar=%23col_customize&levels=aaa) 준수 여부를 평가하는 무료 오픈 소스 개발 도구입니다.
:::

## 접근성 테스트 예제

접근성 테스트는 다른 Playwright 테스트와 마찬가지로 작동합니다. 별도의 테스트 케이스를 만들거나, 기존 테스트 케이스에 접근성 스캔 및 검증을 통합할 수 있습니다.

다음 예제는 몇 가지 기본적인 접근성 테스트 시나리오를 보여줍니다.

### 전체 페이지 스캔하기

이 예제는 자동으로 감지 가능한 접근성 위반 사항에 대해 전체 페이지를 테스트하는 방법을 보여줍니다. 테스트는:

1. `@axe-core/playwright` 패키지를 가져옵니다
2. 일반 Playwright Test 구문을 사용하여 테스트 케이스를 정의합니다
3. 일반 Playwright 구문을 사용하여 테스트할 페이지로 이동합니다
4. 페이지에 대한 접근성 스캔을 실행하기 위해 `AxeBuilder.analyze()`를 기다립니다
5. 일반 Playwright Test [검증](./test-assertions)을 사용하여 반환된 스캔 결과에 위반 사항이 없는지 확인합니다

### 페이지의 특정 부분을 스캔하도록 axe 구성하기

`@axe-core/playwright`는 axe에 대한 많은 구성 옵션을 지원합니다. `AxeBuilder` 클래스와 함께 빌더 패턴을 사용하여 이러한 옵션을 지정할 수 있습니다.

예를 들어, [`AxeBuilder.include()`](https://github.com/dequelabs/axe-core-npm/blob/develop/packages/playwright/README.md#axebuilderincludeselector-string--string)를 사용하여 접근성 스캔을 페이지의 특정 부분에서만 실행하도록 제한할 수 있습니다.

`AxeBuilder.analyze()`는 호출할 때 페이지의 *현재 상태*를 스캔합니다. UI 상호작용을 기반으로 나타나는 페이지 부분을 스캔하려면 `analyze()`를 호출하기 전에 [Locators](./locators.mdx)를 사용하여 페이지와 상호작용하세요:

```js
test('navigation menu should not have automatically detectable accessibility violations', async ({
  page,
}) => {
  await page.goto('https://your-site.com/');

  await page.getByRole('button', { name: 'Navigation Menu' }).click();

  // analyze()를 실행하기 *전에* 페이지가 원하는 상태에 있도록
  // waitFor()를 사용하는 것이 중요합니다. 그렇지 않으면 axe가
  // 테스트가 스캔하기를 기대하는 모든 요소를 찾지 못할 수 있습니다.
  await page.locator('#navigation-menu-flyout').waitFor();

  const accessibilityScanResults = await new AxeBuilder({ page })
    .include('#navigation-menu-flyout')
    .analyze();

  expect(accessibilityScanResults.violations).toEqual([]);
});
```

### WCAG 위반 사항 스캔하기

기본적으로 axe는 다양한 접근성 규칙을 검사합니다. 이러한 규칙 중 일부는 [웹 콘텐츠 접근성 지침(WCAG)](https://www.w3.org/TR/WCAG21/)의 특정 성공 기준에 해당하며, 일부는 WCAG 기준에서 특별히 요구하지 않는 "모범 사례" 규칙입니다.

[`AxeBuilder.withTags()`](https://github.com/dequelabs/axe-core-npm/blob/develop/packages/playwright/README.md#axebuilderwithtagstags-stringarray)를 사용하여 특정 WCAG 성공 기준에 해당하는 "태그"가 지정된 규칙만 실행하도록 접근성 스캔을 제한할 수 있습니다. 예를 들어, [Accessibility Insights for Web의 자동화된 검사](https://accessibilityinsights.io/docs/web/getstarted/fastpass/?referrer=playwright-accessibility-testing-js)는 WCAG A 및 AA 성공 기준 위반을 테스트하는 axe 규칙만 포함합니다. 이 동작을 일치시키기 위해 `wcag2a`, `wcag2aa`, `wcag21a`, `wcag21aa` 태그를 사용할 수 있습니다.

자동화된 테스트는 모든 유형의 WCAG 위반을 감지할 수 없다는 점에 유의하세요.

```js
test('should not have any automatically detectable WCAG A or AA violations', async ({
  page,
}) => {
  await page.goto('https://your-site.com/');

  const accessibilityScanResults = await new AxeBuilder({ page })
    .withTags(['wcag2a', 'wcag2aa', 'wcag21a', 'wcag21aa'])
    .analyze();

  expect(accessibilityScanResults.violations).toEqual([]);
});
```

axe-core가 지원하는 규칙 태그의 전체 목록은 [axe API 문서의 "Axe-core Tags" 섹션](https://www.deque.com/axe/core-documentation/api-documentation/#axecore-tags)에서 확인할 수 있습니다.

## 알려진 문제 처리하기

애플리케이션에 접근성 테스트를 추가할 때 일반적인 질문은 "알려진 위반 사항을 어떻게 억제할 수 있을까요?"입니다. 다음 예제는 사용할 수 있는 몇 가지 기술을 보여줍니다.

### 스캔에서 개별 요소 제외하기

애플리케이션에 알려진 문제가 있는 특정 요소가 몇 개 있는 경우, [`AxeBuilder.exclude()`](https://github.com/dequelabs/axe-core-npm/blob/develop/packages/playwright/README.md#axebuilderexcludeselector-string--string)를 사용하여 문제를 수정할 수 있을 때까지 스캔에서 해당 요소를 제외할 수 있습니다.

이는 일반적으로 가장 간단한 옵션이지만 중요한 단점이 있습니다:

- `exclude()`는 지정된 요소 *및 모든 하위 요소*를 제외합니다. 많은 자식 요소가 포함된 컴포넌트에는 사용하지 마세요.
- `exclude()`는 지정된 요소에 대해 알려진 문제에 해당하는 규칙만이 아니라 _모든_ 규칙이 실행되는 것을 방지합니다.

다음은 한 특정 테스트에서 하나의 요소를 스캔에서 제외하는 예입니다:

```js
test('should not have any accessibility violations outside of elements with known issues', async ({
  page,
}) => {
  await page.goto('https://your-site.com/page-with-known-issues');

  const accessibilityScanResults = await new AxeBuilder({ page })
    .exclude('#element-with-known-issue')
    .analyze();

  expect(accessibilityScanResults.violations).toEqual([]);
});
```

해당 요소가 여러 페이지에서 반복적으로 사용되는 경우, [테스트 픽스처 사용하기](#여러-테스트에서-공통-axe-구성에-대한-테스트-픽스처-사용하기)를 고려하여 여러 테스트에서 동일한 `AxeBuilder` 구성을 재사용하세요.

### 개별 스캔 규칙 비활성화하기

애플리케이션에 특정 규칙의 여러 기존 위반 사항이 포함된 경우, [`AxeBuilder.disableRules()`](https://github.com/dequelabs/axe-core-npm/blob/develop/packages/playwright/README.md#axebuilderdisablerulesrules-stringarray)를 사용하여 문제를 수정할 수 있을 때까지 개별 규칙을 일시적으로 비활성화할 수 있습니다.

`disableRules()`에 전달할 규칙 ID는 억제하려는 위반 사항의 `id` 속성에서 찾을 수 있습니다. [axe 규칙의 전체 목록](https://github.com/dequelabs/axe-core/blob/master/doc/rule-descriptions.md)은 `axe-core` 문서에서 찾을 수 있습니다.

```js
test('should not have any accessibility violations outside of rules with known issues', async ({
  page,
}) => {
  await page.goto('https://your-site.com/page-with-known-issues');

  const accessibilityScanResults = await new AxeBuilder({ page })
    .disableRules(['duplicate-id'])
    .analyze();

  expect(accessibilityScanResults.violations).toEqual([]);
});
```

### 특정 알려진 문제를 허용하기 위해 스냅샷 사용하기

더 세분화된 알려진 문제 집합을 허용하려면 [스냅샷](./test-snapshots.mdx)을 사용하여 기존 위반 사항 집합이 변경되지 않았는지 확인할 수 있습니다. 이 접근 방식은 약간의 복잡성과 취약성을 대가로 `AxeBuilder.exclude()`를 사용하는 단점을 피할 수 있습니다.

전체 `accessibilityScanResults.violations` 배열의 스냅샷을 사용하지 마세요. 이 배열에는 렌더링된 HTML의 스니펫과 같은 해당 요소의 구현 세부 정보가 포함되어 있습니다. 이러한 정보를 스냅샷에 포함하면 관련 없는 이유로 해당 컴포넌트 중 하나가 변경될 때마다 테스트가 깨질 가능성이 높아집니다:

```js
// 이렇게 하지 마세요! 이는 취약합니다.
expect(accessibilityScanResults.violations).toMatchSnapshot();
```

대신, 문제를 고유하게 식별하기 위한 충분한 정보만 포함하는 위반 사항의 *지문*을 만들고 해당 지문의 스냅샷을 사용하세요:

```js
// 전체 위반 배열을 스냅샷하는 것보다 덜 취약합니다.
expect(violationFingerprints(accessibilityScanResults)).toMatchSnapshot();

// my-test-utils.js
function violationFingerprints(accessibilityScanResults) {
  const violationFingerprints = accessibilityScanResults.violations.map(
    (violation) => ({
      rule: violation.id,
      // 이는 해당 규칙의 위반이 있는 각 요소를 고유하게 식별하는
      // CSS 선택자입니다.
      targets: violation.nodes.map((node) => node.target),
    })
  );

  return JSON.stringify(violationFingerprints, null, 2);
}
```

## 스캔 결과를 테스트 첨부 파일로 내보내기

대부분의 접근성 테스트는 주로 axe 스캔 결과의 `violations` 속성에 관심이 있습니다. 그러나 스캔 결과에는 `violations` 외에도 더 많은 정보가 포함되어 있습니다. 예를 들어, 결과에는 통과한 규칙과 일부 규칙에 대해 axe가 결론을 내리지 못한 요소에 대한 정보도 포함되어 있습니다. 이 정보는 예상한 모든 위반 사항을 감지하지 못하는 테스트를 디버깅하는 데 유용할 수 있습니다.

디버깅 목적으로 _모든_ 스캔 결과를 테스트 결과의 일부로 포함하려면 [`testInfo.attach()`](./api/class-testinfo#test-info-attach)를 사용하여 스캔 결과를 테스트 첨부 파일로 추가할 수 있습니다. [리포터](./test-reporters)는 테스트 출력의 일부로 전체 결과를 포함하거나 링크할 수 있습니다.

다음 예제는 테스트에 스캔 결과를 첨부하는 방법을 보여줍니다:

```js
test('example with attachment', async ({ page }, testInfo) => {
  await page.goto('https://your-site.com/');

  const accessibilityScanResults = await new AxeBuilder({ page }).analyze();

  await testInfo.attach('accessibility-scan-results', {
    body: JSON.stringify(accessibilityScanResults, null, 2),
    contentType: 'application/json',
  });

  expect(accessibilityScanResults.violations).toEqual([]);
});
```

## 여러 테스트에서 공통 axe 구성에 대한 테스트 픽스처 사용하기

[테스트 픽스처](./test-fixtures)는 여러 테스트에서 공통 `AxeBuilder` 구성을 공유하는 좋은 방법입니다. 이것이 유용할 수 있는 몇 가지 시나리오는 다음과 같습니다:

- 모든 테스트에서 공통 규칙 집합 사용하기
- 여러 다른 페이지에 나타나는 공통 요소의 알려진 위반 사항 억제하기
- 많은 스캔에 대해 독립 실행형 접근성 보고서를 일관되게 첨부하기

다음 예제는 이러한 각 시나리오를 다루는 테스트 픽스처를 만들고 사용하는 방법을 보여줍니다.

### 픽스처 만들기

이 예제 픽스처는 공유된 `withTags()` 및 `exclude()` 구성으로 미리 구성된 `AxeBuilder` 객체를 만듭니다.

```js title="axe-test.ts"
import { test as base } from '@playwright/test';
import AxeBuilder from '@axe-core/playwright';

type AxeFixture = {
  makeAxeBuilder: () => AxeBuilder,
};

// "makeAxeBuilder"를 제공하여 기본 테스트 확장
//
// 이 새로운 "test"는 여러 테스트 파일에서 사용할 수 있으며, 각 파일은
// 일관되게 구성된 AxeBuilder 인스턴스를 얻게 됩니다.
export const test =
  base.extend <
  AxeFixture >
  {
    makeAxeBuilder: async ({ page }, use, testInfo) => {
      const makeAxeBuilder = () =>
        new AxeBuilder({ page })
          .withTags(['wcag2a', 'wcag2aa', 'wcag21a', 'wcag21aa'])
          .exclude('#commonly-reused-element-with-known-issue');

      await use(makeAxeBuilder);
    },
  };
export { expect } from '@playwright/test';
```

### 픽스처 사용하기

픽스처를 사용하려면 이전 예제의 `new AxeBuilder({ page })`를 새로 정의된 `makeAxeBuilder` 픽스처로 대체하세요:

```js
const { test, expect } = require('./axe-test');

test('example using custom fixture', async ({ page, makeAxeBuilder }) => {
  await page.goto('https://your-site.com/');

  const accessibilityScanResults = await makeAxeBuilder()
    // 자동으로 공유 AxeBuilder 구성을 사용하지만,
    // 추가적인 테스트별 구성도 지원합니다
    .include('#specific-element-under-test')
    .analyze();

  expect(accessibilityScanResults.violations).toEqual([]);
});
```
