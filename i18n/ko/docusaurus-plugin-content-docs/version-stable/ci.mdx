## Introduction

Playwright 테스트는 CI 환경에서 실행할 수 있습니다. 일반적인 CI 제공업체에 대한 샘플 구성을 만들었습니다.

CI에서 테스트를 실행하는 3단계:
1. **CI 에이전트가 브라우저를 실행할 수 있는지 확인**: Linux 에이전트에서 [Docker 이미지](./docker.mdx)를 사용하거나 [CLI](./browsers#install-system-dependencies)를 사용하여 종속성을 설치합니다.
1. **Playwright 설치**:
   
   ```bash
   # NPM 패키지 설치
   npm ci
   
   # Playwright 브라우저 및 종속성 설치
   npx playwright install --with-deps
   ```
   
1. **테스트 실행**:
   
   ```bash
   npx playwright test
   ```
   
## Workers

안정성과 재현성을 우선시하기 위해 CI 환경에서 [workers](./api/class-testconfig.mdx#test-config-workers)를 "1"로 설정하는 것이 좋습니다. 테스트를 순차적으로 실행하면 각 테스트가 전체 시스템 리소스를 확보하여 잠재적인 충돌을 피할 수 있습니다. 그러나 강력한 자체 호스팅 CI 시스템이 있는 경우 [병렬](./test-parallel.mdx) 테스트를 활성화할 수 있습니다. 더 넓은 병렬 처리를 위해 [샤딩](./test-parallel.mdx#shard-tests-between-multiple-machines)을 고려하십시오. 여러 CI 작업에 테스트를 분산합니다.

```js title="playwright.config.ts"
import { defineConfig, devices } from '@playwright/test';

export default defineConfig({
  // CI에서 병렬 테스트를 선택 해제합니다.
  workers: process.env.CI ? 1 : undefined,
});
```

## CI configurations

[명령줄 도구](./browsers#install-system-dependencies)를 사용하여 CI에서 모든 운영 체제 종속성을 설치할 수 있습니다.

### GitHub Actions

#### On push/pull_request

테스트는 main/master 브랜치에 대한 push 또는 pull request에서 실행됩니다. [워크플로](https://docs.github.com/en/actions/using-workflows/about-workflows)는 모든 종속성을 설치하고, Playwright를 설치한 다음 테스트를 실행합니다. 또한 HTML 보고서를 생성합니다.

```yml title=".github/workflows/playwright.yml"
name: Playwright Tests
on:
  push:
    branches: [ main, master ]
  pull_request:
    branches: [ main, master ]
jobs:
  test:
    timeout-minutes: 60
    runs-on: ubuntu-latest
    steps:
    - uses: actions/checkout@v4
    - uses: actions/setup-node@v4
      with:
        node-version: lts/*
    - name: Install dependencies
      run: npm ci
    - name: Install Playwright Browsers
      run: npx playwright install --with-deps
    - name: Run Playwright tests
      run: npx playwright test
    - uses: actions/upload-artifact@v4
      if: ${{ !cancelled() }}
      with:
        name: playwright-report
        path: playwright-report/
        retention-days: 30
```

#### On push/pull_request (sharded)

GitHub Actions는 [여러 작업 간의 샤딩 테스트](https://docs.github.com/en/actions/using-jobs/using-a-matrix-for-your-jobs)를 지원합니다. 샤딩에 대해 자세히 알아보고 여러 장비에서 테스트를 실행하도록 작업을 구성하는 방법과 HTML 보고서를 병합하는 방법을 보려면 [샤딩 문서](./test-sharding)를 확인하십시오. [GitHub Actions 예제](./test-sharding.mdx#github-actions-example)를 확인하십시오.

#### Via Containers

GitHub Actions는 [`jobs.<job_id>.container`](https://docs.github.com/en/actions/using-workflows/workflow-syntax-for-github-actions#jobsjob_idcontainer) 옵션을 사용하여 [컨테이너에서 작업 실행](https://docs.github.com/en/actions/using-jobs/running-jobs-in-a-container)을 지원합니다. 이는 호스트 환경을 종속성으로 오염시키지 않고 다양한 운영 체제에서 스크린샷/시각적 회귀 테스트를 위한 일관된 환경을 갖는 데 유용합니다.

```yml title=".github/workflows/playwright.yml"
name: Playwright Tests
on:
  push:
    branches: [ main, master ]
  pull_request:
    branches: [ main, master ]
jobs:
  playwright:
    name: 'Playwright Tests'
    runs-on: ubuntu-latest
    container:
      image: mcr.microsoft.com/playwright:v1.51.1-noble
      options: --user 1001
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: lts/*
      - name: Install dependencies
        run: npm ci
      - name: Run your tests
        run: npx playwright test
```

#### On deployment

이는 [GitHub 배포](https://developer.github.com/v3/repos/deployments/)가 `success` 상태로 전환된 후 테스트를 시작합니다. Vercel과 같은 서비스는 이 패턴을 사용하므로 배포된 환경에서 엔드 투 엔드 테스트를 실행할 수 있습니다.

```yml title=".github/workflows/playwright.yml"
name: Playwright Tests
on:
  deployment_status:
jobs:
  test:
    timeout-minutes: 60
    runs-on: ubuntu-latest
    if: github.event.deployment_status.state == 'success'
    steps:
    - uses: actions/checkout@v4
    - uses: actions/setup-node@v4
      with:
        node-version: lts/*
    - name: Install dependencies
      run: npm ci
    - name: Install Playwright
      run: npx playwright install --with-deps
    - name: Run Playwright tests
      run: npx playwright test
      env:
        PLAYWRIGHT_TEST_BASE_URL: ${{ github.event.deployment_status.target_url }}
```

#### Fail-Fast

대규모 테스트 스위트는 실행하는 데 매우 오랜 시간이 걸릴 수 있습니다. `--only-changed` 플래그를 사용하여 예비 테스트 실행을 실행하면 실패할 가능성이 있는 테스트 파일을 먼저 실행할 수 있습니다. 이렇게 하면 풀 요청 작업 시 더 빠른 피드백 루프와 약간 더 낮은 CI 소비를 얻을 수 있습니다. 변경 사항의 영향을 받는 테스트 파일을 감지하기 위해 `--only-changed`는 스위트의 종속성 그래프를 분석합니다. 이는 휴리스틱이며 테스트를 놓칠 수 있으므로 예비 테스트 실행 후 항상 전체 테스트 스위트를 실행하는 것이 중요합니다.

```yml title=".github/workflows/playwright.yml" {24-26}
name: Playwright Tests
on:
  push:
    branches: [ main, master ]
  pull_request:
    branches: [ main, master ]
jobs:
  test:
    timeout-minutes: 60
    runs-on: ubuntu-latest
    steps:
    - uses: actions/checkout@v4
      with:
        # $GITHUB_BASE_REF를 참조할 수 있도록 얕지 않은 체크 아웃을 강제합니다.
        # 자세한 내용은 https://github.com/actions/checkout를 참조하십시오.
        fetch-depth: 0
    - uses: actions/setup-node@v4
      with:
        node-version: lts/*
    - name: Install dependencies
      run: npm ci
    - name: Install Playwright Browsers
      run: npx playwright install --with-deps
    - name: Run changed Playwright tests
      run: npx playwright test --only-changed=$GITHUB_BASE_REF
      if: github.event_name == 'pull_request'
    - name: Run Playwright tests
      run: npx playwright test
    - uses: actions/upload-artifact@v4
      if: ${{ !cancelled() }}
      with:
        name: playwright-report
        path: playwright-report/
        retention-days: 30
```

### Docker

[미리 빌드된 Docker 이미지](./docker.mdx)가 있습니다. 이 이미지를 직접 사용하거나 기존 Docker 정의를 업데이트하기 위한 참조로 사용할 수 있습니다.

제안된 구성
1. Chromium을 사용하는 경우 `--ipc=host`를 사용하는 것이 좋습니다. 그렇지 않으면 Chromium이 메모리 부족으로 인해 충돌할 수 있습니다. 이 옵션에 대한 자세한 내용은 [Docker 문서](https://docs.docker.com/reference/cli/docker/container/run/#ipc)를 참조하십시오.
1. Chromium을 시작할 때 다른 이상한 오류가 발생합니까? 로컬에서 개발할 때 `docker run --cap-add=SYS_ADMIN`으로 컨테이너를 실행해 보십시오.
1. PID=1인 프로세스에 대한 특별한 처리를 피하려면 `--init` Docker 플래그 또는 [덤프 초기화](https://github.com/Yelp/dumb-init)를 사용하는 것이 좋습니다. 이는 좀비 프로세스의 일반적인 원인입니다.

### Azure Pipelines

Windows 또는 macOS 에이전트의 경우 추가 구성이 필요하지 않으며 Playwright를 설치하고 테스트를 실행하기만 하면 됩니다.

Linux 에이전트의 경우 [컨테이너화된 작업 실행](https://docs.microsoft.com/en-us/azure/devops/pipelines/process/container-phases?view=azure-devops)을 지원하는 Azure Pipelines와 함께 [Docker 컨테이너](./docker.mdx)를 사용할 수 있습니다. 또는 [명령줄 도구](./browsers#install-system-dependencies)를 사용하여 필요한 모든 종속성을 설치할 수 있습니다.

Playwright 테스트를 실행하려면 다음 파이프라인 작업을 사용하십시오.

```yml
trigger:
- main

pool:
  vmImage: ubuntu-latest

steps:
- task: NodeTool@0
  inputs:
    versionSpec: '18'
  displayName: 'Install Node.js'
- script: npm ci
  displayName: 'npm ci'
- script: npx playwright install --with-deps
  displayName: 'Install Playwright browsers'
- script: npx playwright test
  displayName: 'Run Playwright tests'
  env:
    CI: 'true'
```

#### Uploading playwright-report folder with Azure Pipelines

이렇게 하면 Playwright 테스트 중 하나라도 실패하면 파이프라인 실행이 실패합니다. 테스트 결과를 Azure DevOps와 통합하려면 다음과 같이 `PublishTestResults` 작업을 사용하십시오.

```yml
trigger:
- main

pool:
  vmImage: ubuntu-latest

steps:
- task: NodeTool@0
  inputs:
    versionSpec: '18'
  displayName: 'Install Node.js'

- script: npm ci
  displayName: 'npm ci'
- script: npx playwright install --with-deps
  displayName: 'Install Playwright browsers'
- script: npx playwright test
  displayName: 'Run Playwright tests'
  env:
    CI: 'true'
- task: PublishTestResults@2
  displayName: 'Publish test results'
  inputs:
    searchFolder: 'test-results'
    testResultsFormat: 'JUnit'
    testResultsFiles: 'e2e-junit-results.xml'
    mergeTestResults: true
    failTaskOnFailedTests: true
    testRunTitle: 'My End-To-End Tests'
  condition: succeededOrFailed()
- task: PublishPipelineArtifact@1
  inputs:
    targetPath: playwright-report
    artifact: playwright-report
    publishLocation: 'pipeline'
  condition: succeededOrFailed()

```

참고: JUnit 리포터는 다음과 같이 적절하게 구성해야 합니다.

```js

  reporter: [['junit', { outputFile: 'test-results/e2e-junit-results.xml' }]],
});
```

### CircleCI

CircleCI에서 Playwright를 실행하는 것은 GitHub Actions에서 실행하는 것과 매우 유사합니다. 미리 빌드된 Playwright [Docker 이미지](./docker.mdx)를 지정하려면 다음과 같이 구성에서 `docker:`로 에이전트 정의를 수정하기만 하면 됩니다.

```yml
executors:
  pw-noble-development:
    docker:
      - image: mcr.microsoft.com/playwright:v1.51.1-noble
```

참고: Docker 에이전트 정의를 사용하는 경우 Playwright가 실행되는 리소스 클래스를 [여기](https://circleci.com/docs/configuration-reference?#docker-execution-environment)의 'medium' 티어로 지정합니다. Playwright의 기본 동작은 작업자 수를 감지된 코어 수(중간 계층의 경우 2)로 설정하는 것입니다. 작업자 수를 이 숫자보다 크게 재정의하면 불필요한 시간 초과 및 오류가 발생합니다.

#### CircleCI의 샤딩

CircleCI의 샤딩은 0으로 인덱싱되므로 기본 병렬 처리 ENV VARS를 재정의해야 합니다. 다음 예제에서는 `CIRCLE_NODE_INDEX`에 1을 더하여 `--shard` CLI 인수에 전달하여 병렬 처리 4로 CircleCI에서 Playwright를 실행하는 방법을 보여 줍니다.

```yml
  playwright-job-name:
    executor: pw-noble-development
    parallelism: 4
    steps:
      - run: SHARD="$((${CIRCLE_NODE_INDEX}+1))"; npx playwright test --shard=${SHARD}/${CIRCLE_NODE_TOTAL}
```

### Jenkins

Jenkins는 파이프라인을 위한 Docker 에이전트를 지원합니다. [Playwright Docker 이미지](./docker.mdx)를 사용하여 Jenkins에서 테스트를 실행합니다.

```groovy
pipeline {
   agent { docker { image 'mcr.microsoft.com/playwright:v1.51.1-noble' } }
   stages {
      stage('e2e-tests') {
         steps {
            sh 'npm ci'
            sh 'npx playwright test'
         }
      }
   }
}
```

#### 샤딩

GitLab CI는 [병렬](https://docs.gitlab.com/ee/ci/yaml/index.html#parallel) 키워드를 사용하여 [여러 작업 간의 샤딩 테스트](https://docs.gitlab.com/ee/ci/jobs/job_control.html#parallelize-large-jobs)를 지원합니다. 테스트 작업은 병렬로 실행되는 여러 개의 더 작은 작업으로 분할됩니다. 병렬 작업은 `job_name 1/N`에서 `job_name N/N`까지 순차적으로 이름이 지정됩니다.

```yml
stages:
  - test

tests:
  stage: test
  image: mcr.microsoft.com/playwright:v1.51.1-noble
  parallel: 7
  script:
    - npm ci
    - npx playwright test --shard=$CI_NODE_INDEX/$CI_NODE_TOTAL
```

GitLab CI는 또한 [parallel:matrix](https://docs.gitlab.com/ee/ci/yaml/index.html#parallelmatrix) 옵션을 사용하여 여러 작업 간의 샤딩 테스트를 지원합니다. 테스트 작업은 단일 파이프라인에서 병렬로 여러 번 실행되지만 각 작업 인스턴스에 대해 서로 다른 변수 값을 사용합니다. 아래 예에서는 2개의 `PROJECT` 값과 10개의 `SHARD` 값이 있어 총 20개의 작업이 실행됩니다.

```yml
stages:
  - test

tests:
  stage: test
  image: mcr.microsoft.com/playwright:v1.51.1-noble
  parallel:
    matrix:
      - PROJECT: ['chromium', 'webkit']
        SHARD: ['1/10', '2/10', '3/10', '4/10', '5/10', '6/10', '7/10', '8/10', '9/10', '10/10']
  script:
    - npm ci
    - npx playwright test --project=$PROJECT --shard=$SHARD
```

## Caching browsers

캐시를 복원하는 데 걸리는 시간이 바이너리를 다운로드하는 데 걸리는 시간과 비슷하므로 브라우저 바이너리를 캐싱하는 것은 권장되지 않습니다. 특히 Linux에서는 캐시할 수 없는 [운영 체제 종속성](./browsers.mdx#install-system-dependencies)을 설치해야 합니다.

CI 실행 간에 브라우저 바이너리를 캐시하려면 Playwright 버전의 해시에 대해 CI 구성에서 [이러한 디렉토리](./browsers.mdx#managing-browser-binaries)를 캐시하십시오.

## Debugging browser launches

Playwright는 실행 중에 디버그 로그를 출력하기 위해 `DEBUG` 환경 변수를 지원합니다. `Error: Failed to launch browser` 오류를 디버깅하는 동안 `pw:browser`로 설정하면 유용합니다.

```bash
DEBUG=pw:browser npx playwright test
```

## Running headed

기본적으로 Playwright는 헤드리스 모드에서 브라우저를 시작합니다. 헤드리스 모드에서 테스트를 실행하는 방법은 [테스트 실행](./running-tests.mdx#run-tests-in-headed-mode) 가이드에서 참조하십시오.

Linux 에이전트에서 헤드가 있는 실행을 하려면 [Xvfb](https://en.wikipedia.org/wiki/Xvfb)가 설치되어 있어야 합니다. [Docker 이미지](./docker.mdx) 및 GitHub Action에는 Xvfb가 미리 설치되어 있습니다. Xvfb를 사용하여 헤드가 있는 모드에서 브라우저를 실행하려면 실제 명령 앞에 `xvfb-run`을 추가하십시오.

```bash
xvfb-run npx playwright test
```
